# Procedural terrain: rivers, caves, voxels, and mesh extraction

## 1. River and drainage systems

River generation solves a problem that no amount of noise shaping or erosion simulation fully addresses: producing a **topologically valid, hierarchical drainage network** where every land cell has a defined path to the ocean and rivers never flow uphill. Simpler approaches — such as painting rivers as splines or tracing steepest-descent paths on raw noise — fail because noise-generated heightmaps contain local minima (depressions) that trap flow, producing dead-end rivers and pools in unintended locations. The output of a river system is a directed acyclic graph over the terrain mesh, annotated with flow-accumulation values that encode river width and downstream direction; this graph feeds biome moisture assignment, road/settlement placement constraints, and mesh-level valley carving.Implementationally, river generation sits after heightmap construction and erosion, receiving the eroded elevation grid and optionally a rainfall or moisture map, and producing a flow-direction grid, a flow-accumulation grid, drainage-basin labels, and river polylines or per-edge flow values.

The standard implementation pipeline begins with **depression filling** via the Priority-Flood algorithm (Barnes, Lehman & Mulla, 2014). Priority-Flood floods inward from grid edges using a min-heap ordered by elevation: each cell popped from the queue raises all unvisited neighbours to at least its own elevation, guaranteeing every cell can drain to an edge. The improved variant pushes cells raised inside a flat depression onto a plain FIFO queue rather than the priority queue, achieving up to **37% speedup** because all cells in a depression share the outlet's elevation (Barnes et al., 2014). A gradient-preserving extension, Priority-Flood+ε, adds infinitesimal increments via `NextAfter(elevation, ∞)` so no flat region remains, which prevents undefined flow directions later. Once depressions are filled, **D8 flow directions** assign each cell to its steepest downslope neighbour among the eight Moore neighbours, dividing diagonal height differences by √2 (O'Callaghan & Mark, 1984). Flow accumulation then traverses cells from highest to lowest elevation, adding each cell's upstream count to its downstream neighbour. Cells exceeding a threshold become river segments. The threshold is domain-specific; unconfirmed: for game heightmaps at 512² to 2048² resolution, a threshold of roughly 0.5–2% of total cell count produces visually reasonable density. An alternative to D8 is Rho8, a stochastic variant that randomly selects among downslope neighbours weighted by slope, which reduces axis-aligned grid artifacts in river paths (RichDEM documentation).

Red Blob Games' mapgen4 takes a structurally different approach: rivers are grown **upstream from the coast** on a Delaunay triangle mesh, ensuring validity by construction (redblobgames.com/x/1723-procedural-river-growing). The algorithm starts from ocean-adjacent triangles, uses a priority queue with noise-perturbed priorities to avoid symmetric drainage basins, and assigns each triangle a downstream pointer as it is visited — essentially Priority-Flood+FlowDirs on an irregular mesh. The resulting river tree is a binary forest where each node is classified as a source (0 entries, Strahler number 1), a bend (1 entry), or a fork (2 entries); at a fork, the Strahler number increments only when both tributaries have equal order, otherwise it takes the maximum (mapgen4 source, map.ts via DeepWiki). Rainfall is simulated with a wind-direction-aware pass: humidity is gathered from upwind neighbours, evaporation adds humidity over water, and orographic precipitation triggers when humidity exceeds `(1.0 − elevation)`, producing realistic rain-shadow effects (simblob.blogspot.com/2018/09/mapgen4-rainfall). Flow accumulation traverses the tree array in reverse breadth-first order, summing rainfall contributions. Edges with flow > 0 are rendered as valleys (triangle vertices lowered); all others as ridges (mapgen4 geometry.ts). Daniel Andrino's "terrain-erosion-3-ways" project pushes this further into a **rivers-first** paradigm: rivers are grown upstream on a Poisson-disc Delaunay mesh with a directional-inertia parameter (low inertia → twisty rivers; high inertia → straight), and terrain elevation is then derived from the river graph by traversing uphill and scaling height deltas inversely by water volume (github.com/dandrino/terrain-erosion-3-ways). Note that Sebastian Lague's well-known procedural terrain series does **not** include explicit river generation — his hydraulic erosion produces channel-like features after 70 000+ particle iterations, but these are emergent and have no explicit river graph (github.com/SebLague/Hydraulic-Erosion).

The principal failure modes are well-characterized. **Rivers flowing uphill** occur when steepest-descent tracing encounters unfilled depressions; the fix is Priority-Flood preprocessing or upstream-growth construction. **Parallel, symmetric drainage** results from uniform BFS from grid edges producing a four-sided pyramid pattern; the fix is noise-perturbed priority ordering (redblobgames.com/x/1723). **Grid-axis artifacts** — rivers that run horizontally or vertically rather than diagonally — arise from D8 on a regular grid; switching to an irregular mesh (Poisson disc + Delaunay) or using Rho8 mitigates this. **Disconnected segments** appear when flat areas leave flow direction undefined; Priority-Flood+ε resolves this by ensuring monotonic gradient everywhere (Barnes et al., 2014). **Rivers forming cycles** cannot happen in a spanning-tree representation but can occur with careless random flow assignment; tree construction prevents this by design. Implementation-level documentation for river generation is thinner than for the other techniques in this document — most primary sources are either academic GIS literature (written for geographers, not game developers) or project-specific blog posts. The mapgen4 and Andrino repositories are the most implementation-complete references available; parameter tuning for game-scale worlds remains largely empirical.

## 2. Cave generation

Cave generation produces connected, walkable subterranean spaces in either 2D grids (for roguelikes and top-down games) or 3D density fields (for voxel worlds). The core problem it solves is that hand-authored cave layouts do not scale, while naive random carving produces spaces with no organic structure. Three fundamentally different approaches exist — **cellular automata**, **noise thresholding**, and **agent tunnelling** — each with distinct output character, designer control surface, and pipeline integration.

### Cellular automata

The cellular-automata method, popularised by Jim Babcock's RogueBasin article, generates organic, blobby cavern shapes by repeatedly applying local neighbourhood rules to an initially random binary grid. The algorithm initialises each interior cell as wall with probability **P** (typically **40–45%**; RogueBasin recommends 40%, Jeremy Kun uses 45%), marks all border cells as permanent walls, then iterates a smoothing rule. The most common rule is the **4-5 rule**: a cell becomes wall if the count of walls in its 3×3 Moore neighbourhood (including itself) is ≥ 5, otherwise it becomes floor (RogueBasin, "Cellular Automata Method for Generating Random Cave-Like Levels"). In Life-like notation this is B5678/S45678, which Kun shows converges within roughly **15 iterations** and "prunes sparse areas and fills in dense areas by patching up holes" (jeremykun.com, 2012). A stronger variant uses a **radius-2 check** in an initial phase: `W'(p) = R1(p) ≥ 5 OR R2(p) ≤ 2`, run for 4 iterations, followed by 3 iterations of the simpler R1 ≥ 5 rule (RogueBasin). The R2 term fills cells in the middle of open space, producing pillars and more varied topology. Sebastian Lague's Unity cave series uses 50% initial fill and a variant where T > 4 → wall, T = 4 → unchanged, T < 4 → floor (github.com/SebLague/Procedural-Cave-Generation).

Connectivity is **not guaranteed** by any CA rule. The standard fix is flood-fill from a seed cell followed by culling all unreachable floor cells (RogueBasin: "on large maps, almost always you will select the largest open connected area"). Sebastian Lague's approach detects all regions, finds the closest pair of disconnected rooms, and draws corridors between their nearest cells until every room is reachable. An alternative is horizontal blanking — clearing a 3–4-cell-tall strip before CA iterations to prevent continuous vertical walls from isolating halves (unconfirmed: widely cited in roguelike forums but no primary reference found). Failure modes include **isolated pockets** (fill probability too high or too few iterations), **one giant featureless cavern** (fill probability too low, < 35%), and **checkerboard/camouflage texture** at large grid sizes, which Kun resolves by generating at low resolution, upscaling 4×, then smoothing with B5678/S5678 to remove convex corners (jeremykun.com). CA caves are primarily 2D; extension to 3D using the 26-cell Moore neighbourhood is possible but poorly documented — no standard 3D rule parameters have been published in a primary source.

### Noise thresholding

Noise thresholding generates caves by evaluating a 3D noise function at each voxel and carving space where the value falls below (or within) a threshold. This method is native to 3D, chunk-independent (each voxel's state depends only on its world coordinates), and deterministic from seed — making it ideal for infinite voxel worlds where chunk boundaries must be seamless. A critical insight is that **a single 3D Perlin noise isosurface produces curved shells, not tunnels**. To obtain tunnel-like geometry, the standard technique intersects two independent noise fields: a voxel is air only if `abs(noise1) < width AND abs(noise2) < width`, where each noise is evaluated at the same position with different seeds (Danol's blog, blog.danol.cz; Accidental Noise Library, accidentalnoise.sourceforge.net/minecraftworlds). The `width` parameter (typically **±0.1** around zero) controls tunnel diameter. The Accidental Noise Library recommends 1-octave ridged multifractal at frequency 4 for each field, domain-perturbed by 6-octave fBm at scale 0.5, with a final threshold of **0.48** (accidentalnoise.sourceforge.net). The Voxel Tools engine for Godot documents an alternative "squaring" technique: square a 2D noise value to concentrate values near zero, then threshold the result to reveal path-like patterns, modulated along Y by a parabola `(y² − 1)` to produce 3D worm-shaped caves (voxel-tools.readthedocs.io, procedural_generation). **Depth attenuation** is essential: multiply cave noise by a depth gradient biased toward underground (bias ≈ 0.45) so caves thin to nothing near the surface and grow larger at depth. Without this, caves punch through the terrain surface — the most common failure mode.

Other failure modes include **disconnected small pockets** when the threshold is too high (producing many small voids instead of connected tunnels), **unnaturally smooth tunnels** when domain perturbation is omitted, and **uniformly boring caves** when only a single octave is used. Unconfirmed: typical parameter ranges for game-scale caves are base frequency 0.01–0.1 for large caverns, 0.1–0.5 for narrow tunnels, persistence 0.5, lacunarity 2.0, and threshold −0.1 to 0.2 — these are aggregated from multiple tutorial sources without a single authoritative reference. Integration with a voxel pipeline is arithmetic: `final_density = terrain_density × cave_mask`, where `cave_mask` is 0 in carved regions and 1 elsewhere. The density field then passes directly to marching cubes or an equivalent mesh extractor.

### Agent tunnelling

Agent tunnelling (the "drunkard's walk") builds caves by sending one or more agents on random walks through a wall-filled grid, carving floor tiles as they go. The technique offers the strongest **designer control** of the three methods because agent parameters — spawn position, lifetime, direction weights, brush radius, branching rules — directly and predictably shape the output. The basic algorithm starts with an entirely solid grid, spawns an agent at a chosen position, and lets it walk randomly (equal probability N/S/E/W) for `L` steps, marking each visited cell as floor. This repeats until a target floor percentage is reached (Bracket Productions, "Hands-On Rust" roguelike tutorial, bfnightly.bracketproductions.com). Bracket Productions defines three presets: **Open Area** (spawn at centre, lifetime 400, 50% floor → large central cavern), **Open Halls** (random spawn, lifetime 400, 50% → sprawling layout), and **Winding Passages** (random spawn, lifetime 100, 40% → narrow, labyrinthine tunnels). A single walk from one starting point is **inherently connected** — every carved tile is adjacent to the previous one — so connectivity is guaranteed without post-processing as long as only one start point is used (RogueBasin, "Random Walk Cave Generation").

For 3D voxel worlds, **Perlin worms** replace the uniform random walk with noise-guided movement. The Caveworm plugin (github.com/Maxopoly/Caveworm) uses three independent simplex generators, one per axis: if the noise sample exceeds a positive threshold, the worm moves +1 in that axis; below a negative threshold, −1; within the neutral zone, no movement. At each position, a sphere of variable radius (controlled by a fourth simplex generator with configured `lowerBound` and `upperBound`) is carved. Key parameters are movement octave count (3 per axis — fewer → straighter caves; more → random/cyclic), frequency (0.05–0.1; lower → straighter segments), and per-axis spread amplitude (asymmetric amplitudes produce flatter or taller caves) (Caveworm README). The UPenn Mini-Minecraft assignment describes the same principle: "Perlin worms work by sampling 3D Perlin noise at each walk interval to determine the direction in which the walk should move next" (cis.upenn.edu/~cis4600). The main failure mode is **wasted iterations**: agents wandering over already-carved floor contribute nothing, and long-lived agents at a single spawn point produce shapeless blobs. Fixes include capping by floor percentage, spawning subsequent agents at the boundary of existing floor, and tracking a "did_something" flag. For Perlin worms, excessively high frequency produces chaotic movement indistinguishable from a plain random walk; excessively low frequency produces boringly straight tunnels. Unconfirmed: a frequency of **0.07–0.1** typically balances curvature and coherence in 3D. A fundamental limitation of agent tunnelling is that agent state is **sequential** — the worm's position depends on its entire history — so caves cannot be evaluated per-chunk independently the way noise thresholding can. The workaround for chunk-based worlds is to pre-compute worm paths in a global pass or switch to the dual-isosurface noise method for the primary cave system.

## 3. Voxel terrain

Voxel terrain replaces the 2D heightmap — a function `h(x, z)` returning one elevation per horizontal coordinate — with a **3D scalar field** where each point `(x, y, z)` holds a density or signed-distance value. The surface exists at the zero-crossing of this field: positive values are solid, negative are air (or vice versa by convention). This representation is required whenever the world contains **overhangs, caves, arches, tunnels, or destructible geometry** — features that are topologically impossible for a single-valued height function. As Ryan Geiss writes in GPU Gems 3, Chapter 1: "the terrain surface can be completely described by a single function, called the density function… the boundary between positive and negative values — where the density value is zero — is the surface of the terrain" (NVIDIA GPU Gems 3, 2007). The output of the voxel stage is a per-chunk 3D array of scalar values (and optionally a parallel material-type channel), which feeds directly into an isosurface extraction algorithm — marching cubes, dual contouring, or surface nets — that produces the renderable mesh.

The most common data structure in practice is a **chunked flat array**: the world is divided into fixed-size cubic chunks (typically **16³** for Minecraft-style games, **32³** for smooth-terrain engines), each stored as a contiguous 1D array indexed as `array[x + y·sizeX + z·sizeX·sizeY]`, with chunks held in a hash map keyed by chunk coordinate (bonsairobo, "Smooth Voxel Mapping", Medium). Empty chunks are simply absent from the map. Access is O(1) amortised (hash lookup + direct index), memory is proportional to non-empty chunks only, and linear traversal has excellent cache coherence for mesh generation. Memory per chunk at 16-bit SDF precision is **8 KB** for 16³ and **64 KB** for 32³. A view distance of 16 chunks in each direction at LOD 0 would naively require ~32 768 chunks × 8 KB = **256 MB** for voxel data alone — making a LOD system essential (Voxel Tools documentation, voxel-tools.readthedocs.io). Alternatives include **octrees**, which collapse homogeneous regions to single nodes (O(log N) access, much lower memory for sparse data, but pointer-chasing hurts cache performance), and **sparse voxel octrees (SVOs)**, which use linearised child-mask + pointer encoding for GPU-friendly traversal (Laine & Karras, NVIDIA, 2010). In practice, the hybrid "chunked array" wins for most game engines because it balances simplicity, cache performance, and streaming friendliness; SVOs excel at extremely large-scale or ray-traced rendering.

Populating the density field follows a layered approach. The base layer establishes a ground surface: `density = surfaceHeight(x, z) − y`, where `surfaceHeight` is a 2D noise function (fBm, ridged multifractal, or a pre-computed heightmap from upstream). This yields solid below the noise surface and air above. A 3D noise layer then adds volumetric features: `density += noise3D(x·freq, y·freq, z·freq) · amplitude`, producing overhangs and ledges where the 3D noise pushes the zero-crossing above or below the base surface (GPU Gems 3, Ch. 1). Cave carving subtracts from this field using the techniques from Section 2: either `final = terrain · cave_mask` (where `cave_mask` is 0 in carved regions) or, for SDF data, `final = max(terrain, −caves)` using CSG subtraction (Voxel Tools docs). CSG operations on density fields are trivially arithmetic: union is `max(a, b)`, subtraction is `max(a, −b)`, intersection is `min(a, b)`, and smooth blending uses smooth-min functions for organic transitions. Storing true signed distance (`|∇SDF| = 1`) provides geometrically correct normals and enables sphere-tracing for ray-marched rendering, but maintaining global SDF after edits is expensive; in practice, engines store a **clamped SDF** — distances truncated to a narrow band around the surface (e.g., ±3 voxels) — so modifications remain local (Voxel Tools smooth_terrain docs).

LOD is implemented via **nested grids (clipmaps)**: LOD 0 uses the smallest voxels (e.g., 1 m), LOD 1 doubles the voxel size (2 m) while keeping the same block resolution (16³ or 32³), and so on. Each LOD level covers twice the spatial extent with the same data footprint, so total memory grows logarithmically rather than cubically with view distance (Voxel Tools, voxel-tools.readthedocs.io). The critical failure mode at LOD boundaries is **cracks (T-junctions)**: meshes from adjacent chunks at different resolutions do not share edge vertices, creating visible holes. The Transvoxel Algorithm (Lengyel, 2010) resolves this with special transition cells — discussed in Section 4. Other failure modes include **chunk-boundary seams** (fixed by overlapping density data by 1 voxel so a 32³ chunk samples a 33³ grid), **LOD popping** (mitigated by vertex morphing: storing both current and parent-LOD vertex positions and blending by distance; dexyfex.com, 2016), **memory exhaustion** from O(N³) scaling (mitigated by LOD, streaming, and 16-bit rather than 32-bit storage), and **undersampling** where coarse LOD voxels miss terrain features (mitigated by matching noise frequency to voxel resolution so coarse levels only generate coarse features; blog.mfichman.net, 2021).

## 4. Marching cubes, dual contouring, and surface nets

These three algorithms solve the same problem — extracting a polygonal mesh from a 3D scalar field — but make fundamentally different geometric choices about **where to place vertices** and **how to connect them**, producing different tradeoffs in feature preservation, mesh complexity, implementation effort, and LOD compatibility. All three receive a voxel density field (from Section 3) and produce vertex positions, normals, and triangle/quad index buffers suitable for rendering and collision. This section warrants extended treatment because the algorithmic differences are precise and consequential, and conflating them leads to poor implementation decisions.

### Marching cubes at implementation depth

Marching cubes (Lorensen & Cline, 1987) processes each cubic cell of the grid independently. For each cell, it samples the scalar field at the **8 corner vertices**, builds an 8-bit case index where bit *i* is set if corner *i*'s value is below the isolevel, uses the case index to look up which of the cell's 12 edges are intersected, linearly interpolates the crossing position along each intersected edge, and emits triangles from a precomputed table. The implementation is driven almost entirely by two lookup tables, first published in usable form by Cory Gene Bloyd and documented by Paul Bourke (paulbourke.net/geometry/polygonise, 1994). `edgeTable[256]` maps each case index to a 12-bit mask indicating which edges are cut; `triTable[256][16]` stores up to 5 triangles per case as sequences of edge indices terminated by −1. The 256 configurations reduce to **15 unique geometric cases** by rotational symmetry and inside/outside complementarity (Bourke, 1994). The key step is edge interpolation: given endpoints P₁, P₂ with scalar values V₁, V₂, the surface crossing is at `P₁ + t·(P₂ − P₁)` where `t = (isolevel − V₁)/(V₂ − V₁)` (Bourke, 1994). In pseudocode:

```
for each cell (x, y, z):
    for i = 0..7: val[i] = sample(corner[i])
    caseIndex = 0
    for i = 0..7: if val[i] < isolevel: caseIndex |= (1 << i)
    edges = edgeTable[caseIndex]
    if edges == 0: continue
    for e = 0..11:
        if edges & (1 << e):
            P1, P2 = endpoints(e)
            t = (isolevel - val[P1]) / (val[P2] - val[P1])
            vertList[e] = lerp(P1, P2, t)
    for i = 0; triTable[caseIndex][i] != -1; i += 3:
        emit_triangle(vertList[triTable[caseIndex][i]],
                       vertList[triTable[caseIndex][i+1]],
                       vertList[triTable[caseIndex][i+2]])
```

The original 15-case table contains **face-ambiguous configurations** (cases 3, 6, 7, 10, 12, 13) where two diagonally opposite corners of a face share sign but the other diagonal pair disagree, allowing two topologically distinct triangulations. If adjacent cells resolve the ambiguity inconsistently, **cracks** appear. Nielson & Hamann's Asymptotic Decider (1991) resolves this by computing `Q = A·C − B·D` for face corners A, B, C, D with A, C same-sign; the sign of Q determines which diagonal the hyperbolic contour follows. Chernyaev (1995) extended the table to **33 base cases** ("MC33"), and Lewiner et al. (2003) provided a complete implementation with **730 subcases** guaranteeing manifold, crack-free meshes matching the trilinear interpolant's topology (Lewiner, Lopes, Vieira & Tavares, *Journal of Graphics Tools*, 2003). Other failure modes include **staircase artifacts** on axis-aligned surfaces (inherent to edge-constrained vertex placement; fixable only by increasing resolution or switching to a dual method), **T-junctions between LOD levels** (requiring Transvoxel, discussed below), and **high triangle counts** — up to 5 triangles per cell, many thin and poorly shaped.

### Dual contouring and the QEF

Dual contouring (Ju, Losasso, Schaefer & Warren, SIGGRAPH 2002) is the geometric "dual" of marching cubes: it places **one vertex inside each cell** that contains a sign change and emits **one quad per edge** exhibiting a sign change, connecting the vertices of the 4 cells sharing that edge (BorisTheBrave.com). Because the vertex is not constrained to lie on a cell edge, it can sit at the precise location of a sharp corner or crease — this is the algorithm's primary advantage. The price is a stricter input requirement: dual contouring needs **Hermite data**, meaning not only scalar values at cell corners but also the gradient (normal) of the field at each edge crossing. For each cell, the algorithm finds all edge crossings (same interpolation as MC), evaluates the gradient at each crossing point, and solves a **Quadratic Error Function (QEF)** to find the point minimising the sum of squared distances to the tangent planes implied by the crossings:

```
E(x) = Σᵢ [nᵢ · (x − pᵢ)]²
```

where pᵢ is the crossing position and nᵢ the surface normal at that crossing. This expands to a 3×3 normal-equation system `AᵀA·x = AᵀB` where `AᵀA = Σ nᵢᵀ·nᵢ` (a symmetric 3×3 matrix), `AᵀB = Σ nᵢ·(nᵢ·pᵢ)` (a 3-vector), and `BᵀB = Σ(nᵢ·pᵢ)²` (a scalar for error evaluation). These six unique matrix values plus the vector and scalar can be accumulated incrementally — no dynamic allocation needed (Matt Keeter, mattkeeter.com/projects/qef; Schaefer & Warren, "Dual Contouring: The Secret Sauce"). When normals are nearly coplanar, AᵀA becomes rank-deficient; the standard fix is SVD-based pseudo-inverse: zero out eigenvalues below a tolerance ε, then solve in the reduced space. A **centering trick** prevents rank-deficient solutions from biasing toward the origin: subtract the cell's mass point (average of crossing positions) before solving, then add it back (Schaefer & Warren, "Secret Sauce").

The most common failure mode is **QEF-placed vertices landing outside their cell**, which occurs with large flat surfaces that produce nearly colinear normals. Fixes include clamping the solution to cell bounds, adding a regularisation term biasing toward the cell centre, or falling back to the mass point (BorisTheBrave.com; Schaefer & Warren). **Non-manifold vertices** arise when two distinct surface sheets pass through the same cell and share its single vertex; Manifold Dual Contouring (Schaefer, Ju & Warren) detects these configurations via corner-sign analysis and splits the vertex. DC naturally extends to adaptive octrees: recursive `cellProc`, `faceProc`, and `edgeProc` functions walk the tree, emitting quads only at leaf-level edges with sign changes (Nick Gildea, ngildea.blogspot.com). QEFs from child cells can be summed (just add the compact matrices) to produce simplified parent vertices for LOD, with the accumulated error value controlling simplification aggressiveness.

### Surface nets — simplicity at the cost of sharp features

Surface nets (Gibson, "Constrained Elastic Surface Nets", 1999) use the same dual topology as dual contouring — one vertex per sign-changing cell, quads connecting cells across sign-changing edges — but replace the QEF with a **simple average of edge-crossing positions** (Mikola Lysenko, 0fps.net, "Smooth Voxel Terrain Part 2", 2012). This eliminates the need for gradient data entirely: surface nets require only scalar corner values, the same input as marching cubes. The vertex computation is trivially `v = (1/k) · Σ crossingPositions` for the *k* crossings in the cell. This produces meshes with roughly **half the face count** of marching cubes (quads vs. up to 5 triangles per cell), smoother surfaces, no ambiguity problems, and no lookup tables beyond edge topology (0fps.net). Implementation complexity is comparable to naive Minecraft cube meshing. The fast-surface-nets library in Rust reports throughput of approximately **20 million triangles per second** on a single core (bonsairobo/fast-surface-nets-rs). The tradeoff is that averaging always bevels sharp features — the same limitation as marching cubes. Optional **Laplacian relaxation** (3–10 passes with relaxation factor λ = 0.3–0.7; Gibson, 1999) smooths jagged quad edges, but can cause volume shrinkage if vertices are not projected back toward the isosurface. Chunk-based implementations require **1-voxel boundary padding**: a 16³ chunk must sample an 18³ region, with quads generated only on the negative-direction boundaries, so meshes tile seamlessly (bonsairobo/fast-surface-nets-rs).

### Transvoxel: solving LOD cracks for marching cubes

The Transvoxel Algorithm (Lengyel, 2009–2010; transvoxel.org) introduces **transition cells** that seamlessly stitch marching-cubes meshes at differing LOD resolutions. A transition cell has **9 sample points** on the high-resolution face (the 4 corners plus 5 edge midpoints) and **4 points** on the low-resolution face whose isovalues match the corresponding high-resolution corners, ensuring geometric consistency. This gives **512 possible configurations** falling into **73 equivalence classes** (transvoxel.org), each with a precomputed triangulation stored in lookup tables that Lengyel provides freely (github.com, Transvoxel lookup tables, copyright 2009 Eric Lengyel). The constraint is that adjacent blocks must differ by **exactly a factor of 2** in resolution; Lengyel recommends 16×16×16-voxel blocks at LOD 0, with lower LODs covering 32³ and 64³ volumes while still sampling 17×17×17 data points (counting by 2s and 4s respectively) (transvoxel.org; Lengyel, "Voxel-Based Terrain for Real-Time Virtual Simulations", PhD diss., UC Davis, 2010). Transition cells require only local voxel data, enabling real-time retriangulation when the voxel field changes. Transvoxel is specifically designed for marching-cubes-based systems; dual contouring and surface nets handle LOD differently through adaptive octree refinement, where the recursive `faceProc`/`edgeProc` procedures naturally accommodate varying cell sizes. Note that Lengyel's modified marching-cubes tables (distributed alongside the Transvoxel tables) also fix some ambiguous-case inconsistencies in the original Lorensen & Cline tables.

## Heightmap terrain vs. voxel terrain

A 2D heightmap stores one elevation value per horizontal coordinate and can represent any surface that passes the **vertical line test** — no point on the terrain has two heights at the same (x, z). This makes heightmaps compact (**O(N²)** storage vs. **O(N³)** for voxels), trivially streamable as image tiles, compatible with standard GPU tessellation and displacement-mapping pipelines, and fast to generate and erode. Use a heightmap when the world is predominantly outdoor terrain — rolling hills, mountains, valleys, rivers — and all subterranean or overhang features can be faked with decal geometry or baked meshes. The moment the design requires true caves, arches, floating islands, tunnels, or real-time destructible terrain where arbitrary interior volumes can be removed, a heightmap structurally cannot encode the geometry and a voxel density field becomes necessary. The crossover point is concrete: if fewer than roughly 5–10% of surface area involves overhangs or interior voids, faking them on top of a heightmap (unconfirmed: no published threshold exists; this is a practical heuristic) is cheaper than bearing the O(N³) memory, mesh-extraction cost, and texturing complexity of a full voxel system. If voids are pervasive or player-created destruction is a core mechanic, voxels are the only viable representation. Hybrid approaches — heightmap for the surface, voxel volumes only where needed — are common in shipped titles (e.g., Unconfirmed: several open-world engines use heightmap terrain with voxel "patches" for cave interiors).

## Cellular automata caves vs. noise caves vs. agent tunnels

Cellular automata produce **organic, irregular caverns** with blobby walls and varied room sizes — the output resembles natural limestone dissolution caves. The designer controls only three numbers (fill probability, rule variant, iteration count), and the output is stochastic: different seeds produce qualitatively similar but unpredictable layouts. CA is almost exclusively 2D and requires connectivity post-processing. Noise thresholding produces **Swiss-cheese or worm-tunnel networks** in 3D, is fully deterministic from seed, and — critically — is **chunk-independent**: each voxel's state depends only on its world coordinates, making it ideal for infinite streaming worlds. The dual-isosurface intersection technique is required for tunnel-like results; a single noise field produces shells. Control is via frequency, threshold, and attenuation curves, which are less intuitive than CA parameters but offer finer spatial variation (e.g., larger caves at greater depth). Agent tunnelling produces **corridor-dominant layouts** — narrow tunnels connecting occasional wider spaces — with the most direct designer control: agents can be steered with direction bias, branching rules, and variable brush radius. Perlin worms extend this to 3D with smooth, coherent paths. The fundamental limitation is sequential agent state, which prevents chunk-independent evaluation. Use CA for 2D roguelike cave levels where organic shape matters and exact layout does not; use noise thresholding for 3D voxel games requiring seamless infinite generation; use agent tunnelling when level-design intent must be encoded (e.g., ensuring a path between two specific rooms or creating a dungeon with controlled branching factor).

## Marching cubes vs. dual contouring vs. surface nets

Marching cubes places vertices **on cell edges** via linear interpolation and emits up to 5 triangles per cell from a 256-entry lookup table, requiring only scalar corner values. It is the simplest to implement, the most widely supported in engines and tooling, and the fastest per-cell (table lookup + interpolation, almost branch-free). Its mesh is the densest of the three (highest triangle count for equivalent resolution), it always bevels sharp features because vertices cannot leave edges, and its original lookup table has face-ambiguity issues that require MC33 or the Asymptotic Decider to resolve. Transvoxel provides LOD stitching. Dual contouring places **one vertex inside each sign-changing cell** by solving a QEF from Hermite data (position + normal at edge crossings), emitting quads between cell vertices across sign-changing edges. It uniquely preserves **sharp corners and creases** because the QEF finds the intersection of tangent planes. It requires gradient data, involves per-cell SVD or eigendecomposition (the slowest of the three), and can produce vertices outside cells or non-manifold topology without clamping and manifold-aware post-processing. It naturally extends to adaptive octrees for LOD. Surface nets use identical dual topology to DC but replace the QEF with the **centroid of edge crossings**, needing only scalar data. This is nearly as simple as marching cubes, produces roughly half the face count, generates smoother meshes with better-shaped quads, but cannot preserve sharp features (same beveling as MC). It is the fastest dual method (~20M tri/sec single-core in Rust; bonsairobo/fast-surface-nets-rs). Choose marching cubes for maximum ecosystem compatibility and when sharp features are irrelevant; choose dual contouring when corners and edges must be preserved (architectural or CSG-heavy geometry); choose surface nets for the best balance of simplicity, mesh quality, and performance on organic terrain.

## End-to-end terrain pipeline

1. **Seed and global parameters** → define noise seeds, world size, biome table, sea level.
2. **Base heightmap generation** → layer fBm, ridged multifractal, and domain-warped noise to produce a 2D elevation field.
3. **Island mask and continent shaping** → multiply heightmap by radial falloff or Voronoi-based continent masks to define land/ocean.
4. **Mountain shaping, cliffs, terraces** → apply power curves, step functions, and ridged noise to sculpt elevation detail.
5. **Hydraulic and thermal erosion** → simulate particle-based erosion (Lague/Beyer method) or grid-based erosion to carve realistic drainage patterns into the heightmap.
6. **Depression filling and flow-direction assignment** → run Priority-Flood on the eroded heightmap; compute D8 flow directions.
7. **Flow accumulation and river extraction** → traverse cells highest-to-lowest, sum upstream counts; threshold to extract river network.
8. **River carving and moisture propagation** → lower heightmap along river paths proportional to flow; propagate moisture outward from rivers and rainfall for biome assignment.
9. **Biome classification** → map (elevation, moisture, temperature) to biome IDs via Whittaker-style lookup.
10. **Voxel density field generation (if voxel pipeline)** → convert heightmap to `density = h(x,z) − y`; add 3D noise for overhangs; subtract cave volumes (noise thresholding or Perlin worms).
11. **Mesh extraction** → marching cubes / dual contouring / surface nets per chunk; Transvoxel for LOD transitions.
12. **Texture/material assignment and rendering** → triplanar mapping, splat maps from biome data, normal computation from density gradient or mesh normals.

This pipeline divides naturally into a **2D heightmap pass** (stages 2–9) and a **3D voxel pass** (stages 10–12), with stage 6–8 bridging the two by operating on the 2D heightmap but producing data consumed by both downstream paths. Common developer mistakes across the full pipeline include: **generating noise at world coordinates without chunk-local offsetting**, which causes floating-point precision loss at large distances (fix: always compute noise relative to chunk origin); **forgetting the 1-voxel overlap** at chunk boundaries for mesh extraction, producing visible seams; **applying erosion after river extraction** rather than before, which destroys the carefully computed drainage graph; **using a single 3D noise function for caves** instead of dual-isosurface intersection, producing shell-like voids instead of tunnels; and **not depth-attenuating cave noise**, causing caves to break through the terrain surface as visual holes in the ground.

## Source landscape

**Primary papers and dissertations.** Lorensen & Cline, "Marching Cubes" (SIGGRAPH 1987) — the original algorithm definition; Ju, Losasso, Schaefer & Warren, "Dual Contouring of Hermite Data" (SIGGRAPH 2002) — defines Hermite-data-driven vertex placement and QEF formulation; Lengyel, "Voxel-Based Terrain for Real-Time Virtual Simulations" (PhD diss., UC Davis, 2010) — the most complete treatment of Transvoxel and modified MC tables for games; Gibson, "Constrained Elastic Surface Nets" (1999) — the original surface nets formulation; Lewiner et al., "Efficient Implementation of Marching Cubes' Cases with Topological Guarantees" (JGT 2003) — resolves all 730 MC subcases for manifold meshes; Barnes, Lehman & Mulla, "Priority-Flood" (2014) — the definitive depression-filling algorithm with five variants. Lengyel's dissertation is uniquely valuable for its unified treatment of MC table corrections, LOD transition geometry, and voxel terrain texturing — no other single source covers all three.

**Tutorial series and blogs.** Paul Bourke, "Polygonising a Scalar Field" (paulbourke.net, 1994) — the canonical MC implementation reference with Bloyd's lookup tables in copy-pasteable C; Red Blob Games, mapgen4 articles (redblobgames.com) — the clearest walkthrough of upstream-growth river construction on irregular meshes; Mikola Lysenko, "Smooth Voxel Terrain" (0fps.net, 2012) — the best single comparison of MC vs. DC vs. surface nets with visual examples; BorisTheBrave.com — excellent practical tutorials on both MC ambiguity resolution and DC implementation; Matt Keeter, QEF explainer (mattkeeter.com/projects/qef) — the most accessible derivation of QEF solving for dual contouring; Bracket Productions, Rust roguelike tutorial (bfnightly.bracketproductions.com) — the most implementation-complete treatment of drunkard's walk cave parameters.

**Repositories.** Sebastian Lague, Procedural-Cave-Generation and Marching-Cubes (github.com/SebLague) — Unity implementations with clean code structure but limited documentation; Caveworm (github.com/Maxopoly/Caveworm) — the most configurable open-source Perlin worm implementation with per-axis control; fast-surface-nets (github.com/bonsairobo/fast-surface-nets-rs) — the highest-performance open-source surface nets with benchmarks; dandrino/terrain-erosion-3-ways — three complete river-terrain integration approaches in a single repo, uniquely valuable for comparing teleological vs. ontogenetic pipelines.

**Interactive tools.** Red Blob Games mapgen4 (redblobgames.com/maps/mapgen4) — the only browser-interactive tool demonstrating real-time river generation with paintable elevation and wind-based rainfall; Transvoxel.org — provides downloadable lookup tables and visual case diagrams for all 73 transition-cell equivalence classes, the only freely available resource for these tables; Accidental Noise Library (accidentalnoise.sourceforge.net) — the most detailed walkthrough of Minecraft-style cave noise composition, uniquely covering the dual-ridged-multifractal approach with domain perturbation.
