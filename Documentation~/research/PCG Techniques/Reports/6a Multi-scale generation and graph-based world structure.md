# Pass 6a: Multi-scale generation and graph-based world structure

Procedural worlds that feel infinite demand two capabilities no single-pass generator can provide: the ability to produce detail at multiple spatial scales without holding the entire world in memory, and the ability to impose navigable, designerly structure onto regions that may not yet exist. This pass introduces the architectural patterns that make streaming open worlds and structured adventure spaces possible. The six multi-scale techniques address *how* generation work is subdivided, scheduled, and stitched across spatial boundaries. The four graph-based techniques address *what* high-level structure governs that generation — how regions connect, how progression gates constrain traversal, and how a critical path emerges from abstract topology before any geometry is placed. Together, these ten techniques form the orchestration layer that wraps every technique from earlier passes: fBm evaluation becomes chunk-local, erosion must respect tile boundaries, dungeon grammars operate inside graph-assigned regions, and biome placement answers to an overworld connectivity contract.

---

## 1. Chunk-based generation

The central problem chunk-based generation solves is memory: an infinite or planet-scale world cannot be held in RAM simultaneously, yet the player expects seamless terrain in every direction. Without spatial subdivision, the generator must either cap world size or accept unbounded memory growth. Chunking partitions the world into fixed-size spatial cells — typically axis-aligned squares or cubes — each of which can be generated, loaded, and discarded independently. This transforms an intractable global generation problem into a bounded local one, where the cost of producing any single cell is constant regardless of world size.

Minecraft's implementation is the canonical reference. Each chunk is a **16×16-block column** (384 blocks tall since 1.18) that progresses through a strict multi-stage pipeline: `empty → structure_starts → structure_references → biomes → noise → surface → carvers → features → light → full`. Proto-chunks represent intermediate states; the game can freeze chunks at early stages far from the player and advance them only as the player approaches. During the `noise` stage, layered 3D density noise (an fBm variant with multiple octaves of Perlin/Simplex) determines the solid/air boundary at each block position. The `surface` stage replaces top blocks with biome-dependent materials. The `carvers` stage runs Perlin worms and, since 1.18, three noise-cave types (cheese, spaghetti, noodle) plus localized aquifers. Finally, the `features` stage places trees, ores, and structures using per-feature seeds derived from the world seed and chunk coordinates (minecraft.wiki, "World generation"; minecraft.wiki, "Chunk format").

The seeding architecture deserves close attention. Minecraft's `ChunkRandom` class provides four distinct seeding methods, each mixing the 64-bit world seed with spatial coordinates through a Linear Congruential Generator: `setPopulationSeed(worldSeed, blockX, blockZ)` for decoration, `setDecoratorSeed(populationSeed, index, step)` for individual features, `setCarverSeed(worldSeed, chunkX, chunkZ)` for caves, and `setRegionSeed(worldSeed, regionX, regionZ, salt)` for structure placement within coarser grids (Fabric Yarn API documentation, `ChunkRandom`). A critical limitation is that Java's `Random.setSeed()` truncates to 48 bits, so only the lower 48 bits of the world seed affect most generation features; biomes uniquely use the full 64 bits via the multi-noise system. Known seeding bugs illustrate the fragility: seed `107038380838084` causes the carver seed to evaluate to zero for all chunks along an axis, producing identical caves everywhere; seed `164311266871034` causes Z-axis repetition; and a structural weakness in the hash causes "mirrored caves" at sign-reversed coordinate pairs, affecting up to one-third of chunks in some worlds (minecraft.wiki, "Anomalous world seeds").

Cross-chunk features are handled through a two-phase reference system. During `structure_starts`, the originating chunk stores the full structure definition including a 3D bounding box. During `structure_references`, every neighboring chunk the structure extends into stores a packed 64-bit coordinate pointing back to the start chunk. During the `features` stage, each chunk can write blocks into a **3×3 chunk area** centered on itself — a hard limit that forces anything larger to use the structure system. This design creates a known failure mode: if chunks at the edge of the write area are generated or saved in unexpected order, features can be partially placed, producing half-generated trees or missing decorations (Mojang bug tracker, MC-188076, MC-224729). The fix in modern versions involves deferred block updates stored in the `PostProcessing` field of the chunk NBT, executed when the proto-chunk reaches `full` status. Sebastian Lague's tutorial series takes a simpler approach: because his terrain uses only continuous Perlin noise sampled at absolute world coordinates, height values at shared chunk edges are identical by construction — the noise function itself is the stitching mechanism (Lague, "Procedural Landmass Generation" E06–E07, GitHub). His chunk size of **241 vertices** (240 intervals) is chosen because 240 is evenly divisible by 2, 4, 6, 8, 10, and 12, enabling clean LOD mesh decimation at six levels.

The connection to earlier passes is direct: every noise evaluation from Pass 1a — fBm, domain warping, coordinate hashing — must be formulated as a function of absolute world coordinates, not chunk-local indices, or height discontinuities will appear at chunk edges. Erosion from Pass 1b becomes problematic because hydraulic erosion is fundamentally non-local; water flows across arbitrary distances, and chunk-local erosion produces visible boundary artifacts. The practical solutions are either to restrict erosion to pre-generated island maps where water terminates at ocean edges (3DWorld blog, "Terrain Erosion," 2017) or to run erosion as a global pre-pass at coarser resolution and use the result as a constraint for chunk-local detail. Cave carving from Pass 2a must use the same coordinate-seeded approach — Minecraft's carver seeds derive directly from world seed plus chunk coordinates. Structure-first dungeon generation from Pass 3a must account for the structure_starts/structure_references protocol if dungeons can span multiple chunks.

## 2. Hierarchical world-to-local generation

A flat chunk grid treats every spatial cell identically, but real game worlds have structure at multiple scales: continents, climate zones, biome regions, individual terrain features, prop placement. Without a hierarchy, each chunk must independently rediscover its continental context, biome membership, and regional parameters — duplicating computation and risking inconsistency. Hierarchical generation solves this by organizing production into nested scales, where each level constrains the next: the overworld level determines continent shapes and climate, the regional level assigns biomes and major features like rivers, and the local level fills in terrain detail and props. Information flows strictly downward as constraints and upward as aggregated statistics.

Red Blob Games' mapgen2 demonstrates a clean pipeline at a single regional scale. The system begins with Poisson-disc point sampling to generate ~1,000 sites, constructs a Delaunay triangulation via the Delaunator library, and derives the dual Voronoi mesh — producing a graph where polygon centers (regions) carry gameplay-relevant properties and polygon corners carry geometric detail like river paths. The pipeline then proceeds: assign water/land to corners using noise or radial functions → compute elevation as BFS distance from coast → redistribute elevation to match a target cumulative distribution (`y = 1 - (1-x)²`) → pick river start points in mountains and follow steepest-descent paths corner-to-corner → compute moisture as distance from fresh water → assign biomes via a Whittaker-style lookup on elevation × moisture (Patel, "Polygon Map Generation for Games," redblobgames.com). Mapgen4 inverts this ordering: elevation is painted interactively first, then a wind simulation propagates humidity along a configurable wind vector, orographic precipitation falls when air hits mountains creating rain shadows, and rivers are carved uphill from oceans to guarantee drainage without local minima (Patel, mapgen4 GitHub; Patel, "Mapgen4 Rainfall," simblob.blogspot.com). The architectural lesson is that **pipeline ordering encodes design priorities**: mapgen2's coast-first approach guarantees island shapes; mapgen4's elevation-first approach prioritizes terrain realism.

No Man's Sky extends this to six scales: a single universe seed cascades through galaxy structure → solar system parameters → planet terrain → local voxel detail → flora/fauna population. Each scale derives its seed from the spatial position at the parent scale, ensuring determinism without storing state. Dwarf Fortress uses a parameter-driven approach at world scale: five continuous fields (elevation, rainfall, temperature, drainage, volcanism) are generated via noise with configurable min/max and variance, then erosion, rivers, biome classification, and geological layering are applied sequentially, followed by a history simulation that tracks civilizations over centuries (Dwarf Fortress Wiki, "World generation"). The AMD GPUOpen Work Graphs sample demonstrates a GPU-native hierarchy: a world-level entry node projects the camera frustum onto the ground plane and launches chunk thread groups; each chunk classifies into biomes via Perlin noise weight maps and dispatches biome-specific tile nodes; tile nodes generate terrain mesh and vegetation; detail nodes add grass and particles (AMD GPUOpen, "Procedural Generation with Work Graphs," 2024).

The principal failure mode is **constraint violation across scales**: if the local generator ignores regional constraints (e.g., placing desert vegetation in a tundra biome), the world feels incoherent. The fix is to pass constraints explicitly — biome ID, elevation range, moisture budget, allowed feature set — as parameters into the local generator, and to validate outputs against these constraints. A subtler failure is **scale mismatch**: if the overworld biome map is sampled at 1 km resolution but the local generator operates at 1 m resolution, biome boundaries will be unnaturally sharp unless blended (see technique 6). The connection to earlier passes is that every local technique — fBm terrain, WFC tile placement, structure-first dungeons, prop scattering — becomes a *callee* of the hierarchical system. The hierarchy decides *which* generator to invoke and with *what parameters*; the local generator produces geometry within those constraints.

## 3. Coordinate-based deterministic regeneration

This technique overlaps substantially with coordinate hashing and seeding from Pass 1a, so this section flags the connection and covers only what changes in a streaming context. The core requirement is that discarding and regenerating a chunk must produce bit-identical results — if it does not, players will notice terrain shifting, items vanishing, or structures rearranging when they revisit an area. In a non-streaming context, coordinate hashing simply provides spatially varying randomness. In a streaming context, it becomes a **correctness invariant**: the hash function is the contract that the world is stable across load/unload cycles.

The two paradigms are hash-based (stateless, order-independent) and sequential-PRNG-based (seeded once, consumed in fixed order). Hash-based approaches — evaluating a continuous noise function at absolute coordinates, or applying a hash like MurmurHash or xxHash to `(worldSeed, chunkX, chunkZ)` — are inherently safe because they have no internal state that depends on evaluation order. Sequential PRNG approaches, like Minecraft's LCG-based `ChunkRandom`, are fragile: inserting a new `nextInt()` call during a version update shifts every subsequent random value, changing the world. Minecraft mitigates this by using separate PRNG instances per generation phase (population, decorator, carver, region), each re-seeded from coordinates, so changes to one phase do not cascade into others (Fabric Yarn API, `ChunkRandom`). The failure mode unique to streaming is **non-determinism from generation order**: if feature placement in chunk A can write into chunk B (as in Minecraft's 3×3 write area), and the result depends on whether B has already been generated, the output is path-dependent. Bug MC-188076 documented exactly this — jungle vegetation differed depending on which direction the player approached. The fix is to ensure that all cross-chunk writes use deterministic conflict resolution (e.g., last-write-wins with a fixed priority order) or to constrain cross-chunk writes to commutative operations.

## 4. Lazy generation and on-demand computation

Eager generation — producing the entire world at startup — is infeasible for large worlds and unnecessary for gameplay, since the player occupies only a small region at any moment. Lazy generation defers chunk production until something actually needs the data: a player approaching, a mob pathfinding query, or a structure placement check. This converts generation cost from O(world) at startup to O(visible) per frame, bounded by the player's movement speed and the system's generation throughput.

Minecraft implements this through a **ticket system** where each chunk has a load level (22–44) determined by the highest-priority ticket referencing it. Player tickets are level 31, created at the player's position with a radius based on render distance. Unknown tickets (level 33+, expiring after 1 tick) are auto-generated when game logic queries an ungenerated chunk. Force-loaded tickets keep spawn chunks permanently loaded. Load levels map to processing tiers: level ≤ 22 enables full entity and block ticking; level ≤ 32 enables block ticking only; level ≤ 33 enables entity access for generation; above 33, the chunk is inaccessible and can be unloaded. Background generation runs on a thread pool sized at `(available CPUs - 1)`, capped at 255 (minecraft.wiki, "Chunk"). Sebastian Lague's implementation is simpler but illustrates the essential pattern: each frame, the system computes the viewer's chunk coordinate, iterates all coordinates within the visible radius, looks up each in a `Dictionary<Vector2, TerrainChunk>`, and creates new chunks on cache miss. A **movement threshold** of 25 world units prevents redundant recalculation. Chunk heightmaps and LOD meshes are generated on background threads using a request/callback pattern: `RequestMapData(centre, callback)` spawns a thread that enqueues results into a `Queue<MapThreadInfo>` protected by a lock; the main thread dequeues and invokes callbacks during `Update()` (Lague, E08, GitHub).

The key variants are **priority-ordered queues** (Just Cause's streaming system processes high-priority stream items first, switching priorities only when the next item would cause a disk seek; Game Developer, "Streaming for Next Generation Games") and **coroutine-throttled composition** (limiting mesh uploads to N per frame to prevent stutter). The principal failure mode is **generation stall**: if the player moves faster than chunks can be produced, they see void or placeholder geometry. Fixes include predictive generation (pre-generating chunks in the player's velocity direction), reducing generation cost per chunk (fewer noise octaves, simpler features), and LOD fallback (showing a low-detail version while the full chunk generates). The connection to earlier passes is that every generator invoked lazily must be stateless with respect to evaluation order — erosion, river flow, and other non-local processes must either be pre-computed or constrained to operate within a single chunk's data.

## 5. LOD-aware procedural generation

When terrain stretches to the horizon, rendering every chunk at full resolution wastes GPU cycles on detail the player cannot perceive. LOD-aware generation produces multiple representations of each chunk at different detail levels, displaying coarser meshes at greater distances. Without it, the system must choose between a hard render-distance cutoff (Minecraft's approach: chunks beyond the limit simply vanish behind fog) or unacceptable frame times from rendering millions of triangles.

Sebastian Lague's series implements LOD through vertex skipping: at LOD level *n*, the mesh samples every `2^n`-th vertex, so LOD 0 uses every vertex (full detail), LOD 1 skips every other vertex, LOD 2 every fourth, and so on. The vertices-per-line formula is `(dataSize - 1) / step + 1`. Each `TerrainChunk` maintains an array of `LODMesh` objects, one per level, generated lazily — a mesh is only computed when the chunk actually needs that LOD. LOD selection occurs per-chunk based on the viewer's distance to the chunk's nearest edge; a configurable `LODInfo[]` array maps distance thresholds to LOD indices. Collision meshes are generated only for LOD 0 chunks (Lague, E09–E10, GitHub). This approach uses **instant mesh swaps** with no interpolation, producing visible popping at LOD transitions. The thresholds are set far enough that the geometric difference is barely perceptible, but the artifact is real.

More sophisticated approaches eliminate popping entirely. **CDLOD** (Continuous Distance-Dependent LOD, Strugar 2009) organizes the heightmap into a quadtree and renders all nodes using a single reusable grid mesh displaced by the heightmap in the vertex shader. The key innovation is **per-vertex morphing**: every grid vertex with an odd index (`i % 2 == 1` or `j % 2 == 1`) is a "morph vertex" that smoothly interpolates toward its even-indexed neighbor as the viewer moves away. The morph factor is computed as `clamp((dist - low) / (delta * 0.5) - 1.0, 0, 1)`, where `low` and `high` are the LOD range boundaries. When the morph factor reaches 1.0, the higher-detail mesh is geometrically identical to the lower-detail mesh, so the LOD switch is invisible. This eliminates T-junctions entirely — no skirts or stitching strips are needed (Strugar, "CDLOD," aggrobird.com/files/cdlod_latest.pdf; svnte.se, "CDLOD Terrain"). **Geometry clipmaps** (Losasso and Hoppe, 2004) take a different approach: concentric nested regular grids around the viewer, each ring at progressively lower resolution, with GPU-smoothed transitions between rings. **Chunked LOD** (Ulrich, SIGGRAPH 2002) uses a quadtree of pre-meshed chunks with vertical **skirts** extending downward along every edge to hide cracks between differing LOD levels (vterrain.org, "LOD Papers").

A less-discussed variant is **LOD-dependent noise octave counts**: distant chunks generate heightmaps with fewer fBm octaves, saving CPU time since the fine-detail octaves contribute sub-pixel variation at those distances. The danger is that removing octaves changes the aggregate height — each octave contributes to the sum — so a chunk's silhouette can shift as the player approaches and more octaves are added. The fix is to ensure the lowest-octave-count version provides a stable base and that additional octaves are purely additive detail with bounded amplitude. The connection to earlier passes: every noise function from Pass 1a must support partial octave evaluation; every mesh-generation technique from Passes 1b–2a must accept a resolution parameter; and every placement system from Pass 5a must handle the case where the underlying terrain mesh is an approximation of the full-detail version.

## 6. Boundary matching between chunks or regions

Boundary matching is the most failure-prone aspect of chunk-based generation because it sits at the intersection of multiple independent systems, each capable of introducing its own class of seam. The problem has four distinct sub-cases: height seams between same-LOD chunks, normal/lighting seams at chunk edges, geometric cracks between different-LOD chunks, and content discontinuities where non-local processes (erosion, biome blending, WFC propagation) cross chunk boundaries. Without systematic boundary treatment, every edge in the chunk grid is a potential visual artifact.

**Height seams** between same-LOD adjacent chunks arise when terrain height is computed using chunk-local information rather than global coordinates. If chunks normalize their heightmaps independently (mapping local min/max to [0,1]), adjacent chunks with different extremes will have mismatched heights at shared edges. The fix is fundamental: evaluate all noise functions at absolute world coordinates and use **global normalization** — estimating theoretical min/max from noise parameters (persistence and octave count) rather than actual per-chunk extremes. Lague's implementation estimates the global maximum as the geometric series sum `Σ persistence^i` for all octaves and normalizes against this bound, accepting some dynamic range loss in exchange for cross-chunk consistency (Lague, E11; medium.com/@liux4989). The deeper principle is that **any function used for terrain height must be continuous across chunk boundaries by construction**, which rules out per-chunk random seeds for height (only for feature placement).

**Normal seams** are more insidious because they occur even when height values match perfectly. Per-mesh normal calculation means edge vertices compute normals from only intra-chunk neighbors, producing different surface normals on each side of a boundary — visible as a lighting discontinuity. The fix is the **extended grid technique**: each chunk generates one extra row and column of height data in every direction, computes normals using this extended neighborhood, then trims to the actual chunk bounds for the mesh. This ensures edge normals account for cross-chunk geometry. The cost is duplicated height samples at boundaries, which can be mitigated by caching (Travis Fort, "Procedural Terrain Stitching," travisfort.net).

**LOD boundary cracks** — T-junctions — occur when a high-resolution chunk is adjacent to a low-resolution chunk. The high-res edge has vertices at positions where the low-res edge has only an interpolated surface, producing gaps. Five approaches exist in order of increasing sophistication:

1. **Skirts**: vertical geometry extending downward along every chunk edge, hiding cracks behind opaque walls. Simple but wastes geometry (Ulrich, Chunked LOD).
2. **Degenerate triangles**: on the higher-res side, collapse extra edge vertices to coincide with the nearest low-res vertex, creating zero-area triangles. Requires knowing the neighbor's LOD at mesh build time (GameDev.net forums).
3. **Precomputed index buffers**: keep vertex positions unchanged but swap in edge-specific index buffers for each of the 16 possible LOD-neighbor configurations (4 edges × 2 states each). Avoids runtime vertex modification.
4. **CDLOD morphing**: continuously morph the high-detail mesh to match the low-detail mesh before any LOD switch occurs. Eliminates T-junctions entirely because the higher-LOD mesh becomes geometrically identical to the lower-LOD mesh at the transition point. The constraint is that adjacent nodes may not differ by more than 1 LOD level (Strugar, CDLOD paper).
5. **Geometry clipmap stitching strips**: dedicated fill geometry between concentric rings, matched to both resolutions.

**Content discontinuities** from non-local processes are the hardest sub-case. Erosion simulation, which moves sediment across arbitrary distances, cannot be run independently per chunk without producing visible boundary artifacts — river valleys may go uphill at tile edges, and sediment accumulation will differ depending on generation order. Documented approaches include restricting erosion to enclosed island maps where water terminates at ocean edges (3DWorld blog, 2017), running erosion as a global pre-pass at reduced resolution and using the result to constrain chunk-local detail, and accepting boundary artifacts with post-hoc blending. For **WFC-based tile placement**, constraint propagation inherently crosses chunk boundaries, making naive chunking impossible. Marian42's solution is elegant: pre-generate a finite tileable base map offline (8×8×8 with wrapping boundaries), then pre-generate many replacement patches sharing the same boundary profile. At runtime, patches are selected deterministically by seed per chunk position. Crucially, runtime patches are generated at a **4-block offset** so that the shared boundary stripe is always overwritten, placing the varied interior away from edges (marian42.de, "Infinite WFC," 2023). For **biome blending**, KdotJPG's normalized sparse convolution method distributes data points on a jittered triangular grid (globally, not per-chunk), weights each point's influence by `max(0, r² - d²)²`, and normalizes dynamically. Because the point distribution is global, no grid-aligned artifacts appear at chunk boundaries (noiseposti.ng, "Fast Biome Blending Without Squareness," 2021).

The connection to earlier passes is pervasive. Heightmap construction from Pass 1a must use globally continuous functions. Erosion from Pass 1b must either operate globally or accept boundary constraints. River systems from Pass 1b must flow consistently across chunk boundaries — Minecraft solves this by computing river paths at the biome scale (coarser than chunks) during the `biomes` stage. Cave carving from Pass 2a must use coordinate-seeded carvers that respect the same absolute-coordinate contract. WFC and grammar systems from Pass 4a require explicit boundary protocols. Biome placement from Pass 5a must use a blending strategy that spans chunk edges without chunk-local normalization.

## 7. Overworld graphs

When a game world has narrative structure — regions with distinct identities, progression between them, gates and shortcuts — a flat grid of chunks is insufficient. The generator needs an abstract topology: which regions exist, how they connect, what role each plays in the player's journey. An **overworld graph** provides this by representing regions as nodes and connections (roads, portals, mountain passes) as edges, decoupling the *logical structure* of the world from its *geometric realization*. Without this layer, region connectivity is accidental — determined by whatever terrain the noise function happens to produce — and designing a coherent progression through the world becomes impossible.

The construction of an overworld graph typically follows one of two paradigms. The **graph-first** approach builds abstract topology before any geometry exists. BambooleanLogic's overworld generator starts with a small cycle of nodes, then iteratively adds **graph components**: cycle components (3–5 nodes forming loops) and branch components (a hub with 3–4 leaf nodes). A complexity parameter controls how many components are added; a desired-level-count parameter pads with individual nodes; and a **connectivity parameter** determines large-scale loop density by connecting distant nodes — zero connectivity yields a pseudo-linear world, one connectivity produces a roughly circular layout with two main paths, and two connectivity creates a figure-eight (BambooleanLogic, "Procedural Overworld Generation," 2020/2022). The **mesh-first** approach, exemplified by Red Blob Games' mapgen2, generates spatial points first (Poisson-disc sampling), constructs a Delaunay triangulation for adjacency and its dual Voronoi diagram for region shapes, then assigns properties (biome, elevation) to the resulting polygons. Amit Patel describes the design principle explicitly: "I used a graph structure to model the things directed by gameplay constraints (elevation, roads, river flow, quest locations, monster types) and noise functions to model the variety not constrained by gameplay" (Patel, "Polygon Map Generation for Games," redblobgames.com). The MST-plus-reconnection approach, documented by the TinyKeepDev algorithm, offers a middle path: generate room positions, build a Delaunay triangulation for maximum connectivity, extract the **minimum spanning tree** for guaranteed minimal connectivity, then add back ~15% of non-MST edges to create loops and alternative paths (Game Developer, "Procedural Dungeon Generation Algorithm").

The failure mode is **disconnected subgraphs**: if the graph construction process can produce isolated components, some regions become unreachable. The MST extraction guarantees connectivity by construction — any spanning tree of a connected graph visits every node. For graph-grammar approaches (technique 8), the starting graph must be connected and every rewrite rule must preserve connectivity as an invariant. Validation via flood fill from the start node is a cheap post-generation check. The connection to earlier passes: the overworld graph determines biome assignment (Pass 5a) — each node carries a biome type that constrains local terrain generation. Road placement (Pass 5b) follows graph edges. Dungeon entrances (Pass 3a) are placed at graph nodes marked for dungeon content. Region partitioning (Pass 5a) must respect graph boundaries so that each node's territory forms a contiguous spatial region.

## 8. Critical path generation and optional branching

A world with interesting topology but no designed progression feels aimless. Critical path generation ensures that there exists at least one **guaranteed traversable route** from the starting region to the goal, while optional branches provide exploration, rewards, and variety. Without an explicit critical path, the generator may produce worlds where the goal is trivially adjacent to the start, or where required items are locked behind circular dependencies.

Joris Dormans' graph grammar approach is the foundational algorithm. Generation begins with a trivial mission graph — `Start → Task₁ → Task₂ → ... → Taskₙ → Goal` — and iteratively applies rewrite rules to grow complexity. Each rule has a left-hand side (a subgraph pattern to match) and a right-hand side (the replacement). The core rules, as formalized in Dormans and Bakkes (2011):

1. **Reorganize**: match `A → B`; replace with a fork where `A` branches into two parallel paths that reconverge at `B`. This creates optional routing without changing the task count.
2. **Insert lock-and-key**: match any sequential pair; insert a `Key` node on one branch and a `Lock` node on another, connected by a gray "unlocks" edge. This creates a dependency: the player must find the key before passing the lock.
3. **Move lock forward**: match `Lock → X → Y`; replace with `X → Lock → Y`. This pushes the locked door further from the key, so the player encounters the door before finding the key — a classic design pattern.
4. **Move key backward**: match `X → Key → Y`; replace with `Key → X → Y`. This increases the exploration distance between key and lock.

The algorithm applies rules for a configurable number of iterations, selecting applicable rules randomly or by designer priority (Dormans and Bakkes, "Generating Missions and Spaces for Adaptable Play Experiences," IEEE TCIAIG 3(3), 2011). A critical invariant: **if rules are drafted correctly, the mission is always solvable without generate-and-test**. The key→lock gray edges are preserved through all transformations, maintaining the dependency chain.

Dormans' commercial implementation in **Unexplored** (2017) replaces the tree-based backbone with a **cyclic** one. The generator draws a circular loop on a 5×5 grid using ~40 rewrite rules, places an entrance and goal on the loop, and splits the loop into two arcs. One of **24 predefined major cycle types** determines how the arcs function: a "Lock-and-Key" cycle places a locked door at the goal on the short arc and the key near the end of the long arc; a "ThreeLeverHub" locks the goal behind three keys scattered around the loop; a "Hidden Shortcut" makes one arc the main path and reveals the other as a shortcut after completion (Boris the Brave, "Dungeon Generation in Unexplored," boristhebrave.com, 2021). Minor cycles are then added as detours from the main loop, increasing level size and complexity. This cyclic approach produces levels with a **meaningful arc** — the player progresses around the cycle, eventually returning near the start at a higher capability level — rather than the dead-end-heavy layouts typical of tree-based generators.

An alternative approach is Spelunky's 4×4 grid with a guaranteed random walk from entrance (top row) to exit (bottom row). Each cell on the walk path uses room templates with guaranteed vertical connectivity; cells off the path use templates with optional connectivity. This ensures a critical path exists while allowing organic branching (GDC 2011, Derek Yu on Spelunky level generation). Answer Set Programming (Smith, 2018) offers a constraint-satisfaction approach: the ASP solver finds configurations where all keys are reachable before their corresponding locks, validated by A* search tracking player inventory (Smith, "Graph-based Generation of Action-Adventure Dungeon Levels using ASP," PCG Workshop 2018).

The failure mode is **circular dependency**: key A requires key B to reach, and key B requires key A. The fix is topological ordering — keys must be placed at access levels strictly lower than their locks (where access level = number of gates between the node and the start). The connection to earlier passes: the critical path graph constrains dungeon layout from Pass 3a (the graph determines room connectivity before room geometry is generated), biome assignment from Pass 5a (regions along the critical path may be assigned biomes in difficulty order), and structure-first generation from Pass 3a (the mission→space transformation maps abstract tasks to physical rooms via spring-based force-directed layout, then elaborates with shape grammars).

## 9. Progression gating in generated spaces

Progression gating enforces that players acquire capabilities, items, or knowledge in a designed order, even when the physical space is procedurally generated. Without gating, a player can wander directly to end-game content, trivializing the experience. The core abstraction is the **lock-and-key model**, which generalizes far beyond literal keys and doors: a "lock" is any obstacle requiring a specific capability, and a "key" is whatever provides that capability. Big keys (unique items opening specific locks), small keys (consumable, interchangeable tokens), and ability keys (persistent capabilities like double-jump that open entire classes of locks) each impose different constraints on the generator.

The Metazelda algorithm (Tom Coxon, open-source) provides a practical implementation. It defines **key-levels**: the set of rooms reachable with exactly *n* keys but not *n-1* forms key-level *n*. Generation builds a spanning tree starting from the entrance at key-level 0, iteratively adding rooms and occasionally incrementing the key-level. The key invariant is that **for every key-level *n*, the corresponding key must be placed at some key-level *m < n***. After the spanning tree is built, switch puzzles are added: switches that lock/unlock branching paths, placed in rooms not descended from the switch-affected rooms to avoid circular dependencies. A "graphify" phase then adds non-tree edges between rooms: rooms with identical preconditions share unconditional edges, while rooms differing by one symbol get conditional edges. The algorithm retries up to 20 times if constraints cannot be satisfied (Coxon, Metazelda, GitHub). Dead Cells demonstrates a hybrid industrial approach: the macro structure — which regions connect and where keys gate progression — is entirely **hand-designed** and fixed across runs. Only the interior of each region is procedurally assembled from hundreds of hand-crafted room tiles, selected to comply with per-biome constraints specifying level length, labyrinthine degree, and entrance-to-exit distance (Motion Twin, "Building the Level Design of a Procedurally Generated Metroidvania," Game Developer).

The key failure mode is the **soft-lock**: a state where the player has consumed resources (small keys) in a suboptimal order and can no longer reach a required area. For consumable keys, the generator must prove that **no traversal ordering leads to a dead end**, which is equivalent to showing that the number of accessible keys always exceeds the number of locks on any path from start to goal. For ability-based gating (metroidvania style), the constraint is simpler: abilities are persistent, so the generator only needs to ensure abilities are acquirable before their corresponding locks on the critical path. Verification approaches include A* search with inventory tracking (simulate a full playthrough counting keys and abilities at each step), topological analysis of access levels, and exhaustive state-space search for small graphs. The connection to earlier passes: gating constrains dungeon layout from Pass 3a (rooms containing keys must be placed in accessible locations), biome assignment from Pass 5a (progression-gated regions may receive escalating biome difficulty), and WFC/grammar systems from Pass 4a (room templates must include lock/key furniture consistent with the progression graph).

## 10. Region connectivity and world traversability

The most fundamental requirement for a procedurally generated world is that **every required region is reachable**. A disconnected world — one where the player cannot reach the goal from the start — is a hard failure. Region connectivity analysis ensures this property holds and provides tools for controlling *how* connected the world is: minimally connected (tree-like, forcing specific routes), moderately connected (with alternative paths), or highly connected (open exploration with many routes).

The standard algorithmic approach is **Delaunay-to-MST-to-selective-reconnection**. Given a set of region positions, construct the Delaunay triangulation (which maximizes the minimum angle of all triangles, producing well-shaped connections). Extract the minimum spanning tree, which guarantees that all regions are reachable with the minimum total edge weight — this is the most "efficient" connected graph. Then add back a fraction of non-MST Delaunay edges (typically **10–15%**) to create loops, alternative routes, and shortcuts. The fraction controls the exploration character: 0% yields a pure tree (linear, no choices), 15% yields moderate looping, and 100% recovers the full Delaunay graph (maximum connectivity). The TinyKeepDev algorithm popularized this approach for dungeon rooms (Game Developer, "Procedural Dungeon Generation Algorithm"), but it applies equally to overworld regions. Ondřej Nepožitek's Edgar generator takes a complementary approach: the designer provides a connectivity graph (nodes = rooms, edges = required connections) as input, and the algorithm assigns room shapes and positions such that no rooms overlap and adjacent rooms connect via doors. The algorithm supports cycles — unusual for graph-based generators due to the combinatorial difficulty of placing rooms in non-overlapping configurations that satisfy cyclic connectivity — and uses backtracking when placement fails (Nepožitek, "Graph-Based Dungeon Generator," ondra.nepozitek.cz).

Validation is straightforward: after constructing the connectivity graph, run a **flood fill** (BFS/DFS) from the designated start node. If any required node is unreachable, the graph is invalid. For graphs with one-way connections (portals, drops), reachability becomes directed: use DFS on the directed graph to confirm that the goal is reachable from the start. A more nuanced check for worlds with progression gating is **gated reachability**: simulate a traversal where the player acquires keys/abilities as they become available, and verify that each gate can be passed when reached. This is equivalent to a topological sort of the dependency graph overlaid on the connectivity graph.

The failure modes are **isolated subgraphs** (a region or cluster of regions with no edges to the rest of the world) and **one-way traps** (the player can enter a region but not leave, and the region does not contain the goal). Isolated subgraphs are prevented by construction when using MST extraction. One-way traps require directed-graph analysis: for every region reachable from the start, verify that the goal is reachable from that region (strongly connected component analysis or per-node DFS). The connection to earlier passes: region connectivity determines where roads from Pass 5b are placed (roads follow graph edges between connected regions), where biome boundaries from Pass 5a fall (connected regions of the same biome type can be merged into contiguous territories), and how dungeon entrances from Pass 3a are distributed (each graph node marked for dungeon content receives exactly one entrance placed according to the spatial realization of that node).

## Multi-scale pipeline patterns

In a typical streaming world, the techniques in this pass compose into a layered pipeline that proceeds from the most abstract, largest-scale decisions down to the finest local detail. **First**, an overworld graph (technique 7) is generated once at world creation, defining region nodes, their connections, biome assignments, and any critical path or progression gating (techniques 8–9). This graph is small enough to hold in memory permanently and serves as the top-level authority for all subsequent generation. **Second**, when the player enters a region, the hierarchical system (technique 2) resolves that region's parameters — elevation range, moisture budget, feature set, dungeon count — from the overworld graph and feeds them as constraints to the chunk-level generators. **Third**, chunks are generated lazily on demand (technique 4) using coordinate-seeded determinism (technique 3), invoking the appropriate local generators from earlier passes: fBm terrain (Pass 1a), erosion (Pass 1b, pre-computed or bounded), cave carving (Pass 2a), tile placement (Pass 4a), biome-appropriate props (Pass 5a). **Fourth**, LOD-aware generation (technique 5) produces multiple mesh resolutions per chunk, and boundary matching (technique 6) ensures seamless stitching across every chunk edge and LOD transition. Region connectivity validation (technique 10) runs as a post-check on the overworld graph to guarantee traversability before any spatial realization begins.

The most common architectural mistakes occur at the interfaces between layers. The first is **leaking global state into chunk generation**: any generator that reads or writes state beyond its own chunk coordinates (e.g., a river simulation that modifies downstream chunks) violates the independence assumption and creates order-dependent results. The fix is to compute non-local features at a coarser hierarchical level and pass results down as read-only constraints. The second is **ignoring the overworld graph during local generation**: chunks that generate terrain without consulting their region's biome assignment or progression role produce incoherent worlds. The third is **insufficient boundary matching**: many teams get height continuity right (by using absolute-coordinate noise) but neglect normal seams, LOD cracks, WFC boundary propagation, or biome blending — each of which requires its own dedicated solution. These integration challenges are the focus of Pass 6b's composition patterns.
