# Propagation and grammar techniques for procedural content generation

**Constraint propagation, Wave Function Collapse, L-systems, and shape grammars share a defining feature that separates them from local rule-based approaches: they enforce structure beyond the immediate neighborhood of a single tile or module.** Local systems like autotiling and socket-based placement ask only "does this piece fit its neighbors?" — a question answered by examining one boundary at a time. The techniques in this document either cascade constraint information across an entire domain until a globally consistent assignment emerges (constraint propagation, WFC) or generate content through recursive rewriting rules that encode hierarchical structure in their very grammar (L-systems, shape grammars). This difference is what enables WFC to produce tilemap outputs where every local neighborhood is valid simultaneously, and what enables an L-system to grow a botanically plausible tree from a single axiom. The cost is algorithmic complexity: these techniques introduce contradiction, exponential growth, and non-termination as failure modes that local systems never encounter.

---

## 1. Constraint propagation as the engine beneath WFC

Constraint propagation is not a generation algorithm — it is a **domain-reduction mechanism** that removes impossible values from variables until only mutually consistent assignments remain. Where a local rule-based system checks one adjacency at a time and accepts or rejects a single placement, constraint propagation cascades the consequences of every removal across the entire problem, potentially eliminating possibilities many cells away from the original change. This is what allows WFC to maintain coherence across a full output grid rather than just at each boundary. Any game context where "locally valid everywhere" must hold simultaneously — seamless tilemaps, architecturally plausible 3D structures, puzzle grids with global invariants — requires this kind of propagation.

The formal framework is a **Constraint Satisfaction Problem (CSP)**: a set of variables (grid cells), each with a finite domain of possible values (tiles), linked by constraints specifying which value combinations are permitted (adjacency rules). Constraint propagation reduces domains by enforcing **arc consistency**: a directed arc from variable X to variable Y is arc-consistent if, for every value remaining in X's domain, at least one compatible value exists in Y's domain. The classic **AC-3 algorithm** (Mackworth, 1977) maintains a worklist of arcs to check. When a value is removed from a domain, all arcs pointing to that variable are re-added to the worklist. The process terminates when the worklist empties (arc-consistent state reached) or any domain becomes empty (contradiction — no solution exists). AC-3 runs in **O(e·k³)** time where e is the number of constraint edges and k is the maximum domain size (Boris the Brave, "Arc Consistency Explained," boristhebrave.com, 2021). The more efficient **AC-4 algorithm** (Mohr & Henderson, 1986) maintains explicit support counts — for each variable/value/direction triple, a count of how many values in the neighboring variable's domain are compatible. When a value is removed, support counts for affected neighbors are decremented; any count reaching zero triggers removal of the now-unsupported value. AC-4 achieves optimal **O(e·k²)** worst-case complexity at the cost of higher initialization overhead, and is the algorithm used by most modern WFC implementations (Gumin, WFC README, github.com/mxgmn/WaveFunctionCollapse).

Constraint propagation alone is **incomplete** — it reduces domains but may leave multiple values per variable without resolving the problem. A complete solver requires combining propagation with **search**: making a guess (collapsing a variable to one value), propagating consequences, and backtracking if a contradiction results. This combination is called **Maintaining Arc Consistency (MAC)** in CSP literature (Dechter, UCI constraint processing textbook). WFC is precisely this pattern — the "observe" step is the guess, the "propagate" step is arc consistency enforcement — except that Gumin's original WFC omits backtracking entirely, relying on the empirical observation that well-designed tilesets rarely produce contradictions (Gumin, WFC README). This makes WFC a constraint-based *generator* rather than a constraint *solver*: the key insight, attributed by Boris the Brave to Paul Merrell's Model Synthesis work, is that any constraint solver can be turned into a generator by replacing deterministic variable/value selection with randomized, frequency-weighted selection (Boris the Brave, "Model Synthesis and Modifying in Blocks," boristhebrave.com, 2021).

The failure mode is **contradiction**: propagation drives a domain to empty, meaning no consistent assignment exists given prior choices. The proximate cause is that arc consistency is a local approximation — it cannot detect inconsistencies that require reasoning about three or more variables simultaneously. Higher levels of consistency (path consistency, k-consistency) catch more failures but are exponentially more expensive. The practical fix in WFC contexts is either full restart, backtracking, or Merrell's "modifying in blocks" strategy (discussed in the WFC deep dive below). Constraint propagation connects directly to WFC (which is built on top of it), to Answer Set Programming (which uses SAT/SMT solvers for complete constraint resolution), and to grammar-based systems when those grammars are augmented with constraint checks during derivation — as in Dormans' constraint-based graph grammars (CEUR-WS Vol. 4090, 2024).

## 2. Wave Function Collapse — simple tiled model and overlapping model

WFC produces **tilemap outputs where every local neighborhood matches patterns found in the input**, a guarantee that no local rule-based system provides because local systems reason about one placement at a time without propagating consequences globally. The technique was introduced by Maxim Gumin in 2016 as a reimplementation and extension of Paul Merrell's 2007 Model Synthesis algorithm (Gumin, WFC README). WFC transforms texture synthesis into a constraint satisfaction problem and solves it using arc consistency propagation combined with entropy-guided random collapse. The result is outputs that are locally indistinguishable from the input — every small rectangle of the output matches some small rectangle of the input — while exhibiting emergent global variety. Game contexts requiring seamless tiled environments, architectural layouts, or any scenario where hand-authored adjacency rules must hold everywhere simultaneously are the primary use cases. Caves of Qud was the first commercial game to ship with WFC (Bucklew, GDC 2019); Bad North and Townscaper followed with increasingly sophisticated adaptations.

### The simple tiled model

The simple tiled model takes as input a **list of tile types with weights** (occurrence frequencies) and **explicit adjacency rules** specifying which tiles may be placed beside which other tiles in each cardinal direction. Adjacency rules are typically authored manually or derived from socket/edge-label matching. The output is a 2D grid where every cell contains exactly one tile and every adjacent pair satisfies the rules. The core data structures are:

```
wave[x][y][t]       — boolean; true if tile t is still possible at cell (x,y)
support[x][y][d][t]  — integer; count of tiles in the neighbor cell in direction d
                       that are compatible with tile t at cell (x,y)
weights[t]           — float; frequency weight of tile t
propagation_stack    — stack of (x, y, removed_tile) entries to process
```

Initialization sets every `wave[x][y][t]` to `true` and computes initial support counts: for each cell, direction, and tile, count how many tiles in the adjacent cell's full domain are compatible (Boris the Brave, "WFC Explained," boristhebrave.com, 2020). The algorithm then repeats two phases:

**Observe.** Find the cell with **minimum nonzero Shannon entropy**, calculated as:

```
H(cell) = log(W) − (Σ wₜ·log(wₜ)) / W
```

where wₜ is the weight of each tile still possible at that cell and W = Σ wₜ (gridbugs.org, "Wave Function Collapse"). This formulation avoids recomputing normalized probabilities and is incrementally updatable as tiles are removed. A small random noise term is added to break ties (Gumin, WFC README). Collapse the selected cell by choosing one tile via weighted random selection from its remaining possibilities, then removing all other tiles from its domain. For each removed tile, push an entry onto the propagation stack and update support counts in neighboring cells.

**Propagate.** While the propagation stack is non-empty, pop an entry (x, y, removed_tile). For each direction d, examine the neighbor cell (nx, ny). For each tile t′ still possible at (nx, ny), check whether removed_tile was compatible with t′ in direction d. If so, decrement `support[nx][ny][opposite(d)][t′]`. If this count reaches zero, t′ has no remaining support from (x, y)'s direction and must be removed from (nx, ny)'s domain: set `wave[nx][ny][t′] = false`, update (nx, ny)'s entropy and weight sums, push (nx, ny, t′) onto the propagation stack, and update support counts for (nx, ny)'s other neighbors in turn. If any cell's domain becomes empty, a **contradiction** has occurred (Boris the Brave, "WFC Explained").

The full observe-propagate pseudocode:

```
initialize all cells to full domain; compute support counts
while any cell has domain size > 1:
    cell = argmin { H(c) + noise : domain_size(c) > 1 }
    chosen = weighted_random_choice(possible tiles at cell)
    for each tile t ≠ chosen in cell's domain:
        remove t from cell's domain
        push (cell.x, cell.y, t) to propagation_stack
    
    while propagation_stack is not empty:
        (x, y, removed) = pop from propagation_stack
        for each direction d in {up, down, left, right}:
            (nx, ny) = neighbor(x, y, d)
            if (nx, ny) is out of bounds: continue
            for each tile t' compatible with removed in direction d:
                if wave[nx][ny][t'] is false: continue
                support[nx][ny][opposite(d)][t'] -= 1
                if support[nx][ny][opposite(d)][t'] == 0:
                    wave[nx][ny][t'] = false
                    if domain_size(nx, ny) == 0: CONTRADICTION
                    push (nx, ny, t') to propagation_stack
                    // update entropy tracking for (nx, ny)

output: wave array where each cell has exactly one tile remaining
```

This is an **AC-4** implementation — the support count array provides O(1) arc checks per tile removal rather than the O(k) scan required by AC-3. Termination is guaranteed because each propagation step can only decrease domain sizes; entropy never increases during propagation (Gumin, WFC README). The minimum entropy heuristic is equivalent to the **Minimum Remaining Values (MRV)** heuristic from CSP literature and was Gumin's key improvement over Merrell's scanline ordering, which introduced directional artifacts (Karth & Smith, FDG 2017).

### The overlapping model

The overlapping model replaces authored adjacency rules with rules **automatically extracted from an example image**. The input is a single bitmap; the algorithm slides an NxN window (typically N=2 or N=3) across every pixel position, extracting all NxN patterns and counting their frequencies. Optionally, each pattern is augmented with its rotations and reflections, yielding up to 8× more patterns, then duplicates are merged with accumulated weights (Gumin, WFC README). Two patterns A and B are compatible in direction d if, when B is placed one pixel offset from A in direction d, the overlapping (N−1)×N or N×(N−1) pixel region is identical. This compatibility is precomputed into a propagator structure: `propagator[direction][pattern_id] = list of compatible pattern IDs` (gridbugs.org, "Wave Function Collapse"). The observe-propagate loop is then identical to the simple tiled model, but operating over NxN patterns rather than individual tiles, with each output cell representing the top-left pixel of a pattern.

The practical consequences are significant. The overlapping model **captures N-range correlations** — it knows not just which tiles adjoin but which NxN neighborhoods are valid, making it sensitive to corners, junctions, and local texture. Gumin describes the relationship: "The overlapping model relates to the simple tiled model the same way higher order Markov chains relate to order one Markov chains" (Gumin, WFC README). The tradeoff is speed (many more patterns means a larger propagator and slower propagation), less authorial control (rules are learned rather than designed), and a tendency to reproduce the input too faithfully when N is large. The simple tiled model is better when a designer wants explicit control over adjacency; the overlapping model is better for texture synthesis or when working from example images. Caves of Qud uses the overlapping model with small 16×16 training images (Bucklew, GDC 2019); Bad North uses the simple tiled model with over 300 hand-authored 3D tiles (Stålberg, EPC 2018).

The primary failure mode for both models is **contradiction** — a cell's domain becoming empty during propagation. Gumin notes that "the problem of determining whether a certain bitmap allows other nontrivial bitmaps satisfying condition (C1) is NP-hard," but "in practice the algorithm runs into contradictions surprisingly rarely" (Gumin, WFC README). The proximate cause is that arc consistency is an approximation; it cannot detect all future dead-ends. Mitigation strategies include full restart (Gumin's original approach), backtracking with state restoration (Boris the Brave's DeBroglie library), and Merrell's "modifying in blocks" strategy for large outputs. Karth and Smith found that "shallow backtracking does little to help resolve conflicts" — stalling in unsolvable partial solutions is a real risk (Karth & Smith, IEEE Transactions on Games, 2022). WFC connects upward to constraint propagation (it is a specific CSP solver), laterally to Model Synthesis (its direct precursor), and downward to grammar-based systems when grammars are used to generate macro structure that WFC then fills with detail (the Caves of Qud multi-pass pipeline).

## 3. L-systems generate structure through parallel recursive rewriting

L-systems produce content that **grows from an axiom through iterative parallel rewriting** — a fundamentally different generative mechanism from constraint satisfaction. Where WFC fills a fixed grid with consistent tiles, an L-system starts with a seed string and expands it according to production rules applied simultaneously to every symbol, yielding recursive, self-similar, branching structures that no fixed-grid approach can naturally express. Plants, trees, road networks, river systems, dungeon layouts with fractal branching — any content where structure emerges from recursive growth rather than constraint satisfaction — is the natural domain of L-systems. The technique was formalized by Aristid Lindenmayer in 1968 for modeling biological development and was given its definitive computational treatment by Prusinkiewicz and Lindenmayer in *The Algorithmic Beauty of Plants* (1990; Chapter 1 available free at algorithmicbotany.org).

A **deterministic context-free L-system (DOL-system)** is a triplet G = ⟨V, ω, P⟩: an alphabet V, an axiom ω ∈ V⁺, and a set of productions P where each symbol a ∈ V has exactly one successor string χ, written a → χ (Prusinkiewicz & Lindenmayer, ABOP, p. 4). The critical distinction from Chomsky grammars is that **all productions are applied simultaneously in parallel** at each derivation step — every symbol in the current string is rewritten at once, rather than selecting one non-terminal to expand. Given a string µ = a₁a₂...aₘ, the derived string is ν = χ₁χ₂...χₘ where each aᵢ → χᵢ. The resulting string is interpreted geometrically through **turtle graphics**: the symbols F (draw forward), + (turn left by angle δ), − (turn right by δ), [ (push state onto stack), and ] (pop state) drive a virtual turtle that traces the output geometry. The 3D extension adds &/^ (pitch), \// (roll), and represents orientation as three orthogonal unit vectors H⃗, L⃗, U⃗ with rotation matrices (Prusinkiewicz & Lindenmayer, ABOP, §1.5). Implementation requires only a double-buffered string (read from buffer A, write to buffer B, swap after each step) and a turtle interpreter that walks the final string:

```
function derive(axiom, productions, iterations):
    current = axiom
    for i = 1 to iterations:
        next = empty string
        for each symbol s in current:
            if s in productions: append productions[s] to next
            else: append s to next
        current = next
    return current
```

The three major variants each add one capability. **Stochastic L-systems** assign multiple productions per symbol with probabilities summing to 1: a production is selected randomly each time the symbol is rewritten, introducing variation while preserving overall structural character. The formal definition adds a probability function π: P → (0,1] to the triplet, giving Gπ = {V, ω, P, π} (Prusinkiewicz & Lindenmayer, ABOP, §1.7). Implementation replaces the deterministic lookup with weighted random selection using a seeded PRNG for reproducibility. **Parametric L-systems** attach numeric parameters to symbols, forming modules like A(x, y), and add conditions and arithmetic expressions to productions: `A(x,y) : y > 3 → B(x)A(x/y, 0)` fires only when the condition holds and computes new parameter values (Prusinkiewicz & Lindenmayer, ABOP, §1.10). This enables continuous variation — branch lengths that shrink with depth, angles that widen with age, growth rates governed by the grammar itself. Implementation stores modules as `{symbol, params[]}` objects rather than characters; rules become functions that test conditions and compute output parameters. **Context-sensitive L-systems** make production application depend on neighboring symbols: the notation `b < a > c → aa` means "rewrite a as aa only when preceded by b and followed by c" (Prusinkiewicz & Lindenmayer, ABOP, §1.8). Context-sensitive rules take precedence over context-free ones when applicable. This models signal propagation — nutrients flowing through a plant, information traveling along a structure. All four variants can be combined; Houdini's L-system node supports the full combined syntax: `left_context < symbol > right_context : condition = replacement : probability` (SideFX Houdini docs, sidefx.com).

The dominant failure mode is **exponential string growth**: a production like F → F+F doubles the string length at each iteration, producing O(k^n) symbols after n iterations for average expansion factor k. After 15–20 iterations, strings can exceed practical memory limits (Wikipedia, "L-system"). Mitigation strategies include hard iteration caps (typically 3–8 for vegetation), recursive descent interpretation that expands symbols on-the-fly without materializing the full string (Zucker, mzucker.github.io, 2020), GPU parallelization, and Houdini's **meta-test** approach where conditional rules check whether the next branch will stay within a bounding volume before expanding (SideFX Houdini docs). A second failure mode is **loss of global structure**: because L-systems apply local rewriting rules, they cannot enforce global constraints like "rooms must connect" or "the tree must fit in this bounding box" without external mechanisms. Sean Barrett's influential critique demonstrated that the Parish & Mueller SIGGRAPH 2001 city-generation paper's "L-system" could be mechanically reduced to simple procedural code, arguing that for complex procedural systems the L-system formalism adds obscurity without benefit (Barrett, "L-Systems Considered Harmful," nothings.org). This critique holds when L-systems are being used as little more than notation for sequential procedures; it does not apply when the parallel rewriting and recursive self-similarity are genuinely exploited, as in vegetation generation. L-systems connect to WFC as a complementary technique: L-systems can generate macro-scale branching structure (road networks, dungeon skeletons) that WFC then fills with locally consistent tile detail, as demonstrated in the multi-pass pipeline approach.

## 4. Shape grammars and grammar-based generation encode design knowledge as rewriting rules

Shape grammars produce content by **recursively applying geometric transformation rules to spatial primitives** — they encode architectural and spatial design knowledge directly in the grammar, enabling the generation of structures that respect stylistic conventions, proportional systems, and hierarchical composition that neither WFC nor L-systems naturally capture. Where WFC ensures local tile consistency and L-systems grow recursive branching structures, shape grammars can express design intent like "every facade has a ground floor, a repeated middle section, and a cornice" or "every dungeon has a locked door preceded by a key." Game contexts requiring architecturally coherent buildings, stylistically consistent cities, or levels with guaranteed narrative structure benefit from grammar-based approaches.

The original formulation by Stiny and Gips (1972) defines a shape grammar as a tuple **SG = (S, L, R, I)**: a finite set of shapes S (geometric primitives), labels L (symbols restricting rule application), rules R of the form α → β where α and β are labeled shapes, and an initial shape I (Stiny & Gips, "Shape Grammars and the Generative Specification of Painting and Sculpture," IFIP Congress 1971). Rule application works by geometric **embedding**: the system searches for an occurrence of the left-hand side within the current working shape using Euclidean transformations (translation, rotation, reflection, scaling), then replaces it with the right-hand side. This geometric matching makes classical shape grammars computationally expensive but uniquely powerful for spatial reasoning — shapes can fuse and overlap, and subshapes can be recognized even when embedded in larger compositions. The process repeats until only terminal shapes remain. In practice, three rule types are needed: a start rule establishing the initial structure, transformation rules that refine non-terminal shapes, and termination rules that convert non-terminals to final geometry.

The most influential practical variant is the **CGA Shape grammar** (Müller et al., SIGGRAPH 2006), which replaces geometric embedding with **symbolic matching** — shapes carry explicit labels, and rules match on labels rather than geometry. This makes CGA Shape effectively a parametric set grammar rather than a true shape grammar in Stiny's sense, but it is vastly more practical. The core operation is **split**: `split(axis) { ratio₁ : Successor₁ | ratio₂ : Successor₂ | ... }` subdivides a shape along an axis into parts with relative or absolute sizes. A typical building generation pipeline starts with a lot polygon, extrudes to a mass model, decomposes into facades via component split, subdivides facades into floors, floors into bays, and bays into window frames and ornaments via terminal geometry insertion (Müller et al., SIGGRAPH 2006; CityEngine documentation, doc.arcgis.com). CGA Shape also supports context sensitivity through occlusion queries — rules can check whether geometry is occluded by other building volumes and choose different assets accordingly. CityEngine demonstrated the approach at city scale, generating virtual Pompeii with 150–190 rules per building style.

A parallel grammar tradition addresses **level structure rather than geometry**. Joris Dormans (2010) separated game levels into two independent structures — **mission** (the task graph the player must complete) and **space** (the geometric room layout) — and used **graph grammars** to generate missions and shape grammars to generate spaces (Dormans, "Adventures in Level Design," PCGames@FDG 2010). Graph grammar rules match subgraph patterns and replace them with expanded subgraphs, enabling non-linear structures (branches, hubs, parallel paths) impossible to express with string grammars. This approach powered the game Unexplored (Ludomotion, 2017), which used cyclic dungeon generation to create levels with loops and shortcuts that feel hand-designed. The key failure modes of grammar-based generation are **authoring difficulty** — "authoring level grammars is hard, even for tiny examples" (van Rozen & Heijn, PCG Workshop 2018) — and the risk of producing locally valid but globally problematic output (such as obstacles blocking required doors). Grammars are only as good as their rules, and creating rules that encode genuine design knowledge requires both domain expertise and extensive testing. Grammar systems connect to WFC through pipeline composition: Dormans' mission grammar can generate the structural skeleton, shape grammar rules can place rooms, and WFC can fill each room with locally consistent tile detail. They connect to constraint propagation when grammar derivations are augmented with constraint checks — as in constraint-based graph grammars that ensure generated levels are solvable by construction (CEUR-WS Vol. 4090, 2024).

## 5. WFC deep dive

### Constraint propagation from first principles

Boris the Brave frames WFC as a constraint programming problem before introducing any WFC-specific logic (Boris the Brave, "WFC Explained," boristhebrave.com, 2020). The foundation is a CSP with three components: **variables** (each cell in the output grid), **domains** (for each cell, a boolean array with one entry per tile, all initialized to true), and **constraints** (binary adjacency rules between neighboring cells). The goal of constraint propagation is to shrink domains by removing values that cannot participate in any valid complete assignment, without guessing. Arc consistency is the specific form of propagation used: for every cell and every tile still in that cell's domain, there must exist at least one compatible tile in each neighboring cell's domain. If no such tile exists, the original tile is impossible and must be removed.

The propagation mechanism works through cascading consequences. When a tile is removed from a cell's domain (either by initial constraint application or by collapse), the system examines every constraint involving that cell. For each neighboring cell, it checks whether any tiles in the neighbor's domain have lost their last source of support. Boris the Brave describes two critical implementation optimizations. First, a **propagation queue**: rather than scanning all constraints after every change, maintain a queue of cells whose domains have changed; process cells from this queue one at a time, checking only their adjacencies and adding newly affected cells back to the queue. Second, the **support count array**: `support[cell][direction][tile]` stores, for a given cell and tile, how many tiles in the adjacent cell (in the given direction) are compatible with that tile. When a tile is removed from a cell, the support counts in neighboring cells are decremented. When any count reaches zero, the corresponding tile must be removed, triggering further propagation. This is the AC-4 algorithm, which provides **O(1) per tile removal** rather than the O(k) per-arc check of AC-3 (Boris the Brave, "WFC Explained"). The propagation phase terminates because domain sizes can only decrease — no new possibilities are ever introduced during propagation.

Crucially, propagation alone is incomplete. After propagation reaches a fixpoint, many cells may still have multiple tiles in their domains. This is where the WFC-specific **observe** step intervenes: pick the most-constrained unresolved cell (minimum entropy heuristic), collapse it to a single tile via weighted random selection, then propagate again. This cycle of guess-then-propagate is the standard CSP solving strategy called Maintaining Arc Consistency (MAC). The innovation that makes this a *generator* rather than a solver is the randomization: tile selection is weighted by frequency from the input, making likely patterns appear more often while maintaining constraint satisfaction throughout.

### Simple tiled model versus overlapping model

The two WFC variants differ in **how adjacency rules originate, what constitutes a "tile," and the character of the output**. In the simple tiled model, the developer supplies a finite set of tile types, explicit weights, and an explicit list of permitted adjacency pairs per direction — typically derived from edge labels or socket IDs. Each cell in the output grid maps to exactly one tile. Propagation enforces that adjacent cells hold compatible tiles. The developer has full control over which adjacencies are legal, and the output character depends entirely on the tileset design.

In the overlapping model, the input is a **single example bitmap**. The algorithm extracts all NxN pixel patterns (typically N=2 or N=3) by sliding a window across the image, optionally augmenting with rotations and reflections, deduplicating, and recording frequencies as weights (Gumin, WFC README). Adjacency is determined automatically: two patterns are compatible in direction d if their overlapping (N−1)-pixel-wide strip is identical when one is placed one pixel offset from the other in that direction. Each output cell represents the top-left pixel of an NxN pattern, and the entropy calculation, collapse, and propagation phases are structurally identical to the simple tiled model — they just operate over the (typically much larger) set of NxN patterns rather than individual tiles.

The consequences for entropy and propagation behavior are significant. The overlapping model captures **N-range local correlations**: it knows not just that two tiles can adjoin but that entire NxN neighborhoods are valid. This makes propagation more aggressive — removing one pattern can cascade further because fewer patterns overlap compatibly — which means the model is both more faithful to the input texture and more prone to contradiction. Increasing N improves output quality but exponentially increases the pattern count and propagation cost. The simple tiled model, by contrast, has a smaller domain (just the authored tiles), faster propagation, and gives the designer explicit control — but it only captures pairwise adjacency and cannot enforce larger neighborhood patterns. The practical choice is straightforward: **use the simple tiled model when you have a designed tileset and want control; use the overlapping model when you have an example image and want to synthesize more of its texture** (Gumin, WFC README).

### Contradiction handling and backtracking

A contradiction occurs when propagation empties a cell's domain — no tile can legally be placed there given prior collapses. Gumin observes that this is fundamentally unavoidable because "the problem of determining whether a certain bitmap allows other nontrivial bitmaps satisfying condition (C1) is NP-hard" (Gumin, WFC README). Three strategies address contradictions at increasing cost and reliability:

**Full restart** (Gumin's original approach): discard the entire output and regenerate from scratch. This works when contradictions are rare and output size is small. It fails to scale because contradiction probability grows with output size — on sufficiently large grids, the algorithm may never complete. **Backtracking** (Boris the Brave's DeBroglie library): before each collapse, save the full state (domain arrays, support counts). On contradiction, restore the most recent saved state and remove the guessed tile from the collapsed cell's domain. Multiple saved states form a stack. DeBroglie implements this via a **changelog**: every domain change is logged, and backtracking reads the log in reverse, undoing each change (Boris the Brave, DeBroglie developer documentation, boristhebrave.github.io). The performance cost is memory (storing the full state or changelog at each guess) and, more critically, the risk of **stalling**: the algorithm may be stuck in a partial solution that is unsolvable but not detectably so, repeatedly exploring dead-end variations. Karth and Smith found that "shallow backtracking does little to help resolve conflicts" (Karth & Smith, IEEE Transactions on Games, 2022). **Modifying in blocks** (Merrell, Model Synthesis, 2009): partition the output into overlapping blocks, solve each independently, and restart only the failing block. All block borders are constrained to known-good tile arrangements (e.g., flat bedrock with air above), ensuring each block is self-contained. This avoids both the scaling problem of full restarts and the stalling problem of backtracking (Boris the Brave, "Model Synthesis and Modifying in Blocks," boristhebrave.com, 2021).

### Adding non-local constraints

WFC's core propagation only enforces local adjacency — it has no awareness of global properties like path connectivity, tile counts, or symmetry. Adding non-local constraints requires augmenting the base algorithm.

**Fixed tiles** are the simplest non-local constraint: before the main loop begins, pre-collapse specific cells to specific tiles and propagate. This pins entrance/exit points, boundary conditions, or hand-authored features, and WFC fills around them seamlessly (Boris the Brave, "WFC Tips and Tricks," boristhebrave.com, 2020). The cost is minimal — just one propagation pass at initialization. **Boundary conditions** restrict what tiles can appear at grid edges. DeBroglie's `BorderConstraint` calls `Ban` and `Select` on edge cells at startup, preventing structures from leading off the map edge (DeBroglie documentation). **Path connectivity** is Boris the Brave's most significant constraint contribution: a global constraint that forces a subset of "path tiles" to form a single connected component. Implementation requires backtracking, because the constraint is evaluated after each propagation step — if the current state could potentially leave path tiles disconnected, the constraint triggers `SetContradiction()`, forcing a backtrack (DeBroglie constraint documentation, boristhebrave.github.io). DeBroglie offers three path constraint variants: `ConnectedConstraint` (single connected component), `LoopConstraint` (forces at least two independent paths, preventing chokepoints), and `AcyclicConstraint` (forces tree/forest topology). Path constraints are **performance-heavy** — they require global connectivity analysis at each step and typically make backtracking mandatory. Additional non-local constraints available in DeBroglie include `CountConstraint` (tile count bounds), `MaxConsecutiveConstraint` (limits runs of identical tiles), and `MirrorConstraint` (forced symmetry).

### Multi-cell tiles and their implementation challenges

Multi-cell tiles — single logical tiles spanning multiple grid cells — reduce the grid-like appearance of WFC output and allow larger set pieces. Bad North uses two-cell tiles for curving beaches and chunky houses (Stålberg, EPC 2018; Boris the Brave, "WFC Tips and Tricks"). Implementation is challenging because a multi-cell tile occupies several cells simultaneously: all constituent cells must be collapsed together, adjacency rules must account for the tile's footprint in every direction, and the support count array must correctly track compatibility across the full boundary. Boris the Brave's Tessera asset for Unity handles multi-cell tiles by treating each cell of a big tile as a separate tile type with internal adjacency constraints linking them, effectively encoding the multi-cell structure within the standard WFC framework (Boris the Brave, "WFC Tips and Tricks"). The main risk is that multi-cell tiles dramatically increase the constraint space and the probability of contradiction, since collapsing one cell forces several others simultaneously.

### Practical integration challenges in games

The most persistent integration challenge is **global structure loss**. Boris the Brave states directly: "The key difficulty is WFC doesn't have any global structure to it — all it does is make the output generation look like the input locally" (Boris the Brave, "WFC Tips and Tricks"). Large WFC outputs look homogeneous because the algorithm has no concept of regions, biomes, or narrative progression.

The known approaches to this problem form a loose toolkit rather than a settled methodology. **Map subdivision** (Caves of Qud approach): divide the map into regions, run WFC with different tilesets and settings per region, overlap regions by one cell row for seamless transitions (Bucklew, GDC 2019). **Biome-level tile filtering** (Bad North approach): pre-select a biome per level and disable irrelevant tiles before running WFC (Stålberg, EPC 2018). **Probability field modulation**: vary tile weights spatially, increasing the probability of sand tiles near the coast and forest tiles inland (Marian42, "Infinite WFC," marian42.de). **Hierarchical Semantic WFC** (Alaka & Bidarra, Delft University, 2023): organize tiles into a hierarchy of abstract "meta-tiles" and "leaf tiles"; a first pass assigns meta-tiles (e.g., "castle"), a second pass collapses them into specific sub-tiles. **Pre-seeding with macro generation**: use noise functions, L-systems, or graph grammars to generate macro-scale structure, then run WFC within each region to fill detail — this is essentially the Caves of Qud multi-pass pipeline. Documentation on these approaches is thin; most exist as scattered blog posts, conference talks, and library documentation rather than systematic treatments. The WFC-MDP approach (arxiv.org, 2025) recasts WFC as a Markov Decision Process with an external optimizer handling global objectives, but this is still in the research stage.

### Known production uses

**Caves of Qud** (Freehold Games, first commercial WFC use) employs the **overlapping model** with small 16×16 pixel training images stored as .png files. WFC generates themed ruins and architectural structures — crypts, gardens, historic sites — that reflect procedurally generated cultures. The key innovation is a **multi-pass pipeline**: early passes establish coarse regional structure (partitioning the map), middle passes run WFC with specific templates per sub-area, and final passes ensure connectivity using A* pathfinding to identify disconnected rooms and place doors. Bucklew identified that adding detail to training images often causes overfitting, reducing output variability — the solution was subdividing the map and using different templates per region (Bucklew, GDC 2019).

**Bad North** (Plausible Concept) uses a **bespoke 3D simple tiled model** with over 300 hand-authored tiles (rock faces, stairwells, beaches, cliffs) built in Maya. Edge compatibility is established by scanning tile edges automatically upon import into Unity. The system generates islands by placing houses first (which all levels need), then collapsing outward — a house high on the island forces land around it, which forces connecting terrain down to the beach. A **navigability heuristic** ensures each tile knows its entry/exit points; houses are only placed in navigable areas, and unnavigable areas receive forests. Some tiles span two cells (multi-cell tiles) for smoother curves. The generator does **not backtrack** — on failure, it restarts entirely from a saved initial state. Stålberg noted that WFC is "pretty bad for long, thin things" like walls or pipes, and that "even to this day, in Bad North, occasionally you'll get a level that fails" (Stålberg, AI and Games interview, gamedeveloper.com, 2022).

**Townscaper** (Oskar Stålberg) combines three techniques: an **irregular quadrilateral grid** (hexagonal grid broken into quads, then quads again, with relaxed point positions), **WFC** for determining which building modules fit the current structure, and a **marching cubes** approach for placing corner segments that fit the irregular cells. Unlike Bad North, Townscaper runs WFC **in real time** as the player places and removes blocks — every placement triggers re-evaluation of the connected structure. The system uses approximately 500 tiles with **priority-weighted "recipes"** for decorations: gardens can only exist in fully enclosed flat areas (edge constraint ensures garden pieces only appear where they border a wall). Townscaper handles contradictions by **failing silently** — if two cells cannot find perfectly matching modules, the mismatch is simply ignored, most visibly with steel support structures. Stålberg's design philosophy: "I want the larger shapes to feel very predictable, but the smaller shapes are allowed to vary quite a bit" (Stålberg, AI and Games interview).

## 6. When autotiling suffices and when WFC adds value

Simple autotiling — selecting tile variants based on the immediate neighbor configuration — is sufficient when **the tileset is designed so that every neighbor combination has a valid tile and no global consistency is required**. Marching-squares terrain with one biome transition (grass/water), interior wall placement where every L/T/straight configuration exists in the tileset, and platformer tilesets where edges are fully covered all work with autotiling because every local decision is independent. The moment any of three conditions arises, autotiling breaks down and WFC becomes necessary: first, **when valid tiles are context-dependent beyond the immediate boundary** — a tile that is locally valid for its four neighbors may create an impossible situation two cells away, which only propagation can detect. Second, **when the tileset is deliberately incomplete** — Boris the Brave notes that WFC "plans around missing tiles," meaning it avoids placing tiles whose neighbors would require tiles that don't exist, whereas autotiling would place them and produce visible errors (Boris the Brave, "WFC Tips and Tricks"). Third, **when frequency control matters** — WFC's weighted selection ensures rare tiles appear sparingly and common tiles dominate, producing naturalistic distributions that autotiling's deterministic lookup cannot provide. The concrete decision criterion: if your tileset has any adjacency combinations that lack a valid tile, or if you need frequency-weighted variety with guaranteed local consistency across the full map, use WFC. If every possible neighbor configuration has exactly one correct tile, autotiling is simpler, faster, and sufficient.

## 7. Grammar systems versus WFC — what each excels at

Grammar systems and WFC solve fundamentally different generation problems and take fundamentally different inputs. **WFC takes a tileset (or example image) and produces a flat grid of locally consistent tiles** — it excels at filling space with texture-like content where the desired output is defined by local neighborhood patterns. Its input is spatial and visual; its output is spatially uniform (no inherent hierarchy or progression). **Grammars take a set of rewriting rules and produce hierarchically structured content** — they excel at generating content with intentional design structure: buildings with floors composed of bays composed of windows (CGA Shape), dungeons with locked doors requiring keys found in specific branches (Dormans' graph grammars), or trees with realistic branching topology (L-systems). Their input is symbolic and rule-based; their output reflects the rule hierarchy.

Choose WFC when the content is **texture-like**: tilemaps, terrain, city blocks where local coherence is the primary goal and global structure can be supplied externally. Choose grammars when the content has **intentional hierarchical structure**: levels that must be solvable, buildings that must follow architectural conventions, vegetation that must branch realistically. The most powerful approach combines both: use grammars to generate macro-scale structure (mission graph, building skeleton, road network) and WFC to fill the resulting regions with locally consistent tile detail. This is the pattern demonstrated by Caves of Qud's multi-pass pipeline, where WFC operates within structurally pre-defined regions rather than generating structure itself.

## 8. Source landscape

### Tutorials and implementation guides

**Boris the Brave, "Wave Function Collapse Explained" (boristhebrave.com, 2020)** — The clearest conceptual walkthrough of WFC as a constraint satisfaction problem; uniquely valuable for understanding propagation mechanics, the support count optimization, and the connection to AC-3/AC-4.

**Boris the Brave, "Arc Consistency Explained" (boristhebrave.com, 2021)** — Explains AC-3 and AC-4 algorithms with animated diagrams and direct connection to WFC's propagation; essential for understanding the constraint propagation foundations.

**gridbugs.org, "Wave Function Collapse" (gridbugs.org)** — Gold-standard implementation walkthrough in Rust with full code, data structures, entropy formula derivation, and a documented off-by-one bug in weighted random selection that serves as a cautionary example.

**Boris the Brave, "WFC Tips and Tricks" (boristhebrave.com, 2020)** — Practical tileset design advice, multi-cell tiles, path constraints, biome filtering; the most implementation-focused WFC reference for game developers.

**Prusinkiewicz & Lindenmayer, *The Algorithmic Beauty of Plants*, Chapter 1 (algorithmicbotany.org, free PDF)** — Definitive formal treatment of all L-system variants with mathematical precision; irreplaceable for implementing parametric and context-sensitive variants correctly.

**SideFX Houdini L-System SOP Documentation (sidefx.com)** — Complete production rule syntax supporting all four L-system variants simultaneously, with meta-test boundary constraint; the most comprehensive practical L-system reference available.

**GameDev.net, "Procedural Modeling of Buildings with Shape Grammars" (gamedev.net)** — Implementation-focused tutorial covering the shape tree data structure, grammar parser, and geometry export pipeline for CGA-style building generation.

### Academic papers

**Karth & Smith, "WaveFunctionCollapse is Constraint Solving in the Wild" (FDG 2017)** — First formal academic analysis of WFC; uniquely valuable for its ASP formulation and finding that propagation matters more than backtracking.

**Karth & Smith, "WaveFunctionCollapse: Content Generation via Constraint Solving and Machine Learning" (IEEE Transactions on Games, 2022)** — Extended analysis of backtracking, search heuristics, and pattern classification; essential for understanding WFC's failure modes at scale.

**Paul Merrell, "Example-Based Model Synthesis" (PhD dissertation, 2009; paulmerrell.org)** — WFC's direct precursor; uniquely valuable for the "modifying in blocks" strategy for large-scale generation that Gumin's WFC lacks.

**Müller et al., "Procedural Modeling of Buildings" (SIGGRAPH 2006)** — Foundational CGA Shape paper; essential for understanding split grammars and the full building-generation pipeline used in CityEngine.

**Dormans, "Adventures in Level Design" (PCGames@FDG, 2010)** — Introduces the mission/space separation and graph grammars for level generation; uniquely valuable for understanding how grammars encode game design structure.

**Stiny & Gips, "Shape Grammars and the Generative Specification of Painting and Sculpture" (IFIP Congress 1971)** — Original shape grammar formulation; necessary for understanding the geometric embedding concept that distinguishes true shape grammars from symbolic variants.

**van Rozen & Heijn, "Measuring Quality of Grammars for Procedural Level Generation" (PCG Workshop, 2018)** — Identifies grammar authoring failure modes and proposes the MAD metric; essential reading on grammar quality assessment.

### GDC talks and developer presentations

**Brian Bucklew, "Math for Game Developers: Tile-Based Map Generation using Wave Function Collapse in Caves of Qud" (GDC 2019)** — First-hand account of shipping WFC in a commercial game; uniquely valuable for the multi-pass pipeline, overfitting diagnosis, and map subdivision strategy.

**Oskar Stålberg, "Wave Function Collapse in Bad North" (EPC 2018) and "The Story of Townscaper" (Konsoll 2021)** — Practical 3D WFC implementation details, navigability heuristic, failure rates, and the evolution from Bad North to Townscaper; essential for understanding real-world WFC limitations.

### Interactive tools and repositories

**Maxim Gumin, WaveFunctionCollapse (github.com/mxgmn/WaveFunctionCollapse)** — The original implementation and README; uniquely valuable as the canonical algorithm description and the definitive list of ports, forks, and production uses.

**Boris the Brave, DeBroglie library (boristhebrave.github.io/DeBroglie)** — C# WFC library with path constraints, backtracking, hex grids, and comprehensive documentation; the most feature-complete open-source WFC implementation.

**Oskar Stålberg, interactive tiled model demo (oskarstalberg.com/game/wave/wave.html)** — Browser-based interactive WFC demo where users click to collapse cells and watch propagation animate; the fastest way to build intuition for WFC behavior.

**nylki/lindenmayer (github.com/nylki/lindenmayer)** — Feature-complete JavaScript L-system library supporting branching, context-sensitive, and parametric rules with functional production definitions.

**Sean Barrett, "L-Systems Considered Harmful" (nothings.org)** — Critical analysis arguing that complex L-system formalisms can be reduced to simpler procedural code; essential counterpoint for evaluating when L-system formalism genuinely adds value versus when it adds obscurity.
