# Pass 5b: object placement, scattering, and road generation

**Every procedural world reaches the same crossroads: terrain exists, biomes are assigned, but the landscape is bare.** Pass 5b transforms abstract spatial data—density fields, heightmaps, biome maps, and region graphs—into the concrete objects and connective infrastructure a player actually sees. This document covers the ten core techniques that populate a procedural world with vegetation, props, landmarks, settlements, and roads, presented at implementation depth. It assumes familiarity with the Pass 5a concepts (Voronoi partitioning, region graphs, biome assignment, climate simulation, and biome transition blending) and with the Poisson disk sampling algorithm itself; the focus here is exclusively on how that algorithm and these upstream data structures are applied in placement and routing pipelines.

The techniques divide into two domains. The first six address **object placement and scattering**: converting a density field into actual placed instances, filtering those instances by terrain properties, preventing placement in forbidden zones, ordering placement by object priority, creating organic-looking clusters, and positioning landmarks and settlements. The last four address **road and path generation**: routing roads across terrain cost maps, ensuring network connectivity via spanning trees, smoothing jagged paths into natural curves, and assembling these pieces into complete road network strategies. Each section covers the problem and motivation, the algorithm at implementation depth, known failure modes with fixes, and pipeline connections to upstream and downstream stages.

## 1. Density fields turn biome data into placement probability

A density field is a scalar field over the terrain surface, with values in [0, 1], that controls where and how thickly objects appear. Without density modulation, placement is either uniform everywhere (unrealistic) or requires hand-painting per biome zone (unscalable). The density field solves this by compositing multiple upstream inputs into a single value at each candidate point, which then drives placement probability or spacing.

The standard construction composites four multiplicative layers: `final_density(x,y) = biome_density(x,y) × noise(x,y) × slope_mask(x,y) × exclusion_mask(x,y)`, clamped to [0, 1]. The biome layer assigns a per-biome, per-object-type density weight—forest biome might set tree density to 0.8 and cactus density to 0.0, while desert inverts that relationship. The noise layer, typically one to three octaves of Simplex noise, breaks spatial regularity so that a biome doesn't fill uniformly. Red Blob Games' terrain-from-noise article notes that "for object placement we want to use higher amplitudes with higher frequencies ('blue noise')" and recommends Poisson disk or jittered grids over raw noise for the final point positions. The remaining two layers—slope and exclusion—are covered in the following sections but enter the density field as multiplicative masks.

Two methods consume this density field to produce placed points. **Radius modulation** adjusts the minimum-distance parameter of Poisson disk sampling per point: `min_dist = min_radius + density_value × (max_radius - min_radius)`, where a density of 0 yields the tightest packing and a density of 1 the sparsest (DevMag, Herman Tulleken, 2009). This requires that the background grid cell size be computed from the *maximum* possible radius, not the local radius, and that the neighbourhood check iterate through all cells in the point's vicinity rather than just the immediate cell (DevMag, 2009). The kchapelier/poisson-disk-sampling JavaScript library implements this directly via a `distanceFunction` callback that returns a [0, 1] value at each candidate point. **Rejection probability** is simpler: generate candidates at maximum density using fixed-radius Poisson disk sampling, then reject each candidate with probability `1 - density(x,y)`. The ezEngine procedural generation system uses this pattern, exposing a "Density" parameter in [0, 1] that "determines how likely it is that an object gets spawned" (ezEngine documentation). Rejection is faster to implement and integrates trivially with any existing Poisson disk sampler, but unlike radius modulation it does not guarantee minimum spacing—two surviving points near a high-density / low-density boundary may end up closer together than either point's local density would suggest. This is the **density-boundary clumping** artifact; it is mitigated by running a post-placement minimum-distance cull or by using radius modulation instead.

The density field consumes the biome map and terrain heightmap from upstream (Pass 5a), plus any authored or procedural noise layers. Its output is the set of candidate placement points, each tagged with an object type and the density value at its location. Downstream stages—slope filtering, exclusion, hierarchical scattering—consume these candidates and progressively thin them.

## 2. Slope and altitude filtering prevents objects from floating on cliffs

Placing trees on a vertical cliff face or cacti below the waterline produces immediate visual artifacts. Slope-aware and altitude-aware filtering eliminates candidates whose terrain properties fall outside the acceptable range for their object type. This is the single most visible quality gate in a placement pipeline: without it, the density field can produce geometrically valid but physically absurd placements.

Slope is computed from the terrain heightmap using **central finite differences**. Given a heightmap `h(x, y)` with grid spacing `s`, the surface normal at each cell is:

```
u = (2s, 0, h(x+1, y) - h(x-1, y))
v = (0, 2s, h(x, y+1) - h(x, y-1))
normal = normalize(cross(v, u))
```

The factor `2s` converts from grid-cell units to world units; omitting it is the most common implementation error and produces normals that are either too flat or too steep depending on the height scale (Rastertek Tutorial 14; GameDev.net terrain normal discussions). Slope is then derived as `steepness = 1.0 - dot(normal, up_vector)`, yielding 0.0 for perfectly flat terrain and approaching 1.0 for vertical surfaces (Rastertek Tutorial 14). Alternatively, `acos(dot(normal, up))` gives the angle in radians. The Sobel operator applied to a 3×3 heightmap neighbourhood produces smoother gradients than simple central differences and is preferred for GPU terrain pipelines (math-araujo, procedural terrain generation). Altitude filtering is simpler: each object type defines a `min_height` and `max_height`, and candidates outside this band are rejected. Common bands include shoreline (0–5 m), lowland deciduous forest (5–200 m), montane coniferous forest (200–800 m), alpine meadow (800–1200 m), and snow/bare rock (1200 m+) (unconfirmed—values are illustrative and project-dependent).

**Hard cutoffs** at slope or altitude thresholds are the primary failure mode: a tree line that stops in a perfectly straight horizontal band looks unmistakably procedural. The fix is to replace boolean thresholds with **smooth ramp functions**: `slope_weight = smoothstep(threshold - feather, threshold + feather, steepness)`, producing a gradual density falloff rather than a hard edge. Houdini's HeightField Mask by Feature node exposes this as a "Slope Ramp" control and supports five simultaneous criteria—slope, height, curvature, direction, and occlusion—intersected via multiplication (SideFX documentation). In a lighter-weight implementation, multiplying a feathered slope mask and a feathered altitude mask into the density field achieves the same effect. A second failure mode is **edge-pixel errors**: border cells of the heightmap lack neighbours on one or more sides for finite differences, producing incorrect normals. Clamping or mirroring at boundaries is required.

This stage consumes the terrain heightmap and the candidate placement points from the density field. It produces a filtered candidate set with slope and altitude masks that can also be reused by the exclusion mask stage and by road routing cost maps.

## 3. Exclusion masks compose boolean constraints into a single gate

An exclusion mask is a scalar field—binary (0 or 1) or soft (0.0 to 1.0)—that prevents placement in forbidden areas. Roads, water bodies, building footprints, cliff faces, and player-defined zones each generate a separate mask layer. These layers are composed by **multiplication**, which functions as boolean AND: if *any* single mask is 0 at a point, the final mask is 0 and placement is blocked. The dphrygian.com devlog for the game *Second Order* describes this exact pattern: "floating point values are quantized at a threshold to produce boolean masks," then combined with AND operators for building placement zones and OR operators for street identification (dphrygian.com procedural generation blog).

Each mask source has its own construction logic. **Water masks** threshold the heightmap: `mask = (height > water_level) ? 1 : 0`. **Road masks** rasterize spline centerlines into a bitmap with a configurable width buffer—typically road width plus 2–5 m on each side. In Unreal Engine 5's PCG framework, road splines write to Runtime Virtual Textures that affect spawn distribution (UE5 PCG documentation). **Building footprint masks** are generated by rasterizing placed building bounding boxes; UE5's Difference node takes two point sets (buildings and vegetation) and removes vegetation overlapping building bounds (Deaco, "PCG in a Nutshell," Medium). **Cliff masks** reuse the slope computation from the previous stage: cells with slope above a cliff threshold (typically 55–70°) are masked to 0. **Previously placed object masks** are generated dynamically during hierarchical scattering (see next section): each placed object's footprint radius is rasterized as an exclusion circle.

Buffer padding around exclusion sources prevents clipping: objects placed exactly at the boundary of a road or building will visually intersect the excluded geometry. Implementation is a **morphological dilation** of the binary mask by the buffer radius, or equivalently, inflating each exclusion polygon before rasterization. **Edge feathering** prevents the hard-cutoff artifact where vegetation stops in a perfect line at the road edge. The standard approach is `feathered_mask = smoothstep(inner_radius, outer_radius, distance_to_edge)`, where the transition zone width equals `outer_radius - inner_radius` (unconfirmed as a universal formula—the principle is consistent across Houdini, UE5, and ezEngine, but parameterization varies). Gaussian blur of the binary mask is an alternative that produces softer transitions at the cost of blurring small features. A critical failure mode is **mask resolution mismatch**: if masks are stored at lower resolution than the placement grid, narrow features like footpaths or small buildings can fall between mask pixels and fail to exclude. Mask resolution should match or exceed the minimum placement spacing.

Exclusion masks consume the water level, road spline data, building footprints from earlier placement passes, and slope data from the heightmap. They produce the final binary or soft gate multiplied into the density field before candidate point generation.

## 4. Hierarchical scattering places the largest objects first

In any natural environment, large organisms establish first and smaller ones fill the remaining space. Hierarchical scattering replicates this ecological ordering by executing multiple placement passes in decreasing order of object size, with each pass's placed objects generating exclusion zones for subsequent passes. Without this ordering, small ground-cover objects can consume positions that should be occupied by large trees, producing an unnaturally uniform size distribution—the **size-starvation** artifact.

The algorithm is a multi-pass loop:

1. **Pass 1 (large objects—canopy trees, boulders):** Generate candidate points using Poisson disk sampling with a large minimum radius matching the tree canopy diameter. Filter by density field, slope mask, altitude mask, and exclusion masks. Place survivors. Record each placed object's position and footprint radius into a spatial data structure.
2. **Pass 2 (medium objects—bushes, medium rocks):** Generate candidates with a smaller minimum radius. Filter by all original masks *plus* the exclusion zones from Pass 1. Place survivors, update the spatial data structure.
3. **Pass 3 (small objects—flowers, debris):** Finer spacing, same filtering pattern, now including exclusions from Passes 1 and 2.
4. **Pass 4 (ground cover—grass, moss):** Dense grid or high-density Poisson disk sampling. Ground cover may partially overlap medium and large objects for realism (grass around tree trunks), so exclusion radii for this pass are typically reduced.

Footprint tracking is the implementation crux. Three approaches are common: a **spatial hash grid** (insert positions + radii for O(1) proximity lookups, the same data structure used by Poisson disk sampling's background grid), **mask rasterization** (rasterize footprint circles into a bitmap after each pass, which subsequent passes sample as an additional exclusion mask), and **KD-tree or R-tree** insertion (handles non-uniform radii efficiently but adds implementation complexity). The UE5 PCG framework uses a physics-based approach: Bounds Modifier nodes define collision volumes per mesh, and the Self Pruning node prevents new meshes from spawning inside existing bounds (Deaco, Medium). The ezEngine system assigns each placement type a Footprint radius in meters and filters via surface-type raycasts—"prevents procedural objects from spawning on top of other procedural objects" (ezEngine documentation).

Mick West's foundational article "Random Scattering: Creating Realistic Landscapes" (Game Developer magazine) describes the underlying principle as a per-object "kill zone radius proportional to its size" and notes that biologically, trees do not have hard kill zones but rather "inhibit fields—an exponentially decaying function of distance where successful growth becomes less likely near existing trees" (West, Game Developer). This suggests that sharp boolean exclusion zones, while simpler, are less naturalistic than soft exclusion with exponential falloff—a refinement worth implementing for vegetation but typically unnecessary for artificial objects like props and structures. A second failure mode is **unnatural uniformity**: if exclusion zones are perfectly circular and rigidly enforced, the boundary between populated and empty zones is too clean. Adding noise to exclusion radii (e.g., `effective_radius = base_radius × (0.8 + 0.4 × noise(x,y))`) breaks the regularity. A third is **order-dependent determinism**: different placement orders produce different results even from the same seed, so the pass order and intra-pass sorting must be fixed and documented for reproducible generation.

Hierarchical scattering consumes the density field, all mask layers, and the terrain heightmap (for slope/altitude filtering within each pass). It produces the final set of placed object instances with positions, types, and scales, plus the accumulated exclusion data structure that downstream passes (e.g., road generation) can query to avoid routing roads through placed structures.

## 5. Clustering via deformation kernels and noise thresholds

Uniform-density Poisson disk sampling produces a pleasant blue-noise distribution, but natural environments are *clustered*: trees grow in groves, rocks accumulate in talus fields, flowers carpet meadows in patches. Generating this clustering requires biasing the placement distribution away from uniformity toward organic groupings—without collapsing into pure randomness and its associated clumping artifacts.

The most rigorous approach is the **deformation kernel method** introduced by Lane and Prusinkiewicz (Graphics Interface 2002). The algorithm maintains a 2D probability density field `f(x,y)` discretized over an `n×n` grid, where `f(x,y) dx dy` is the probability of placing the next object in the infinitesimal area at `(x,y)`. Each time an object of radius `r_t` is placed at `(x_t, y_t)`, the field is multiplied by a deformation kernel: `f_new(x,y) = f(x,y) × κ(√((x - x_t)² + (y - y_t)²) / r_t)`, then renormalized so it integrates to 1 (Lane and Prusinkiewicz, 2002). The kernel function `κ(d)` controls the spatial effect: **inhibitory kernels** (κ < 1 near the placed object) produce dispersed distributions like self-thinning forests; **promotional kernels** (κ > 1 near the placed object) produce clustering; and **mixed kernels** (inhibitory at short range, promotional at medium range) produce the most naturalistic effect of tight clusters separated by gaps (Lane and Prusinkiewicz, 2002). Placement proceeds by inverse-transform sampling from the deformed distribution. The Hopkins index—a standard measure of spatial randomness—is controllable from **0.4** (overdispersed) through **1.0** (random) to **2.4+** (strongly clustered) by varying kernel parameters (Lane and Prusinkiewicz, 2002). Complexity is O(mn) with incremental row-sum updates, where m is the number of placed objects. The failure mode is that kernel interaction between multiple species requires a full kernel matrix M, where `M[i][j]` encodes the effect species *i* has on the placement probability of species *j*—an `O(s²)` design surface for s species that becomes unwieldy without tooling.

The commercially dominant alternative is **noise-threshold clustering**: generate a Perlin or Simplex noise texture over the terrain, threshold it to create per-species density maps (high noise = dense placement zone), then run Poisson disk sampling within the high-density zones. This naturally produces clusters wherever noise peaks coincide with suitable biome conditions. The approach is used in UE5's PCG framework ("Use Perlin noise to vary density. This creates natural clustering instead of uniform placement"—UE5 PCG tutorials) and is the foundation of the Horizon: Zero Dawn GPU placement pipeline (GDC 2017, van Muijden). The AutoBiomes system (Fischer et al., The Visual Computer 2020) extends Poisson-disk dart-throwing with a per-asset **clustering probability** parameter that biases candidates toward existing instances of the same type, combined with shadow tolerance and repelling distance (Fischer et al., 2020). This is the closest documented system to an explicit "attraction field" per object type, though the implementation details of how clustering probability modifies dart-throwing acceptance criteria are not fully specified in the paper.

Clustering consumes the density field (which already encodes biome and noise) and the spatial data structure from hierarchical scattering. It modifies the distribution of placed objects within each hierarchical pass rather than operating as a separate pipeline stage. The placed-object set is its output, indistinguishable in data format from non-clustered placement but with a measurably different spatial statistics profile.

## 6. Landmark and settlement placement relies on multi-criteria terrain scoring

Landmarks (temples, towers, viewpoints) and settlements (towns, cities) serve as navigation anchors and gameplay hubs. Their placement is more constrained than vegetation scattering: they must be on buildable terrain, visible from approach routes, connected to road networks, and separated by playable distances. Yet **landmark placement is the least documented technique in practical PCG**—nearly all published sources discuss what makes a good landmark from a level-design perspective (visibility, silhouette distinctiveness, "from landmark to landmark" navigation chains per level-design.org) without providing a scoring function or placement algorithm. In shipped games, landmarks are almost universally hand-placed by designers. This section documents what is available and flags gaps honestly.

Settlement placement is better documented. The most concrete open description is Miguel Cepero's Procedural World blog (2011), which describes a five-step pipeline: (1) sample terrain for flat areas where the derivative of height approximates zero—both valleys and mesas; (2) enforce minimum separation between candidates; (3) expand each settlement outward in an anisotropic Voronoi partition where expansion rate is modified by terrain slope; (4) aggregate neighbouring settlements into political entities using shared-border-length and terrain-steepness heuristics; (5) generate roads between settlements via A* with a cost function considering slope, existing roads (reuse discount), and border-crossing penalties (Cepero, Procedural World, 2011). Red Blob Games' 2025 mapgen2-towns experiment describes a simpler version: for each candidate town location, run Dijkstra's algorithm within a limited range to evaluate reachable resources, require "≥1 fresh water, ≥2 arable tiles," use "variable movement costs (depend on biome, discourage crossing rivers, encourage following roads)," rank candidates, and place the top-ranked town, iterating with roads between placed towns influencing subsequent candidate scores (Patel, redblobgames.com/x/2503-mapgen2-towns, 2025). Patel notes candidly that "this kind of algorithm requires a *lot* of tweaking to balance the different factors for town and road placement" (Patel, 2025). The ScienceDirect paper on semantically plausible small-scale towns (2023) provides the most rigorous scoring system: six weighted principles (security, privacy, sustainability, social life, economy, beauty) with terrain-derived scoring maps including a **walkability map** (zero score at slope ≥ 35°, full score below 3°), an **accessibility map** (Floyd-Warshall path costs between sample points), a **water proximity map** (walking-cost distance to nearest water source, not Euclidean), and a **view quality map** (scenic value from nearby trees, rivers, landscape features) (ScienceDirect, S1524070323000012, 2023). Emilien et al. (The Visual Computer, 2012) introduce **interest maps** that combine sun exposure, wind shelter, water proximity, safety, and convenience into a single terrain-scoring field; settlement seeds are iteratively placed at interest-map maxima, and new roads feed back into the interest map for subsequent placements (Emilien et al., 2012).

For landmark placement specifically, a reasonable scoring function can be reconstructed from general PCG principles but **no published algorithm or implementation was found that provides it explicitly**: `Score(candidate) = w1 × elevation_prominence + w2 × visibility_score + w3 × min_separation_penalty + w4 × road_centrality + w5 × terrain_flatness` (unconfirmed—reconstructed from aggregated level-design heuristics and PCG principles, not from any single source). Elevation prominence measures height above surrounding terrain within a radius; visibility score estimates the viewshed area visible from the candidate (a computationally expensive raycast operation); road centrality uses betweenness centrality on the road network graph to prefer locations that many shortest paths pass through; and minimum separation enforces a hard constraint that no two landmarks are closer than a threshold distance. The GDMC (Generative Design in Minecraft Competition) has produced diverse algorithmic approaches to settlement placement—agent-based systems where autonomous agents build structures determined by nearby resources, fitness-function approaches combining modified A* for roads with plot analysis, and founding-seed-expansion methods using DFS over height maps (gendesignmc.wikidot.com, 2022)—but these are competition entries optimized for Minecraft's voxel grid and do not generalize cleanly to continuous-terrain PCG.

Settlement placement consumes the biome map, heightmap, water body locations, and optionally the region graph from Pass 5a. It produces a set of settlement positions with population estimates and a candidate list for road network generation. Landmark placement, if automated, consumes the road network and terrain data. Both feed into road generation as the node set that roads must connect. **The gap in documentation is real and practical**: developers implementing settlement placement will find conceptual frameworks but will need to design their own scoring functions and tune weights experimentally. Landmark placement is even less supported—expect to build from scratch.

## 7. A* and Dijkstra on terrain cost maps route roads through landscape

Given two settlements to connect with a road, the problem reduces to finding a minimum-cost path across the terrain. A* and Dijkstra's algorithm solve this on a weighted graph derived from the terrain, where edge weights encode construction difficulty. Dijkstra finds optimal paths from a source to all reachable nodes; A* adds a heuristic to focus the search toward a single target. Both are well-documented, so this section focuses on the **cost map construction** that makes them useful for road routing rather than on the algorithms themselves.

The terrain is discretized into a grid where each cell is a graph node. Red Blob Games' implementation article provides the canonical weighted-graph interface: a `cost(from_node, to_node)` method that returns the traversal cost of moving between adjacent cells (Patel, redblobgames.com/pathfinding/a-star/implementation.html). For road routing, this cost function is a weighted sum of terrain properties. **Slope cost** is the dominant term: `c_slope = μ(slope_in_direction_of_travel)`, where the transfer function μ returns infinity if slope exceeds the maximum authorized grade (preventing impossible roads) and increases monotonically otherwise (Galin et al., Computer Graphics Forum 29(2), 2010). Red Blob Games' graphs article explicitly models directional slope: "we might mark downhill edge B→C with weight 2 and mark uphill edge C→B with weight 5 to make it easier to walk downhill" (Patel, redblobgames.com/pathfinding/grids/graphs.html). **Biome cost** adds a per-biome multiplier: forest ≈ 2–3× (clearing required), plains ≈ 1× (easy), marsh ≈ 5–10× (difficult construction). **River crossing cost** imposes a high but finite penalty (requiring a bridge), while lake or ocean crossings return infinity (Galin et al., 2010). **Existing-road discount** reduces cost to 0.1–0.5× at cells where a road already exists, encouraging road merging and the formation of T-junctions (Vazgriz, procedurally-generated-dungeons blog; Patel, mapgen2-towns, 2025).

A critical implementation detail from Galin et al. (2010) is the **connectivity mask**: on a standard grid with 4- or 8-connectivity, paths are constrained to 45° angle increments, producing visibly grid-aligned roads—the **staircase artifact**. Galin et al. solve this by connecting each grid point to all neighbours within radius k, where only segments with `gcd(i, j) = 1` are retained to avoid redundancy. At **k = 5** (the recommended value), this produces 80 candidate segments per node with **11.3° angular resolution**, which yields visually smooth paths in under 1 second on a 100×100 grid (Galin et al., 2010). The cost over each segment is approximated as a line integral: `c(p_k, p_{k+1}) ≈ Σ c(p(t), ṗ(t), p̈(t)) Δt`, sampling position, direction, and curvature at discrete intervals along the segment (Galin et al., 2010). When curvature cost is included—penalizing sharp turns to ensure realistic road geometry—the search space expands from 2D to 3D: Galin et al. discretize the angular domain [0, 2π] into m intervals and search over an `n² × m` grid of oriented points, increasing computation time roughly proportionally to m (Galin et al., 2010).

Red Blob Games provides complete Python implementations of both Dijkstra's algorithm and A* with priority queues, path reconstruction, and the weighted-graph interface (Patel, implementation.html). The heuristic for A* in road routing is the cost of a hypothetical straight-line road from the current node to the goal—"admissible and monotonic" per Galin et al. (2010). Failure modes include **local minima** when the heuristic poorly estimates true cost (rare with admissible heuristics but possible on extreme terrain), **grid-resolution sensitivity** (too coarse misses mountain passes; too fine explodes computation), and the **parameter-tuning problem**: relative weights between slope, biome, water, and curvature costs dramatically change road behaviour, and "few papers provide systematic tuning guidance" (observed across all reviewed sources). Cost map construction consumes the heightmap, biome map, water body data, and any existing road network. It produces a weighted graph that A* or Dijkstra searches to output an ordered sequence of grid cells representing the road path.

## 8. Minimum spanning trees guarantee network connectivity with minimum cost

Connecting N settlements with roads requires at minimum N-1 road segments—a tree. The minimum spanning tree (MST) finds the N-1 connections that minimize total edge weight (typically terrain-aware cost or Euclidean distance), guaranteeing that every settlement is reachable while minimizing total construction. Real road networks contain loops, so the MST serves as a **skeleton** that is then augmented with additional edges.

The standard pipeline, well-documented in dungeon generation and directly transferable to overworld road networks, is: (1) compute the **Delaunay triangulation** of settlement positions, producing a planar graph with no crossing edges—this serves as the candidate edge set and avoids the O(N²) cost of a complete graph (Vazgriz, vazgriz.com/119/procedurally-generated-dungeons/); (2) compute the **MST** of this Delaunay graph, which is always a subgraph of the Delaunay triangulation (proven geometric property); (3) **re-add a random subset of non-MST Delaunay edges**—typically each non-MST edge is included with 10–15% probability—to create loops and alternate routes (Vazgriz, 2019). Kruskal's algorithm (O(E log E): sort all edges, add cheapest non-cycle-forming edge) and Prim's algorithm (O(E log V): grow from a starting node, always adding the cheapest frontier edge) both produce identical MSTs. In practice, **Prim's is more natural when growth should radiate from a capital city**, while Kruskal's is simpler when all edges are pre-computed (unconfirmed as a universal preference—both are used interchangeably in PCG implementations).

Edge weights should reflect **terrain difficulty**, not just Euclidean distance. The ideal approach computes the A*-based terrain-cost path between each pair of Delaunay-connected settlements (using the cost map from section 7) and uses the resulting path cost as the edge weight for MST computation. This ensures the MST avoids connections that would require crossing impassable mountains even if the settlements are geographically close. Herzog (UCL) describes this for archaeological least-cost network reconstruction: MST edges use least-cost-path distances through terrain rather than Euclidean distance (Herzog, CAA 2012). The re-addition heuristic can be refined beyond uniform random probability: prefer adding shorter non-MST edges (which create small local loops rather than long redundant highways), or add edges that maximize the reduction in average shortest-path length between all settlement pairs (betweenness-centrality-based selection). Genre affects the target: strategy games benefit from more redundancy (alternate routes for armies), while adventure games may prefer tree-like chokepoints for narrative pacing (unconfirmed—common design wisdom but not formally studied).

The MST stage consumes the settlement positions from the landmark/settlement placement stage and the cost map (or Euclidean distances as a simpler alternative). It produces a graph of road connections—a subset of the Delaunay triangulation—that the pathfinding stage (section 7) then routes individually across the terrain. The primary failure mode of a pure MST is that it contains **no loops**: a single blocked road disconnects part of the network. The re-addition step is not optional for any network intended to feel realistic.

## 9. Spline fitting smooths grid-aligned paths into natural curves

A* on a grid produces a sequence of grid-aligned waypoints—a piecewise-linear path that looks jagged and artificial as a rendered road. Spline fitting converts this into a smooth curve. This is a well-documented technique, so the focus here is on which spline types are used for roads specifically and how control points are derived from pathfinding output.

**Catmull-Rom splines** are the most common choice for procedural roads because they are **interpolating**: the curve passes exactly through every control point, which means waypoints from pathfinding become points on the final road with no approximation gap (Alan-Baylis, Unity-Road-Generator, GitHub). Tangents are auto-computed from neighbouring points: at point P_i, the tangent is `ẋ_i = (P_{i+1} - P_{i-1}) / 2` in the uniform case. The **centripetal parameterization** (α = 0.5), formalized by Yuksel, Schaefer, and Keyser (2010), eliminates cusps and self-intersections even when control points have highly irregular spacing—a common situation when A* waypoints cluster near terrain obstacles. Red Blob Games' mapgen2 uses splines for road rendering: "Roads don't get the noisy edge treatment. Instead, they are drawn with splines from edge midpoint to midpoint. For polygons that have more than two road neighbors, I instead draw an intersection, with splines from all the edge midpoints to the polygon center" (Patel, polygon map generation article, 2010). Galin et al. (2010) use **piecewise clothoid curves** (Euler spirals) via the method of Walton and Meek (2005), which have linearly varying curvature matching real highway design standards—more realistic than Catmull-Rom but more complex to implement (Galin et al., 2010).

Before spline fitting, the raw A* path should be **simplified** using the Ramer-Douglas-Peucker algorithm or a similar polyline simplification method to remove collinear or near-collinear points. Too many control points produce unnecessary undulations; too few miss important turns. For Catmull-Rom splines, the first and last control points must be handled specially—typically by duplicating the endpoints (P₀, P₀, P₁, P₂, ... P_{n-1}, P_n, P_n) or by extrapolating phantom points. The road mesh is generated by sampling the spline at regular parameter intervals, then extruding a ribbon perpendicular to the tangent at each sample: left and right edges offset from the centerline by ±width/2. UV coordinates map linearly along the road length for texture tiling. **Self-intersection** with uniform Catmull-Rom parameterization is the primary failure mode—it occurs when control points are very close together, and the centripetal variant is the standard fix (Yuksel et al., 2010).

**Terrain modification** under the road is a required post-processing step. Galin et al. (2010) describe the process: for every terrain mesh vertex v within distance R of the road trajectory Γ, if v is within the asphalt width r, set v's height to the road height at the nearest curve point; if v is between r and R (the blending region), interpolate between road height and terrain height using a profile curve. This creates natural excavations and embankments. Adaptive mesh refinement in the road neighbourhood ensures the terrain mesh is detailed enough to deform smoothly (Galin et al., 2010). Without this step, roads either float above valleys or clip through hillsides—the **terrain gap** artifact. Vegetation exclusion masks (section 3) must also be updated to clear trees from the road corridor: Galin et al. remove all tree instances within distance R of the trajectory (Galin et al., 2010).

Spline fitting consumes the ordered waypoint sequence from A* pathfinding and the terrain heightmap. It produces a smooth curve representation of the road (control points plus spline type), a road mesh for rendering, and terrain modification data for the heightmap. The road corridor also feeds back into the exclusion mask system for subsequent placement passes.

## 10. Road network strategies from spanning trees to tensor fields

The preceding sections describe individual road-generation components—cost maps, pathfinding, MST connectivity, spline smoothing. This section covers the **high-level strategies** that compose these components into complete road networks. Documentation quality varies sharply: L-system and tensor-field approaches are well-published in academic literature, while hub-and-spoke and organic-branching strategies are described informally in dev blogs or not at all.

**Spanning-tree networks** (section 8's pipeline: Delaunay → MST → selective re-addition → per-edge A* routing → spline smoothing) are the simplest complete strategy and dominate roguelike and strategy-game road generation. They produce organic-looking regional networks where every settlement is connected. The weakness is uniformity: all roads are the same priority level, with no distinction between highways and local paths. **Hierarchical road networks** address this by generating multiple tiers—Galin, Peytavie, Guérin, and Beneš (Computer Graphics Forum, 2011) describe highways connecting major cities (generated first, with the most generous cost thresholds), secondary roads connecting towns to the nearest highway node, and local roads connecting villages (Galin et al., 2011). Each tier uses different cost functions and road standards (width, maximum slope, minimum curve radius). The tiers interact: lower-tier roads benefit from an existing-road discount when they coincide with higher-tier routes.

**L-system networks** are the foundational approach for city-scale street generation. Parish and Müller (SIGGRAPH, 2001) introduced extended L-systems with **global goals** (highways shoot rays toward population-density peaks) and **local constraints** (intersection detection, road snapping, water avoidance) to produce realistic urban road networks from image-map inputs (population density, water boundaries, elevation) (Parish and Müller, 2001). This led to the commercial CityEngine (now ESRI CityEngine). However, L-systems suffer from "parameter bloat, limited user control, difficulty handling complex exceptions" (observed across subsequent literature); Chen et al. (2008) moved to **tensor-field tracing** as a more controllable alternative. In this approach, tensor fields encoding desired road orientations—grid, radial, heightmap-following, or polyline-following—are blended with distance-based falloff, and road segments are traced as streamlines through the field's eigenvectors using RK4 integration. Martin Devans' open-source implementation (martindevans.me, 2015) confirms that "tracing vectors is extremely sensitive to errors because the errors accumulate along the entire length of the streamline" and reports 5–10 seconds per city (Devans, 2015). Tensor-field methods produce the most visually natural city-scale street grids but are overkill for regional overworld roads.

**Hub-and-spoke patterns**, where a capital directly connects to all satellite settlements, are intuitive but **poorly documented in PCG literature**—no academic paper or detailed implementation was found describing this strategy formally. It appears in game dev forum discussions and is straightforward to implement (connect hub to all nodes, optionally run MST on the remaining unconnected pairs), but formal analysis of its properties is a gap. **Organic branching** and **agent-based growth**—where roads extend from seeds, branch based on local population density and terrain, and merge when they encounter existing roads—have been explored by Song and Whitehead (TownSim, 2019) and in GDMC competition entries, but implementation-level documentation remains sparse. **GAN and diffusion-based approaches** (SIGGRAPH 2020 posters; DRoLaS, 2025) are emerging but lack the procedural control of algorithmic methods and are not yet practical for most game pipelines (unconfirmed as to current state—the field is moving rapidly).

Road network strategy selection consumes the settlement positions, terrain cost map, and genre requirements. It produces the complete road graph—the set of all road segments connecting settlements—which is then individually routed (section 7), smoothed (section 9), and rendered. **The honest documentation assessment: the MST-to-A* pipeline is well-documented enough to implement from published sources. L-systems and tensor fields have strong academic papers. Hierarchical road networks have Galin et al. (2011) but thin implementation detail beyond that paper. Hub-and-spoke, organic branching, and agent-based growth are at best blog-post level. Developers working on overworld road networks will find the spanning-tree approach most implementable; city-scale street networks have more options but steeper complexity.**

## A typical vegetation and prop placement pipeline

The following numbered stages describe data flow from biome map and terrain data through to final placed objects:

1. **Density field construction.** For each object type, composite a per-biome density weight with one or more noise layers (typically 1–3 octaves of Simplex noise) and any authored density paint masks. Output: a per-object-type scalar field over the terrain, values in [0, 1].
2. **Slope and altitude mask generation.** Compute surface normals from the heightmap via central finite differences. Derive slope and apply per-object-type threshold ramps. Generate altitude band masks with smoothstep feathering. Multiply into the density field.
3. **Exclusion mask composition.** Gather water, road, building, cliff, and any authored exclusion layers. Dilate each by its buffer radius. Optionally feather edges. Multiply all layers together; multiply the result into the density field.
4. **Hierarchical Poisson disk sampling.** For each size tier (large → small), generate candidate points using Poisson disk sampling with the density field modulating the minimum radius. Accept or reject each candidate based on the final density value. Record placed objects' footprints as new exclusion zones for the next tier.
5. **Clustering adjustment.** Within each tier, apply noise-threshold clustering or deformation-kernel biasing to shift the spatial distribution from uniform toward organic groupings.
6. **Surface alignment and variation.** Project each placed point onto the terrain surface, align to the surface normal (optional, typically partial alignment), apply random scale jitter (±10–20%) and rotation, and select mesh/texture variants.

Five common mistakes occur when density, slope, and exclusion interact incorrectly. **First**, multiplying slope masks before density noise causes the noise to "bleed" density into steep areas where the slope mask should block entirely—apply slope as a hard gate after noise, or use `min(slope_mask, density)` rather than multiplication. **Second**, applying exclusion masks at a coarser resolution than placement spacing allows objects to spawn on narrow roads or small buildings that fall between mask pixels. **Third**, failing to update exclusion masks between hierarchical passes causes small objects to overlap large ones because the large-object footprints were never rasterized into the mask. **Fourth**, using hard binary slope thresholds without feathering produces visible contour lines in vegetation density at specific elevations—always use smooth ramp functions. **Fifth**, compositing density noise with a frequency similar to the biome transition width creates moiré-like interference patterns at biome boundaries; the noise frequency should be significantly higher or lower than the transition width.

## Road and settlement generation: the state of documentation

In practice, road networks connecting settlements follow a recognizable pipeline: place settlements using terrain-scoring heuristics (flat ground, water access, resource proximity), compute a Delaunay triangulation of settlement positions, extract an MST for guaranteed connectivity with optional loop-creating re-additions, route each edge of the resulting graph across the terrain cost map using A* or Dijkstra with slope/biome/water-crossing weights, simplify the resulting waypoint paths, fit splines for smooth curves, modify terrain under the road corridor, and update exclusion masks for subsequent vegetation passes. This pipeline is described *piecemeal* across many sources—Galin et al. (2010) for cost functions and spline fitting, Vazgriz for the Delaunay-MST pipeline, Red Blob Games for pathfinding implementation, Cepero for settlement scoring—but **no single source describes the complete integrated pipeline from settlement placement through final road rendering**.

What a concrete cost-map-based routing step looks like: a 100×100 to 300×300 grid over the terrain, where each cell stores a base cost of 1.0 modified by slope (exponential increase with gradient, infinity above ~30°), biome (forest 2–3×, marsh 5–10×, plains 1×), water (rivers 10–50× for bridge construction, lakes/ocean infinity), and existing roads (0.1–0.5× discount). A* searches this graph with connectivity mask k = 5 (80 candidate directions per node, 11.3° angular resolution per Galin et al.) and a heuristic of straight-line road cost to the goal. The output is a sequence of grid cells that is then simplified and spline-fitted.

**Specific documentation gaps** that developers will encounter: (1) No published source provides a complete, tested set of cost-function weights—every implementation requires experimental tuning, and the relative magnitudes between slope, biome, and water costs dramatically change road behaviour. (2) The transition from MST graph to individually-routed roads is described in isolation by different sources but the interaction (e.g., how to handle road segments that share portions of their path, forming natural T-junctions) is underspecified. (3) Terrain modification under roads—excavation, embankment, blending profiles—is covered well by Galin et al. (2010) but almost nowhere else. (4) Settlement placement scoring functions are described conceptually by several papers (Emilien et al. 2012, ScienceDirect 2023) but no open-source reference implementation exists. (5) Landmark placement has no published algorithmic treatment at all—this is a genuine gap, not an oversight. (6) Hub-and-spoke network topology, organic branching, and agent-based road growth have no implementation-level documentation beyond brief descriptions. Developers building settlement-connected road networks should expect to assemble the pipeline from components documented in separate sources and to spend significant time on parameter tuning.

## How Pass 5a and 5b connect end-to-end

Pass 5 transforms raw terrain geometry into a populated, connected game world. The two sub-passes divide at a clean data boundary: Pass 5a produces spatial structure and classification; Pass 5b consumes that structure to place objects and generate infrastructure.

**Pass 5a stages and their outputs:**

1. **Region partitioning** (Voronoi, Lloyd-relaxed). Input: terrain heightmap, world bounds. Output: a set of polygons (Voronoi cells or Delaunay dual mesh) with adjacency relationships stored as a region graph.
2. **Climate simulation** (temperature, moisture, wind). Input: heightmap, latitude, parameters. Output: per-region or per-cell temperature and moisture scalar fields.
3. **Biome assignment** (Whittaker-table lookup). Input: elevation field, moisture field. Output: per-region biome classification enum and associated parameter sets (vegetation density weights, colour palettes, object type tables).
4. **Biome transition blending**. Input: biome map, region graph. Output: per-cell blend weights for adjacent biomes at region boundaries, typically stored as a multi-channel texture or per-vertex interpolation weights.

**Pass 5b stages, consuming 5a outputs:**

5. **Density field construction**. Consumes: biome map (biome density weights), noise layers, heightmap. Produces: per-object-type density scalar fields.
6. **Slope and altitude mask generation**. Consumes: heightmap. Produces: slope and altitude mask layers.
7. **Exclusion mask composition**. Consumes: water level, heightmap, any pre-existing structures. Produces: composite exclusion mask.
8. **Hierarchical object placement with clustering**. Consumes: density fields, slope/altitude masks, exclusion masks. Produces: placed object instances (positions, types, scales, rotations), updated exclusion mask with object footprints.
9. **Settlement placement**. Consumes: biome map, heightmap, water bodies, region graph. Produces: settlement positions with population estimates.
10. **Road network topology** (Delaunay → MST → re-addition). Consumes: settlement positions. Produces: road graph (which settlements to connect).
11. **Road routing** (A*/Dijkstra on cost map). Consumes: road graph, heightmap, biome map, water data. Produces: per-road waypoint sequences.
12. **Road spline fitting and terrain modification**. Consumes: waypoint sequences, heightmap. Produces: smooth road curves, modified heightmap, road exclusion masks.
13. **Post-road vegetation adjustment**. Consumes: updated exclusion masks from roads. Produces: final placed-object set with road corridors cleared.

**Data structures at boundaries:** The heightmap is a 2D float array (or tiled float texture) handed off from terrain generation and modified in-place by road terrain flattening. The biome map is a per-cell enum (or per-region enum with blend weights at boundaries). The region graph is an adjacency list of polygon IDs. The density field is a per-cell float array per object type. Exclusion masks are per-cell binary or float arrays, accumulated across stages. Placed objects are stored as an array of structs (position, type ID, scale, rotation, footprint radius). The road network is a graph of settlement IDs with per-edge waypoint sequences and spline parameters. Downstream passes—rendering, gameplay logic, AI navigation mesh generation—consume the final placed-object list, the road network graph and meshes, and the modified heightmap.

## Source landscape for Pass 5a and 5b

**Interactive tools and demos**

Red Blob Games' mapgen4 (redblobgames.com/maps/mapgen4/) is the most hands-on exploration of the full pipeline from terrain painting through biome assignment, with rainfall simulation and river generation—uniquely valuable because every parameter is adjustable in real time, which no other source offers. Azgaar's Fantasy Map Generator (azgaar.github.io/Fantasy-Map-Generator/) is the only freely available tool that integrates settlements, roads, trade routes, nations, and names on procedurally generated terrain, making it invaluable for seeing how all Pass 5 systems interact at once even though its internals are not deeply documented. The kchapelier/poisson-disk-sampling library (github.com/kchapelier/poisson-disk-sampling) provides a production-ready variable-density Poisson disk implementation with a `distanceFunction` callback, which is the most directly usable density-field-modulated placement code available.

**Tutorials and educational series**

Red Blob Games' pathfinding articles (redblobgames.com/pathfinding/a-star/) are the definitive interactive tutorial for A* and Dijkstra with cost maps, uniquely valuable because they provide complete working implementations in Python, C++, and C# alongside interactive visualizations that no textbook matches. Sebastian Lague's Poisson Disc Sampling video and GitHub repository (github.com/SebLague/Poisson-Disc-Sampling) is the most-referenced Unity implementation of Bridson's algorithm, uniquely clear as a teaching resource though it covers only uniform-density placement and does not address variable-density modulation. Red Blob Games' polygon map generation article (www-cs-students.stanford.edu/~amitp/game-programming/polygon-map-generation/) remains the most comprehensive single-page tutorial on the full terrain-to-biome-to-road pipeline using Voronoi dual meshes, and its modular architecture (roads as a separate module consuming the core map) is a design pattern worth studying. Martin Devans' procedural road generation blog post (martindevans.me/game-development/2015/12/11/Procedural-Generation-For-Dummies-Roads/) is the only detailed walkthrough of tensor-field road generation with open-source code, and it includes honest assessments of failure modes that academic papers omit.

**Academic papers**

Galin, Peytavie, Maréchal, and Guérin, "Procedural Generation of Roads" (Computer Graphics Forum, 2010) is the foundational paper on cost-function-based road routing with anisotropic shortest paths, connectivity masks, bridge/tunnel detection, clothoid smoothing, and terrain modification—no other single paper covers all of these components. Parish and Müller, "Procedural Modeling of Cities" (SIGGRAPH, 2001) introduced L-system road generation with global goals and local constraints and remains the most-cited work in procedural city generation. Lane and Prusinkiewicz, "Generating Spatial Distributions for Multilevel Models of Plant Communities" (Graphics Interface, 2002) provides the only rigorous treatment of deformation-kernel-based clustering for vegetation placement, with controllable Hopkins-index distributions that no other method matches for statistical precision. Fischer et al., "AutoBiomes" (The Visual Computer, 2020) is the most complete published system for multi-biome procedural vegetation with per-asset clustering probability—uniquely valuable for its end-to-end pipeline from climate simulation through asset placement in a game engine. Emilien et al., "Interactive Procedural Generation of Fields with Emerging Paths" (The Visual Computer, 2012) introduces interest-map-based settlement placement with iterative road feedback, a technique not described elsewhere.

**Dev blogs and GDC talks**

The Horizon: Zero Dawn GDC 2017 talk by van Muijden (Guerrilla Games) is the industry-standard reference for GPU-based runtime procedural placement at AAA scale, and the only public source describing how a shipped open-world game handles density maps, artist-driven rule graphs, and GPU compute placement simultaneously. Mick West's "Random Scattering: Creating Realistic Landscapes" (Game Developer magazine) is the clearest practitioner's guide to the ontogenetic vs. teleological distinction in vegetation scattering and provides the grid-perturbation method that remains viable for simpler projects. Miguel Cepero's Procedural World blog (procworld.blogspot.com, 2011) provides the most concrete open description of an integrated settlement-placement-to-road-generation pipeline, including political territory formation, that is not paywalled behind an academic publisher. The ezEngine procedural generation documentation (ezengine.net) is uniquely valuable as a complete, publicly documented game-engine implementation of density-based placement with footprint radii, surface-type filtering, and Horizon: Zero Dawn-inspired architecture. The Tobias Mansfield-Williams citygen project (tmwhere.com/city_generation.html) provides a browser-playable interactive demo of Parish and Müller-style L-system road generation with source code—the only implementation that lets you see the algorithm running in real time.
