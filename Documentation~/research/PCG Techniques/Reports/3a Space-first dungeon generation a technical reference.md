# Space-first dungeon generation: a technical reference

Space-first dungeon generators produce geometry directly — carving rooms, corridors, and caves into a tile grid without first building a structural or progression graph. This pass covers the eight core techniques that constitute this family, ordered from the most structured (BSP partitioning) to the most organic (cellular automata caves), then the support techniques that wire them together (connectivity verification, MST, Delaunay triangulation). Every technique here takes a blank grid as input and returns playable geometry as output. A developer reading each section should be able to implement a basic version and diagnose common failure modes without additional sources. Where documentation is thin or a claim could not be verified against a consulted source, this is noted explicitly.

---

## 1. BSP dungeon generation

Binary Space Partitioning is the most architecturally structured of the space-first methods. It produces axis-aligned, non-overlapping rooms distributed evenly across a rectangular map, connected by corridors that follow the BSP tree hierarchy. The technique is well-suited to games that want a "designed" dungeon feel — NetHack and Dungeon Crawl: Stone Soup both use BSP-derived layouts. It is ill-suited when organic or irregular geometry is desired, because the output inherits the rigid, grid-like character of recursive rectangular subdivision. BSP is among the best-documented dungeon generation algorithms, with complete implementations available in C, C++, JavaScript, Rust, and C#.

The algorithm operates in three phases. **Phase 1 builds the BSP tree:** begin with a single rectangle covering the entire map, then recursively split it into two child rectangles by choosing a random axis (horizontal or vertical) and a random split position along that axis. The JICE article on RogueBasin specifies that the split position must not be too close to the container border — each child must be large enough to hold a room. Recursion terminates when leaf nodes approximate the desired room size. The Liapis PCG Book chapter formalizes this as: "a leaf node is not split any further if it is below a minimum size (we will consider a minimal width of w=4 and minimal height of h=4 for this example)." **Phase 2 places rooms:** for each leaf node, generate a room with random dimensions that fits within the leaf's bounds with some padding. The JICE article states: "we create a room with random size in each leaf of the tree. Of course, the room must be contained inside the corresponding sub-dungeon. Thanks to the BSP tree, we can't have two overlapping rooms." The libtcod tutorial subtracts 2 from the node dimensions to ensure rooms do not touch node borders, preventing joined rooms. **Phase 3 connects rooms:** traverse the tree bottom-up, connecting each pair of sibling nodes. The JICE article describes: "we loop through all the leafs of the tree, connecting each leaf to its sister. If the two rooms have face-to-face walls, we can use a straight corridor. Else we have to use a Z shaped corridor." At each level, the algorithm connects the two subtrees represented by sibling nodes, ascending until the root's children are linked. This guarantees full connectivity — every room is reachable because every sibling pair is connected.

The main variants concern split-position constraints and room sizing. The JICE article notes that restricting the split position to 0.45–0.55 of the container produces homogeneous sub-dungeons, while allowing 0.1–0.9 produces heterogeneous ones with dramatically different room sizes. The eskerda.com JavaScript implementation adds **aspect-ratio filtering**: if either child rectangle's width-to-height ratio falls below a threshold (0.45 in their code), the split is rejected and retried, preventing long, thin partitions. The 7YRL blog implementation skips roughly 10% of splits on small partitions to allow occasional larger rooms — useful for placing "special" rooms later. Herbert Wolverson's Rust tutorial implements a variant that subdivides each rectangle into four quadrants rather than two halves, then iteratively attempts to place rooms into random sub-rectangles, checking a 2-tile expanded buffer for overlap. This produces results that are "very Nethack like" while using a simpler placement loop rather than strict tree traversal. The Liapis PCG Book notes an advanced use of the BSP hierarchy for theming: "Non-leaf partition cells can be used to define groups of rooms following the same theme; for instance, a section of the dungeon may contain higher-level monsters," with a single corridor entrance creating natural chokepoints.

The primary failure mode is **degenerate splits**: splitting along the same axis multiple times in succession produces extremely elongated partitions. The RogueBasin Talk page notes this explicitly. The fix is axis alternation or aspect-ratio rejection. A second failure is **stack overflow from overly strict ratio constraints** — the eskerda.com implementation documents that "too aggressive ratios will follow with growing the stack size out of the heap limit" when the recursive retry loop cannot find a valid split. The fix is a maximum retry count or relaxed constraints. A subtler problem is **visual monotony**: because every room occupies its own partition cell, the output can look grid-like. The JICE article suggests allowing some rooms to fill their entire leaf ("If we allow some rooms to fill the whole leaf, we can even have less boring dungeons"), which breaks the uniform padding and creates occasional large chambers.

BSP connects naturally to several other techniques. Its room output feeds directly into MST/Delaunay connectivity (Section 8) as an alternative to the tree-based corridor strategy. Cellular automata (Section 5) can be applied as a single-iteration post-process to erode the hard rectangular edges, producing a hybrid of structured layout and organic texture — Wolverson's tutorial demonstrates this through its builder-chain system. Room templates (Section 6) can replace the random room-generation step in leaf nodes, injecting hand-designed content into the BSP scaffold. In structure-first pipelines, a mission graph can drive which BSP partitions receive which room types, adding progression awareness to the spatial layout.

## 2. Room-and-corridor methods

Where BSP works top-down as an "architect" dividing space, room-and-corridor methods work bottom-up, placing rooms one at a time and connecting them with corridors. The output tends to feel more organic and sprawling than BSP — rooms vary in spacing, corridors meander, and the layout lacks the implicit grid. Two major variants dominate the literature: **random placement with pathfinding connection** (place all rooms, then connect) and **incremental growth** (grow the dungeon outward from a seed). Both are extremely well-documented.

**Mike Anderson's Dungeon-Building Algorithm** (RogueBasin, used in the Tyrant roguelike) is the canonical incremental-growth approach. The algorithm starts by filling the entire map with solid earth, then digs a single room in the center. It then repeatedly: picks a random wall tile adjacent to an existing floor tile, selects a feature type (room, corridor, circular arena, vault) based on weighted probabilities, checks whether the feature's bounding rectangle plus a one-tile wall buffer consists entirely of solid earth, and if so, carves it. Anderson states: "the rule is to always add to the existing dungeon. This makes the connections look good, and also guarantees that every square is reachable." The rejection rate is high — "Tyrant tries to add 300 or so features to each dungeon, but usually only 40 or so make it past this stage" — but processing time is negligible per attempt. Feature-type weights control the dungeon's character entirely: heavy corridor weights produce labyrinthine layouts; heavy room weights produce open complexes.

**Bob Nystrom's Rooms and Mazes** algorithm represents the random-placement-then-connect family. It operates in four steps. First, rooms of random size are placed at odd-aligned grid positions, rejecting any that overlap. Second, a maze generator (randomized flood fill) fills every remaining solid region between rooms, assigning each room and each maze section a unique region ID. Third, "connectors" — wall tiles adjacent to two different regions — are identified, and a spanning-tree algorithm merges regions by opening connectors one at a time until all regions are unified. Nystrom writes: "This process is agnostic about rooms and mazes. It just deals in 'regions'. That means it can connect rooms directly to other rooms sometimes." Fourth, dead ends are iteratively removed: any tile with only one open neighbor is filled back in as wall. This "uncarving" step eliminates winding maze passages that would be tedious to traverse. A tunable **extra-connector chance** controls how many non-tree connectors remain, creating loops — setting this to zero produces a perfect (tree-structured) dungeon; higher values add alternate paths. Nystrom notes the algorithm's main weakness: "It tends to produce annoyingly windy passages between rooms. You can tune that by tweaking your maze generation algorithm, but making the passageways less windy tends to make them wander to the edge of the dungeon."

Failure modes differ between variants. Anderson's incremental growth can leave large map regions unfilled if the rejection rate is too high or if early features cluster in one area. Dead-end corridors that lead nowhere are common; the Future Data Lab reimplementation specifically modifies the algorithm to "reduce the number of dead ends and increase the likelihood of a room being generated at the end of a corridor." Nystrom's approach avoids dead corridors through explicit dead-end removal but produces odd-aligned geometry that gives a slightly artificial look. Both approaches can produce corridors that enter rooms at awkward positions (e.g., corners); the PCG Wiki suggests marking tiles adjacent to room entrances as impassable to prevent two corridors entering a room side by side. A general failure across random-placement methods is **room packing failure**: if room sizes are too large relative to the map, most placement attempts fail, and the dungeon ends up sparse. The fix is constraining room dimensions relative to map area or using a physics-based separation step (as in the TinyKeep pipeline described in Section 8).

These methods connect to almost every other technique in this pass. Nystrom's region-merging step is essentially a connectivity verification pass (Section 7). The corridor-connection step can be replaced with MST/Delaunay triangulation (Section 8) for more controlled topology. Room templates (Section 6) can substitute for randomly generated rooms. The incremental-growth variant is the direct ancestor of agent-based diggers (Section 4) — Anderson's algorithm is sometimes classified as agent-based, since the "feature placer" acts as an agent growing the dungeon outward.

## 3. Random walk and drunkard's walk

The drunkard's walk is the simplest space-carving algorithm that produces playable geometry. A single agent starts at a position on a solid grid, moves one step in a random cardinal direction, and converts each visited tile to floor. This continues until a target percentage of the map is carved. The technique produces cave-like layouts without any explicit room or corridor concept, making it well-suited to natural environments — cavern systems, eroded ruins, organic hive structures. It is ill-suited to any context requiring rectangular rooms, chokepoints, or structured progression, unless combined with other methods.

The complete algorithm, as described in Herbert Wolverson's Rust tutorial and the RogueBasin "Random Walk Cave Generation" article, is: initialize the grid as solid wall; pick a central starting point and mark it floor; while the floor percentage is below the target (typically **40–50%**), spawn a walker at the origin with a finite lifetime (e.g., 400 steps); each step, roll a four-sided die to choose north, south, east, or west, clamp the position to stay within map borders (at least 2 tiles from edges), and mark the current tile as floor; when the walker's lifetime expires, spawn a new walker. Wolverson implements three presets that demonstrate how parameters shape output: `open_area` (all walkers start at center, lifetime 400, 50% floor) produces a single large cavern; `open_halls` (walkers spawn at random positions, lifetime 400, 50% floor) produces a sprawling multi-chamber cave; `winding_passages` (random spawn, lifetime 100, 40% floor) produces distinct pockets connected by narrow paths.

The key variants are **spawn mode**, **directional bias**, and **multi-walker execution**. Starting all walkers from the same point guarantees connectivity but tends toward a "boring blob" — the jrheard blog notes that "sometimes it'll generate super cool tunnely cavey levels, and sometimes it'll just generate a boring-looking blob." Random spawn locations produce more varied layouts but risk disconnected regions, requiring post-processing via flood fill. **Biased walks** — where the walker has a higher probability of continuing in its current direction — produce longer, more corridor-like passages rather than amorphous blobs. The PCG Wiki recommends biasing movement toward the map center to prevent edge-clustering. A **tunnel-based variant** (used in the FreeCodeCamp implementation) replaces single-cell steps with multi-cell tunnel segments of random length, producing more structured output. The **symmetric variant**, implemented in Wolverson's later chapters, mirrors each carved tile across one or more axes, producing alien or insectoid-looking layouts.

A single walker from a single origin produces **guaranteed connectivity** — each step is adjacent to the previous one, so the carved region is always a single connected component. This is the technique's key advantage over cellular automata, which frequently produce disconnected regions. The primary failure modes are: **edge-clustering**, where walkers bounce against map borders and produce unnatural flat walls (fix: increase border buffer or add center bias); **too-open results**, where center-spawn walkers produce featureless blobs (fix: reduce lifetime, increase floor target, use random spawn); and **unpredictable coverage**, where the algorithm cannot guarantee specific layout features. Disconnected regions arise only with the random-spawn variant and are fixed with Dijkstra-map culling (Wolverson's approach) or flood-fill-and-connect.

The drunkard's walk is a natural complement to cellular automata (Section 5). An SBGames 2020 paper notes that "Drunkard Walk and Cellular Automata can be used together to create cave systems. As the Drunkard Walk algorithm generates chaotic maps, and CA is specifically made to work with chaotic inputs, they are naturally associative." A common pipeline is: carve a rough layout with random walks, then smooth it with one or two CA iterations. The technique also serves as a building block for agent-based diggers (Section 4), which extend the walker with behavioral rules for room carving and corridor width.

## 4. Agent-based diggers

Agent-based diggers extend the random walk concept by giving each carving agent behavioral rules: when to turn, when to carve a room, how wide to dig, and when to stop. The output occupies a middle ground between the pure chaos of drunkard's walks and the rigid structure of BSP — recognizable rooms and corridors emerge, but with an organic, exploratory character. The technique is well-suited to games that want dungeons with varied room sizes and winding corridors that feel "discovered" rather than "designed." Documentation quality varies: the Liapis PCG Book provides formal pseudocode for two variants, and the Cogmind developer (Kyzrati) has published detailed descriptions of a production tunneling system, but the general technique is less standardized than BSP or cellular automata.

The Liapis PCG Book describes two formal variants. The **blind stochastic digger** maintains two escalating probabilities: a chance to change direction (Pc, starting at 5%, incrementing by 5 each step without a turn, resetting to 0 on turn) and a chance to place a room (Pr, same escalation). Each step, the agent digs one tile forward, then rolls against Pc and Pr independently. Room dimensions are randomized between 3 and 7 tiles. The linear escalation ensures that long straight corridors and long room-free stretches become progressively less likely, producing a natural rhythm of corridor→room→corridor. The **look-ahead digger** adds collision checking: before placing a room or corridor, the agent verifies that the candidate area does not intersect any existing room. This prevents overlapping rooms but introduces a new failure mode — the agent can become trapped in an area where no valid placement exists, terminating prematurely with large unexplored regions. The fix is to teleport the agent to a random wall tile adjacent to existing floor, or to spawn a new agent elsewhere.

The most sophisticated documented implementation is **Cogmind's tunneling algorithm** by Kyzrati (Grid Sage Games). Its key innovation is that tunnelers **mutate their own parameters** as they move — width, turning probability, room-carving probability, and padding all drift over time, producing varied environments across a single map. Tunnelers can spawn child tunnelers and "room excavators" at branch points, some of which are deferred for later activation. Loops are created by allowing tunnelers to detect and connect with existing excavated space at an adjustable probability. The result is dungeons with variable corridor widths (1-wide maze-like to 3-wide combat corridors), rooms that organically branch off corridors, and natural junctions at turns or width changes. This is the closest a pure space-first agent system comes to producing designer-quality layouts, but parameter tuning is extensive — Liapis notes that "the impact of the AI behavior's parameters on the generated dungeons' appearance is difficult to guess without extensive trial-and-error."

Anderson's Dungeon-Building Algorithm (Section 2) is sometimes classified as agent-based, since its feature-placement loop behaves like a single agent growing the dungeon outward. The distinction is one of emphasis: Anderson's approach treats rooms and corridors as discrete features selected from a weighted menu, while true agent-based systems model the digger as a persistent entity with internal state. The practical difference is that agent-based systems more naturally produce layouts where corridor width, room density, and layout character vary across the map, while Anderson's approach tends toward uniform feature density. Failure modes shared across all agent-based methods include **dead-end corridors** (corridors that terminate without connecting to anything — mitigated by post-processing or by biasing agents toward unexplored regions), **corner clustering** (agents digging into corners and unable to escape — mitigated by movement heuristics), and **unplayable room-to-corridor ratios** (too many rooms or too many corridors — mitigated by tuning the escalation parameters). Agent-based diggers combine naturally with connectivity verification (Section 7) as a safety net and with room templates (Section 6) as a replacement for the agent's stochastic room-carving step.

## 5. Cellular automata for cave-like spaces

Cellular automata produce the most visually organic output of any space-first technique — smooth, rounded cave walls with natural-looking chambers and passages that would be extremely difficult to design by hand or produce with rectangular room methods. The technique is well-suited to natural cave systems, forest clearings, eroded ruins, and any environment where regularity would break immersion. It is ill-suited to structured dungeons with distinct rooms and corridors unless combined with other methods. Documentation is excellent: the RogueBasin article by Jim Babcock, Jeremy Kun's mathematical treatment, and the Liapis PCG Book chapter together provide complete coverage of rules, parameters, and failure modes.

The algorithm initializes a 2D grid by randomly setting each interior cell to wall with probability **fillprob** (typically 40–45%), with border cells always wall. It then applies a transition rule simultaneously to all cells for N iterations, writing results to a second buffer before swapping. The foundational rule is the **"4-5 rule"**: a cell becomes or remains wall if 5 or more of its 9-cell Moore neighborhood (including itself) are wall. As Babcock writes on RogueBasin: "a tile becomes a wall if it was a wall and 4 or more of its eight neighbors were walls, or if it was not a wall and 5 or more neighbors were. Put more succinctly, a tile is a wall if the 3x3 region centered on it contained at least 5 walls." Wolverson's Rust implementation uses a slight variation: a tile also becomes wall if it has **zero** wall neighbors (filling isolated floor tiles), and uses 15 iterations with 55% initial wall probability. Jeremy Kun formalizes an alternative as **B678/S345678** in Life-like notation: a dead cell is born with 6–8 live neighbors; a live cell survives with 3–8. He notes this "is designed to prune the sparse areas of the grid, and fill in the dense areas by patching up holes."

The RogueBasin article describes an **enhanced two-phase rule** that produces superior results: Phase 1 (4 iterations) uses two neighborhood radii simultaneously — a cell becomes wall if R1 (3×3) contains ≥5 walls OR if R2 (5×5 minus corners, 21 cells) contains ≤2 walls. The R2 condition fills in the centers of large open areas that the standard rule leaves hollow. Phase 2 (3 iterations) applies the standard 4-5 rule for smoothing. Recommended parameters are fillprob=40%, R1 cutoff=5, R2 cutoff=2. Initial density is the single most impactful parameter: Jeremy Kun notes that "increasing [initial density] a mere 5% [from 45% to 50%] we get a picture [with] many more disconnected caverns." Fewer iterations leave noisy artifacts; more iterations converge toward smooth but potentially featureless caves. The B678/S345678 rule "almost always converges within 15 steps." A **resolution-scaling variant** (Kun) generates at low resolution, upscales by splitting each cell into four, then smooths with B5678/S5678 — this avoids the "awkward camouflage pattern" that fine-grid CA tends to produce. Johnson et al. (cited in Liapis) generate infinite caves by processing 50×50 room chunks in 3×3 groups, drilling tunnels between disconnected largest-open-spaces, then running two more CA passes to smooth the connections — the entire process takes "less than a millisecond on a modern computer."

**Disconnected caverns** are the primary failure mode and are almost unavoidable at higher fill percentages. The standard fix is flood fill from a seed point: any floor tile unreachable from the seed is converted back to wall (or the largest connected component is kept and everything else discarded). A RogueBasin contributor advises: "pick a random, open point on the map and flood fill. Any open point outside the flood filled portion gets turned back into a wall. I then check to see if the flood filled portion of the map is more than some threshold percent of the map. If the percent is below this, I just start over." Wolverson uses Dijkstra maps for the same purpose — any tile with distance equal to f32::MAX is unreachable and gets filled. Other failure modes include **too-open spaces** at low fill percentages (the map becomes a single featureless cavern) and **single-tile artifacts** (isolated wall or floor pixels that survive iteration — fixable with additional smoothing passes). A practical variant for preventing map bisection is **horizontal blanking** (documented on GitHub by AdamWhiteHat): after random fill but before CA iterations, clear a horizontal or vertical strip through the center, preventing solid walls from cutting the map in half.

Cellular automata combine with nearly every other technique in this pass. The most natural combination is with random walks (Section 3): carve a rough layout with drunkard's walks, then smooth with 1–2 CA iterations to round the edges. CA can also serve as a post-process on BSP or room-and-corridor output to erode rectangular geometry into something more natural — Wolverson's builder-chain system demonstrates this. CA output feeds directly into connectivity verification (Section 7) since disconnected regions are the norm. For structure-first pipelines, CA generates the cave geometry that fills regions defined by a higher-level mission graph, with connectivity between mission nodes ensured by drilling tunnels at the narrowest separation points and re-smoothing.

## 6. Room templates and prefab placement

Room templates solve a problem that purely procedural methods cannot: guaranteeing specific gameplay moments, hand-tuned aesthetic quality, or puzzle configurations within a procedurally generated layout. A prefab is a pre-designed room stored as a tile grid (or equivalent data structure) and placed into the dungeon during generation. The technique is well-suited to any game that needs repeatable, designer-authored encounters — boss arenas, puzzle rooms, shops, spawn chambers — embedded in procedural structure. It is ill-suited as a standalone method because variety is bounded by the number of authored templates. **Spelunky's level generation** is the gold standard for prefab-driven design and is extremely well-documented.

Spelunky divides each level into a **4×4 grid of 16 rooms**, each 10×8 tiles. The generator first computes a solution path from a random top-row room to a random bottom-row room using a biased random walk: at each step, a 40% chance to go left, 40% right, and 20% to drop down. Rooms along the solution path are assigned types that guarantee the necessary exits — Type 1 (left and right exits), Type 2 (left, right, and bottom), Type 3 (left, right, and top). Non-path rooms receive Type 0 (no guaranteed exits). Each type has a pool of hand-designed templates stored as text matrices where characters encode tile types: '0' for empty, '1' for wall, 'L' for ladder. Critically, some template positions contain **wildcard characters** (e.g., '6') that are replaced at runtime by random obstacles — spikes, webs, arrow traps — in a second pass. A third pass places monsters by scanning for valid empty locations. The Liapis PCG Book summarizes this as three phases: "layout phase" (template selection), "obstacle generation" (wildcard resolution), and "monster generation" (entity placement). This three-phase structure is the canonical pattern for prefab-driven generation.

For non-grid-based systems, the key challenges are **overlap checking**, **connection-point alignment**, and **rotation**. Overlap checking is typically grid-based: before placing a template, verify that all target cells are empty (or wall, depending on convention). Ondřej Nepožitek's Edgar system (documented in detail on his blog) formalizes connection points: "The algorithm allows game designers to explicitly specify possible door positions of individual room shapes." Door positions can be defined explicitly per room, defined by length and distance from corners, or allowed anywhere along walls. The algorithm takes a connectivity graph as input and assigns a room shape and position to each node such that no two rooms intersect and all neighbors can be connected by doors — supporting cycles, not just trees. A simpler approach used in many roguelikes is to define rooms with fixed exit directions (NESW pattern) and select templates based on required connectivity, as described in the GameDev Academy Unity tutorial. Rotation and mirroring multiply the effective template pool: a single L-shaped room template yields up to 8 variants under the dihedral group.

Failure modes center on **template exhaustion** (too few templates for a required exit configuration — fix: design templates for all needed connectivity patterns), **placement failure** (no valid position found — fix: retry with different templates or fall back to procedural rooms), and **predictability** (players recognize repeated templates — fix: wildcard tiles, rotation, and a large template pool). The fundamental tradeoff is designer control versus variety: more hand-designed rooms yield more predictable quality but less surprise. Prefabs combine with every room-based technique: BSP (Section 1) can select templates for leaf nodes instead of generating random rectangles; room-and-corridor methods (Section 2) can inject prefabs at special positions; the Delaunay/MST pipeline (Section 8) treats prefabs as nodes in the connectivity graph. Wolverson's builder-chain system layers prefabs as a modifier on any base generator, injecting hand-designed sections and vault rooms into otherwise procedural output.

## 7. Connectivity verification

Connectivity verification is not a generation technique but a validation and repair step applied after generation to ensure that all playable space is reachable. It answers a simple question: starting from any given tile, can a player reach every other floor tile? Every technique in this pass except Anderson's incremental-growth method and single-origin drunkard's walks can produce disconnected regions, making connectivity verification essential in most pipelines.

**Flood fill** (BFS or DFS from a seed tile) is the standard approach. Starting from the player's spawn point, mark every reachable floor tile. Any floor tile that remains unmarked is unreachable. The two standard responses are **region deletion** (convert unreachable tiles back to wall, keeping only the largest connected component) and **corridor drilling** (carve a corridor from the nearest point of the unreachable region to the nearest point of the main region). Wolverson's tutorial uses Dijkstra maps as a generalized flood fill: build a distance map from the starting position, then treat any tile with distance equal to the maximum float value as unreachable. This simultaneously identifies unreachable regions and locates the most distant reachable tile — a natural placement for exit stairs. Bob Nystrom's Rooms and Mazes algorithm integrates connectivity verification into the generation process itself: each room and maze section receives a unique region ID, and a spanning-tree algorithm merges regions by opening "connector" wall tiles between them. He writes: "Remove any extraneous connectors. There are likely other existing connectors that connect the two regions that just merged. Since they no longer connect two separate regions and we want a spanning tree, discard them." Extra connectors are randomly kept to create loops.

**Union-Find** (disjoint set union) is an efficient alternative to repeated flood fills when tracking connectivity incrementally — for instance, during corridor carving, where each new corridor potentially merges two previously disconnected regions. Both operations (find and union) run in effectively constant time with path compression and union by rank. Union-Find is particularly natural in Kruskal's MST algorithm (Section 8), where it tracks which rooms are already connected. The PCG Wiki notes that "the usual approach is to use a spanning tree algorithm to validate that all rooms are connected correctly, by adding rooms at each endpoint of the corridor to the spanning tree." In practice, connectivity verification should be applied **after all geometry is finalized** but before entity placement. For cellular automata caves, it is non-negotiable. For BSP dungeons with tree-based corridor connection, it serves as a safety net. The most common pipeline pattern is: generate geometry → flood fill from spawn → delete or connect unreachable regions → place entities.

## 8. MST and Delaunay triangulation for room connectivity

Delaunay triangulation and minimum spanning trees are not generation methods but **connectivity algorithms** applied to a set of room centers after rooms have already been placed. They answer: given N rooms at known positions, which pairs should be connected by corridors? The approach is extremely well-documented, with complete implementations available from Vazgriz, the Game Developer (Gamasutra) article on the TinyKeep algorithm, Jagaco Games, and the KidsCanCode Godot tutorial.

The pipeline works in three steps. First, treat each room's center as a point and compute a **Delaunay triangulation** — typically via the Bowyer-Watson algorithm, which incrementally inserts points and retriangulates. Delaunay triangulation is preferred over arbitrary triangulation because it maximizes the minimum angle of all triangles, avoiding sliver triangles and ensuring that connections are geometrically well-distributed between nearby rooms. Second, extract a **minimum spanning tree** from the Delaunay edges using Prim's or Kruskal's algorithm. The MST guarantees that every room is reachable with the minimum total corridor length. Vazgriz writes: "The MST guarantees that every room will be reachable. However, since it is a tree, it contains no cycles. There is only a single path from one room to any other room." Third — and critically — **add back a random subset of non-MST Delaunay edges** to create loops. The TinyKeep developer uses **15%** of remaining edges; the Game Developer article author (ADNZZZZZ) found **8–10%** better for their purposes; Vazgriz used **12.5%**. This percentage is the primary tuning parameter: too low produces dead-end-heavy linear dungeons; too high produces dungeons where every room connects to many others, eliminating directionality.

Corridors are then carved along the selected edges. The simplest approach is L-shaped (Manhattan) corridors: go horizontal until aligned, then vertical. Vazgriz describes a more sophisticated method using **A* pathfinding** where the cost function makes it cheaper to traverse existing corridors, encouraging shared hallways. Jagaco Games notes that extending room exits outward before connecting them Manhattan-style produces better-looking results than connecting room centers directly. Variants include the **Gabriel graph** and **relative neighborhood graph**, which are subgraphs of the Delaunay triangulation with progressively fewer edges (MST ⊆ RNG ⊆ Gabriel ⊆ Delaunay). These can replace the "MST + random edges" approach with a naturally connected graph of moderate density, though this is not commonly implemented in practice (unconfirmed whether any shipping game uses these alternatives). Vazgriz extended the entire pipeline to 3D using Delaunay tetrahedralization, with staircases replacing flat corridors for vertical connections.

Failure modes include **crossing corridors** (when added non-MST edges produce paths that intersect — mitigated by using A* with existing-corridor awareness), **overly long corridors** (if rooms are unevenly distributed — mitigated by physics-based room separation before triangulation, as in TinyKeep), and **corridors through other rooms** (when an L-shaped path between two rooms passes through a third — mitigated by pathfinding-based corridor carving rather than naive straight lines). The technique connects upstream to any room-placement method (BSP, random placement, physics separation) and downstream to connectivity verification (Section 7) as a final safety check.

## Cellular automata caves vs. random walks vs. agent diggers

These three techniques all produce organic, non-rectangular geometry, but they differ sharply in connectivity guarantees, output character, and designer control. **Cellular automata** produce the smoothest, most natural-looking caves — rounded chambers, flowing walls, organic shapes — but offer **no connectivity guarantee**. Disconnected regions are the norm, and post-processing (flood fill, corridor drilling) is always required. Designer control comes through fill probability, rule set, and iteration count, all of which have intuitive effects on density and smoothness. CA is the right choice when you want visually natural caves and are willing to pay the post-processing cost. **Random walks** produce guaranteed connectivity from a single origin (each step is adjacent to the last) but generate less natural shapes — more tunneled, more angular, with a tendency toward amorphous blobs at center-spawn or scattered pockets at random-spawn. The technique offers fewer parameters and less predictable fine-grained control. It is the right choice when connectivity must be guaranteed cheaply and the aesthetic goal is "eroded tunnels" rather than "natural caverns." **Agent-based diggers** produce the most dungeon-like output of the three — recognizable rooms connected by corridors of varying width — but require the most parameter tuning. They offer the most designer control (turn frequency, room probability, corridor width, parameter mutation) at the cost of the most trial-and-error to achieve a specific look. They are the right choice when the goal is an organic dungeon with identifiable rooms and corridors, and the developer is willing to invest in parameter exploration.

In practice, these techniques are often combined. Random walks sketch a rough backbone, CA smooths it into natural-looking geometry, and agent rules inject room-like chambers along the path. The SBGames 2020 paper confirms this: drunkard's walk and CA "are naturally associative" because CA is designed to process chaotic inputs. The choice between them is ultimately an aesthetic and workflow decision: CA for natural caves, random walks for connected tunnels, agent diggers for organic dungeons.

---

## Space-first pipeline patterns

A typical space-first dungeon generator sequences these techniques in a predictable order: **room generation** (BSP partitioning, random placement, or template selection) → **connectivity planning** (MST/Delaunay triangulation over room centers, or tree-based sibling connection for BSP) → **corridor carving** (L-shaped, A*-based, or agent-dug corridors along planned edges) → **organic modification** (optional CA erosion, random walk extension, or agent-based detailing) → **template injection** (replacing selected rooms with hand-designed prefabs for key encounters) → **connectivity verification** (flood fill or Union-Find to confirm reachability, with corridor drilling to repair any gaps). Wolverson's builder-chain system makes this pipeline explicit, producing over 2,000 possible combinations by composing room builders, corridor strategies, room sorters, spawn systems, and post-processors. Structure-first techniques (mission graphs, lock-and-key systems, grammar-based generation — covered in Pass 3b) can wrap this entire pipeline: the structure graph defines the required room types, connectivity constraints, and progression order, then hands off to the space-first pipeline to realize each node as physical geometry. The space-first pipeline does not know about keys or missions; it only ensures that the rooms exist, are connected, and are reachable.

## Source landscape

**Tutorials with complete implementations.** Herbert Wolverson's *Hands-On Rust* roguelike tutorial covers BSP, cellular automata, drunkard's walk, DLA, Voronoi, prefabs, and builder chaining in Rust with full runnable code — uniquely valuable for its builder-chain composition system that demonstrates how techniques combine in practice. Bob Nystrom's "Rooms and Mazes" article provides the clearest description of region-based connectivity merging with spanning trees and dead-end removal — the best single source for understanding how maze-fill and room-placement integrate. The eskerda.com BSP tutorial provides a minimal, readable JavaScript implementation with interactive visualization that isolates the core algorithm from game-engine dependencies.

**Reference articles and wikis.** The RogueBasin BSP article by JICE remains the canonical introduction to BSP dungeon generation, concise and widely cited. The RogueBasin Dungeon-Building Algorithm by Mike Anderson is the foundational description of incremental-growth dungeon building, with the clearest articulation of the "always grow from existing dungeon" principle. The RogueBasin cellular automata article by Jim Babcock provides the enhanced two-phase rule (R1+R2) that is not replicated elsewhere with equivalent rigor. Jeremy Kun's mathematical treatment of CA cave generation is uniquely valuable for its resolution-scaling technique and formal Life-like notation.

**GDC and conference talks.** Brian Bucklew's Roguelike Celebration 2019 talk on Wave Function Collapse in Caves of Qud is the most detailed public description of a multi-pass generation pipeline combining WFC with constructive methods in a commercial game — though its primary focus is WFC rather than the space-first techniques covered here. Kyzrati's blog posts on Cogmind's tunneling algorithm are the best source for production-quality agent-based dungeon generation with self-modifying tunneler parameters — no other public source documents this level of agent sophistication.

**Academic work.** The Liapis/Shaker/Togelius PCG Book chapter on constructive dungeon generation provides the only formal pseudocode for both blind and look-ahead agent-based diggers, plus the clearest taxonomy distinguishing macro (BSP) from micro (agent) approaches. The companion code at github.com/sentientdesigns/constructive provides reference Processing implementations. Green, Khalifa, et al.'s "Two-Step Constructive Approaches" paper (FDG 2019) is uniquely valuable for its formal separation of layout generation from furnishing and its use of procedural personas to evaluate output quality.

**Implementation repositories.** Vazgriz's blog post on procedural dungeon generation provides the most complete single description of the Delaunay→MST→corridor pipeline, including a 3D extension. The Game Developer (Gamasutra) article on the TinyKeep algorithm by ADNZZZZZ adds practical tuning guidance (8–10% extra edges) that the original TinyKeep developer's description omits.
