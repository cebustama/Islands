# Randomness primitives and noise for procedural generation

**Every procedural system rests on a small set of randomness primitives—hash functions, selection algorithms, spatial distributions, and coherent noise generators—that determine whether the output looks organic or obviously machine-made.** This reference covers eight foundational techniques in the order a developer typically encounters them: from raw random number generation through spatial sampling to the gradient and cellular noise functions that drive terrain, textures, and world-building. Each section is written at implementation depth: inputs, outputs, key algorithmic steps, known failure modes, and how the technique's output feeds into the next stage of a generation pipeline. Where a canonical primary source exists, the description is verified against it, and discrepancies between sources are noted.

---

## 1. Seeded PRNG and coordinate hashing

The fundamental problem with traditional pseudorandom number generators in games is that they are *sequential*. A standard PRNG like Mersenne Twister maintains internal state and produces its next output by mutating that state, which means generating the 75th value requires computing all 74 before it. Squirrel Eiserloh's GDC 2017 talk, "Math for Game Programmers: Noise-Based RNG," argues that this design is wrong for procedural generation, where you need deterministic random access: the mountain village at coordinates (340, 712) must produce the same layout regardless of when or in what order you visit it. Sequential PRNGs also create problems for parallelism (shared mutable state requires locking), network synchronization (dropped packets desync the sequence), and save/load reproducibility (the entire generation order must be identical). Mersenne Twister compounds these issues with 2.5 KB of internal state and a warm-up period during which output quality is poor (Eiserloh, GDC 2017).

The solution is to treat randomness as a *stateless function* rather than a stateful sequence. A coordinate hash takes an integer position and a seed as inputs and returns a pseudorandom 32-bit unsigned integer—no internal state, no ordering dependency. The function behaves "as if it is an endless, stateless table of possibilities from which you select the result you want based on which position in the table you request" (Eiserloh, GDC 2017, as summarized by Anna Dunster). This means the same (position, seed) pair always returns the same value, different seeds produce entirely different "tables," and evaluations are trivially parallelizable because there is no shared state.

Eiserloh's recommended implementation is **SquirrelNoise5**, an improved version of the Squirrel3 hash he presented at GDC 2017. The algorithm operates on a single unsigned 32-bit integer through a sequence of multiplications by large odd constants, additions of the seed, and XOR-right-shifts of increasing width. The exact procedure from `eiserloh.net/noise/SquirrelNoise5.hpp` is:

```
SquirrelNoise5(positionX, seed):
    mangledBits = (unsigned) positionX
    mangledBits *= 0xd2a80a3f
    mangledBits += seed
    mangledBits ^= (mangledBits >> 9)
    mangledBits += 0xa884f197
    mangledBits ^= (mangledBits >> 11)
    mangledBits *= 0x6C736F4B
    mangledBits ^= (mangledBits >> 13)
    mangledBits += 0xB79F3ABB
    mangledBits ^= (mangledBits >> 15)
    mangledBits *= 0x1b56c4f5
    mangledBits ^= (mangledBits >> 17)
    return mangledBits
```

The pattern alternates multiplications and additions with XOR-right-shifts that increase from 9 to 17 in steps of 2. This ensures every input bit influences every output bit; the worst-case bit influence (input bit 30 → output bit 0) measures **49.99%**, against an ideal of 50.00% (Eiserloh, SquirrelNoise5.hpp commentary). The earlier Squirrel3 used only three constants and three shift amounts, which Peter Schmidt-Nielsen discovered allowed certain high input bits to avoid influencing low output bits, causing noise repetition at extreme position values (Eiserloh, Twitter/X, 2021). SquirrelNoise5 eliminates this flaw.

For multi-dimensional hashing, Eiserloh's approach collapses N coordinates to one integer using large primes before calling SquirrelNoise5. The 2D variant computes `SquirrelNoise5(x + 198491317 * y, seed)`, the 3D variant adds `6542989 * z`, and the 4D adds `357239 * t` (Eiserloh, SquirrelNoise5.hpp). The primes are chosen to be large, coprime, and have "non-boring" bit patterns. To convert the uint32 output to a float in [0, 1], divide by `0xFFFFFFFF`; for [−1, 1], cast to a signed int and divide by `0x7FFFFFFF` (Eiserloh, SquirrelNoise5.hpp). A coordinate hash trivially becomes a sequential PRNG by maintaining a counter: call `SquirrelNoise5(counter++, seed)` to produce a stream with all the quality of the hash plus deterministic replay from any position.

Other hash functions suitable for game use include Thomas Wang's integer hash (three rounds of XOR-shift and multiply), xxHash (fast with excellent distribution, but more complex), and PCG (a linear congruential generator with an output permutation that achieves excellent statistical quality) (unconfirmed: specific comparative benchmarks between these and SquirrelNoise5 were not found in the consulted sources). Eiserloh warns that "many bit-noise sample code functions commonly found on the web are hugely biased or terribly patterned, e.g. having bits which are on (or off) 75% or even 100% of the time" (SquirrelNoise5.hpp commentary). The primary failure modes are: **poor avalanche** (small input changes fail to flip ~50% of output bits, producing visible correlation between adjacent coordinates), **visible patterns** in 2D scatter plots of output, and **platform-dependent behavior** when using signed overflow or type-punning tricks that differ across compilers. The fix for all three is to use a well-tested hash function (SquirrelNoise5, xxHash, or Wang hash) and validate its output with basic statistical tests—scatter plot visualization and chi-squared frequency analysis are sufficient for game use.

The output of a coordinate hash is a single uint32 (or float derived from it). This value feeds directly into every other technique in this document: it selects weighted loot table entries, initializes shuffle bags, provides the per-cell random offsets in Poisson disk sampling, assigns gradient vectors in Perlin noise via permutation table lookups, and jitters feature point positions in Worley noise. The coordinate hash is, in Eiserloh's framing, the foundational primitive that replaces `rand()` throughout a procedural pipeline.

## 2. Weighted random selection and shuffle bags

Uniform random selection—picking each option with equal probability—is rarely what a game actually needs. Loot drops, enemy spawns, dialogue choices, and critical hit rolls all require *weighted* distributions, where some outcomes are more likely than others. And even when a distribution is correct on average, pure random sampling produces streaks that feel unfair: a 10% critical hit chance can easily produce five crits in a row or thirty consecutive misses. These are two distinct problems—weighted selection and distribution control—that require two distinct mechanisms.

The simplest weighted selection builds a cumulative distribution function (CDF) array from the weights. Given weights `w[0..n−1]`, compute `cdf[i] = w[0] + w[1] + ... + w[i]`. To sample, generate a uniform random value in `[0, cdf[n−1])` and find the first index where `cdf[i]` exceeds it. A linear scan takes O(n) per sample; binary search on the CDF reduces this to O(log n). For small tables (fewer than ~20 entries), the linear scan is often faster due to cache locality, but for larger tables the asymptotic cost dominates.

**Walker's alias method** achieves O(1) sampling after O(n) preprocessing (Vose, 1991; definitive tutorial by Keith Schwarz at keithschwarz.com/darts-dice-coins/). The core insight is to redistribute probability mass across n equal-width columns so each column contains at most two outcomes. Two arrays are constructed: `Prob[i]` stores the probability threshold within column i, and `Alias[i]` stores the identity of the second outcome sharing that column. Sampling requires one uniform random integer in `[0, n−1]` to pick a column and one uniform random float in `[0, 1)` to choose between the column's two outcomes—two random numbers, two array lookups, one comparison, no loops. Vose's construction algorithm partitions the n entries (after scaling each probability by n so the average is 1.0) into two worklists: "Small" (probability < 1.0) and "Large" (probability ≥ 1.0). It repeatedly pairs the least-underfull entry with the most-overfull entry: the underfull entry keeps its own probability as the column threshold, and the overfull entry donates its excess to fill the column, then moves to whichever worklist its new probability qualifies for. A final cleanup pass sets any remaining entries' thresholds to 1.0 to absorb floating-point rounding errors (Wikipedia, "Alias method"; Schwarz).

Two failure modes are critical. First, **modulo bias**: converting a uniform 32-bit random integer to the range `[0, n−1]` using `r % n` over-represents values `0` through `(2³² mod n) − 1` by exactly one extra mapping each. For n = 9 and a 32-bit source, values 0–3 are each ~0.000000023% more likely than values 4–8, which is negligible—but for small RAND_MAX (C's `rand()` often yields only 15 or 31 bits), the bias becomes visible. The fix is rejection sampling (discard values above the largest multiple of n that fits in the range) or Daniel Lemire's nearly-divisionless method, which uses a 64-bit multiplication to avoid division in the common case (lemire, pcg-random.org/posts/bounded-rands.html). Second, **floating-point accumulation error** in alias table construction can misclassify entries between the Small and Large worklists; Vose's cleanup step (setting remaining entries to probability 1.0) is essential to handle this gracefully.

A **shuffle bag** (also called a grab bag) solves a different problem: controlling the *period* of a distribution rather than just its mean. The Tetris Guideline Random Generator is the canonical example—it fills a bag with all seven tetromino types, shuffles them via Fisher-Yates, deals them out one at a time, and refills when empty. This guarantees every piece appears exactly once per period of seven, limiting the maximum drought for any piece to 12 (the last piece of one bag, the entire next bag of 7 without it, the first few of the next) and the maximum streak of frustrating S/Z pieces to 4 (Tetris Wiki, "Random Generator"). For game mechanics like critical hits, a shuffle bag with 1 crit token and 9 non-crit tokens ensures exactly 10% crit rate over every 10-attack window while eliminating both long droughts and lucky streaks.

The shuffle bag depends on a correct Fisher-Yates shuffle. The modern (Durstenfeld/Knuth) algorithm iterates backward from index n−1 to 1, swapping each element with a randomly chosen element from index 0 through i inclusive. The forward variant iterates from 0 to n−2, swapping each element with a random element from i through n−1 inclusive. Both produce every permutation with equal probability in exactly n−1 swaps. Three common implementation errors corrupt this uniformity. The **naïve shuffle** (picking `j` from `[0, n−1]` at every step instead of the shrinking range) produces n^n execution paths rather than n!, skewing the distribution—for a 3-element array, permutations {2,3,1} and {2,1,3} and {1,3,2} are over-represented by 50% relative to the others (Jeff Atwood, "The Danger of Naïveté," codinghorror.com). The **off-by-one error** (picking `j` from `[0, i)` instead of `[0, i]`) accidentally implements Sattolo's algorithm, which produces only cyclic permutations—(n−1)! outcomes instead of n!, systematically excluding every permutation that leaves any element in place (Wikipedia, "Fisher-Yates shuffle"). And using `rand() % (i+1)` introduces modulo bias at every step, which cascades through the shuffle to produce measurably skewed final distributions.

The output of weighted selection is an index into a table of outcomes (an integer); the output of a shuffle bag is a sequence of items drawn in a controlled-random order. Both consume the uint32 output of a coordinate hash (converted to a bounded integer or float) and feed into game logic—loot generation, enemy selection, event sequencing—rather than into spatial or noise algorithms. The alias table is a static structure (two parallel arrays of length n) that must be rebuilt if weights change; the shuffle bag is a mutable array with a cursor.

## 3. Poisson disk sampling

Scattering objects uniformly at random across a game world produces two immediately visible defects: clumps where points happen to land close together, and gaps where no points land at all. A forest placed by uniform random sampling has clusters of overlapping trees and bare clearings that no real forest exhibits. The problem is not the *average* density but the absence of any minimum spacing guarantee. Poisson disk sampling generates point sets where every pair of points is separated by at least a minimum distance r, while remaining stochastic—irregular enough to avoid the mechanical look of a grid. The result is a **blue noise** spatial distribution: visually uniform without visible regularity.

Robert Bridson's 2007 SIGGRAPH sketch, "Fast Poisson Disk Sampling in Arbitrary Dimensions," provides the standard O(N) algorithm that replaced earlier impractical approaches. The inputs are the sample domain extent, the minimum distance r, and a constant k (the number of candidate attempts before rejection; Bridson recommends k = 30). The algorithm proceeds in three stages (Bridson, 2007, cs.ubc.ca/~rbridson/docs/bridson-siggraph07-poissondisk.pdf):

**Step 0** initializes a background grid with cell size `r / √n` where n is the dimensionality (in 2D, cell size ≈ **0.707r**). This cell size ensures the cell diagonal equals exactly r, so each cell contains at most one sample. The grid is a flat array of integers, with −1 indicating an empty cell and a non-negative integer giving the index of the sample in that cell.

**Step 1** places an initial sample at a uniformly random position within the domain, inserts it into the grid, and adds it to an *active list* (an array of sample indices from which candidates will be spawned).

**Step 2** is the main loop. While the active list is non-empty: pick a random entry i from it. Generate up to k candidate points uniformly distributed in the **spherical annulus** between radius r and 2r around sample i. For each candidate, use the background grid to check whether any existing sample lies within distance r (in 2D, this requires checking a 5×5 neighborhood of grid cells, at most 25 lookups). If the candidate passes, emit it as a new sample and add it to the active list. If all k candidates fail, remove i from the active list. The algorithm terminates when the active list is empty.

The total number of iterations is exactly 2N − 1 for N output samples: each iteration either adds a point (growing the active list by one) or removes one. Since each iteration does at most k distance checks and k is constant, the overall complexity is **O(N)**—linear in the number of output points (Bridson, 2007). This is the key advantage over **naïve dart-throwing** (Dippé and Wold, 1985; Cook, 1986), which generates candidates uniformly over the entire domain and rejects those too close to existing samples. As the domain fills, the acceptance rate drops toward zero, making the algorithm "impractically inefficient" for dense samplings (Bridson, 2007). Bridson's insight is to generate candidates only where they have a chance of being valid—in the annulus around existing points.

Several variants extend the base algorithm. **Variable-radius Poisson disk sampling** modulates the minimum distance r across the domain using a density map (e.g., a grayscale image). The grid cell size must be set to the *maximum* r value, cells must store lists of points rather than single indices, and neighbor checks must use the local r at each point's position (Dev.Mag, devmag.org.za/2009/05/03/poisson-disk-sampling/). **Tileable Poisson disk sampling** uses toroidal (wraparound) distance when checking neighbors, ensuring points near one edge respect the minimum distance to points near the opposite edge. This is essential for any application where the point set will be tiled, and Cem Yuksel notes that even non-tiling applications benefit from toroidal wrapping because sample elimination produces spuriously dense boundaries without it (Yuksel, cemyuksel.com/cyCodeBase/soln/poisson_disk_sampling.html). **Weighted sample elimination** (Yuksel, 2015) takes an entirely different approach: start with a large white-noise point set and iteratively remove the point with the smallest weighted distance to its neighbors until the desired density is reached.

The most visually striking failure mode is the **Nebraska Problem**, a term coined by Casey Muratori during development of *The Witness*. When Poisson disk points are viewed in a 3D scene (e.g., as grass blade positions), certain camera angles reveal long, narrow gaps between aligned "rows" of points—exactly like seeing between rows of corn while driving through Nebraska. The cause is intrinsic to blue noise: because points maintain minimum spacing, they inevitably form locally quasi-regular arrangements, and some viewing angles align with these rows. Tighter packing exacerbates the problem because the quasi-regularity becomes more pronounced (Muratori, caseymuratori.com/blog_0011). Muratori found that no blue noise variant (denser packing, second-pass gap-filling, line-of-sight–based generation) fully eliminates the artifact; his ultimate solution for grass in *The Witness* was a non-blue-noise radial placement scheme. Other failure modes include: **wrong grid cell size** (if larger than `r/√n`, multiple samples can land in one cell, but the grid stores only one, causing missed neighbor checks and minimum-distance violations); **k too low** (premature removal from the active list produces visible large gaps—k = 5 is borderline acceptable, k = 30 is the standard, k > 30 shows diminishing returns); and **boundary density bias** (without toroidal wrapping or ghost points, points near domain edges have fewer neighbors, producing denser packing at boundaries).

The output of Poisson disk sampling is an array of 2D or 3D point positions, typically stored as a flat list of floats. Downstream systems consume these as spawn locations for trees, rocks, NPCs, or any placed object. The background grid built during generation can be retained as a spatial index for fast nearest-neighbor queries. When combined with a coordinate hash, the point positions can drive further per-object variation: hash each point's position to determine tree species, scale, rotation, or color.

## 4. Stratified sampling and blue noise

Poisson disk sampling is one way to produce a blue noise distribution, but it is not the only one, and some applications need blue noise in a different form—not as a point set but as a texture, a per-pixel value, or a cheap approximation that avoids the complexity of Bridson's algorithm. Stratified sampling and pre-computed blue noise textures fill these roles.

**Stratified sampling** (jittered grid) is the simplest spatial anti-clumping technique. Divide the domain into a regular grid of equal cells. Place exactly one random point per cell, offset from the cell center by a random jitter within the cell bounds. The result is guaranteed to have at most one point per cell (preventing tight clumps) and at least one point per cell (preventing large gaps). The algorithm is trivial—two nested loops and two random numbers per cell—and runs in O(N) time with minimal code. However, Red Blob Games' analysis (redblobgames.com/x/1830-jittered-grid/) demonstrates a fundamental tension: **jitter ≥ 0.9 of the cell width** is needed for the point distribution to appear angularly isotropic (irregular, not grid-like), but **jitter ≤ 0.6** is needed to avoid extreme distance outliers (points from adjacent cells landing almost on top of each other). These constraints are contradictory—no single jitter value satisfies both. The practical consequence is that jittered grids always look either slightly griddy or slightly clumpy, which is why Poisson disk sampling exists as a strictly better (but more complex) alternative.

**Blue noise** is a property of a signal's power spectrum: it has high-frequency energy but no low-frequency energy. In spatial terms, a blue noise point distribution is irregular (high frequency) but uniformly covering (no low-frequency clumping or gapping). Poisson disk distributions exhibit blue noise characteristics, confirmed by periodogram analysis (Bridson, 2007, Figure 1). The human retina distributes its cone photoreceptors in a blue noise pattern, which researchers hypothesize accounts for the visual system's resistance to regular-grid aliasing artifacts (Muratori, caseymuratori.com/blog_0010, citing Don Mitchell's work). **White noise** contains all frequencies equally and produces visible clumps and gaps. **Red noise** (Brownian) contains only low frequencies—neighboring values are correlated—and looks like a smooth random walk.

**Mitchell's best-candidate algorithm** (Mitchell, SIGGRAPH 1991) generates progressive blue noise point sets. For each new sample, generate M random candidates (where M scales with the current sample count: `M = i × c + 1` for some constant c). Find each candidate's minimum distance to all existing samples. Place the candidate with the largest minimum distance. This produces a point set where any prefix is also approximately blue noise (a property called *progressiveness* that Poisson disk sampling lacks). The cost is O(n² × M)—far slower than Bridson's O(N)—which limits it to offline pre-computation (demofox.org, "Generating Blue Noise Sample Points with Mitchell's Best Candidate Algorithm").

**Void-and-cluster** (Ulichney, 1993) generates blue noise *textures* rather than point sets. The output is a 2D array where each pixel has a unique rank value (0 to width×height−1), and thresholding this array at any value produces a blue noise point set at that density. The algorithm initializes a sparse blue noise binary pattern (~10% of pixels set), then iteratively removes the point at the tightest cluster (highest Gaussian-blurred energy among set pixels) and adds points at the largest void (lowest energy among empty pixels), assigning rank values as it goes. The result is inherently tileable (using toroidal blur) and has a flat histogram (each brightness appears exactly once). Generation is expensive—~30–60 seconds for 128×128—so textures are pre-computed and shipped as assets (Christoph Peters, momentsingraphics.de/BlueNoise.html, who provides free CC0 blue noise textures used in Unity's post-processing stack).

Pre-computed blue noise textures are used in shaders by tiling a small texture (typically 64×64) across the screen and reading its value per pixel as a random offset for stochastic effects (dithering, ambient occlusion sampling, stochastic transparency). A powerful animation trick adds a golden-ratio-based offset per frame: `value = fract(blueNoise + 0.61803398875 × (frame % 64))`, which makes the noise low-discrepancy over time while remaining blue over space (demofox.org).

The failure modes of stratified sampling are the jitter tension described above and visible grid alignment at low jitter values. Blue noise textures have no runtime failure modes (they are pre-computed and validated), but generating them incorrectly—using a non-toroidal Gaussian blur, or initializing with a non-blue seed pattern—produces textures whose thresholded subsets are not blue noise. The output of stratified sampling is a point set (same format as Poisson disk); the output of a blue noise texture is a 2D scalar array. Both feed into the same downstream systems as Poisson disk samples—object placement, per-pixel randomization, dithering—with the texture form being preferred when per-pixel values are needed rather than discrete point positions.

## 5. Value noise

The techniques above generate discrete random values or point positions. Value noise is the simplest way to produce a *continuous* random scalar field—a smooth function that varies randomly across space. This is the bridge between discrete randomness and the smooth, organic-looking variation needed for terrain heightfields, texture patterns, and material blending.

Value noise works by hashing integer lattice points to obtain random scalar values, then interpolating between them. For a 2D input point (x, y), compute the integer cell `(i, j) = (floor(x), floor(y))` and the fractional offset `(f, g) = (fract(x), fract(y))`. Hash each of the four corners—(i, j), (i+1, j), (i, j+1), (i+1, j+1)—to get four pseudorandom scalars a, b, c, d (using any suitable hash function, such as SquirrelNoise5 with the coordinate collapsed to one dimension). Then bilinearly interpolate: `lerp(lerp(a, b, u), lerp(c, d, u), v)`, where u and v are the smoothed fractional coordinates. The interpolation curve is critical: **linear interpolation** (`u = f`) produces visible discontinuities in the first derivative at every lattice line—the noise looks faceted, with sharp creases at cell boundaries. **Cubic Hermite / smoothstep** (`u = 3f² − 2f³`) achieves C1 continuity (first derivative zero at endpoints), producing much smoother results. **Quintic / smootherstep** (`u = 6f⁵ − 15f⁴ + 10f³`) achieves C2 continuity (first and second derivatives zero at endpoints), which matters when the noise is used for bump mapping or any application that consumes noise derivatives (The Book of Shaders, chapter 11; Inigo Quilez, iquilezles.org/articles/morenoise/).

Value noise has the **most prominent grid-aligned artifacts** of any coherent noise type—it "resembl[es] a blurred pixel grid with clearly recognizable cells" (NoisePosti.ng). The cause is structural: each lattice point holds a random value that is a local extremum of the interpolation, so the maxima and minima of the noise field are located at or very near the lattice points themselves. Features align to the horizontal, vertical, and diagonal axes of the grid by construction. This is why Ken Perlin developed gradient noise: by assigning random *directions* instead of random *values* to lattice points, the extrema move away from lattice positions and the directional bias is reduced (though not eliminated). Value noise does have one advantage—it is conceptually and computationally simpler than gradient noise—and remains useful when the application will layer many octaves or apply heavy post-processing that obscures the grid artifacts.

The input is an N-dimensional float coordinate; the output is a single scalar, typically in [0, 1] (though remapping to [−1, 1] is common). The output feeds directly into downstream uses: terrain heightfield generation (as input to fractal layering, which is outside this document's scope), material blending weights, color variation, or as a density function. The output data structure is simply a float per query point—there is no persistent spatial structure to maintain between evaluations.

## 6. Perlin noise (classic and improved)

Value noise suffers from pronounced grid artifacts because its extrema sit on lattice points. In 1985, Ken Perlin introduced **gradient noise** to solve this problem, and the technique became so widely used that "Perlin noise" is often (incorrectly) used as a generic term for any coherent noise. The core insight is to assign a random *gradient vector* rather than a random scalar to each lattice point, then compute a dot product between that gradient and the displacement vector from the lattice point to the query point. Because the dot product is zero when the displacement is zero, gradient noise always passes through zero at lattice points—the extrema are forced *between* lattice points, producing more natural-looking variation (Perlin, "An Image Synthesizer," SIGGRAPH 1985).

The algorithm for 3D improved Perlin noise (Perlin, "Improving Noise," 2002; reference implementation at cs.nyu.edu/~perlin/noise/) proceeds as follows. For input point (x, y, z):

1. Compute the integer cell: `X = floor(x) & 255`, `Y = floor(y) & 255`, `Z = floor(z) & 255`. The bitmask with 255 wraps coordinates into the permutation table's range.
2. Compute the fractional offset within the cell: `x -= floor(x)`, `y -= floor(y)`, `z -= floor(z)`.
3. Compute the **fade curves**: `u = fade(x)`, `v = fade(y)`, `w = fade(z)`.
4. Hash the eight cube corners through the **permutation table**: `A = p[X]+Y`, `AA = p[A]+Z`, `AB = p[A+1]+Z`, `B = p[X+1]+Y`, `BA = p[B]+Z`, `BB = p[B+1]+Z`.
5. For each of the eight corners, compute `grad(hash, dx, dy, dz)`: the dot product of the corner's gradient vector with the displacement from that corner to the query point.
6. Trilinearly interpolate the eight dot products using u, v, w: four lerps along x, two along y, one along z (seven scalar lerps total).

The **permutation table** is a random shuffle of integers 0–255, doubled to 512 entries to avoid index-overflow checks: `p[i] = p[i+256] = permutation[i]`. Perlin published a specific canonical permutation starting with `{151, 160, 137, 91, 90, 15, ...}` (cs.nyu.edu/~perlin/noise/). The doubling means that adding Y or Z coordinates to a table lookup never exceeds the array bounds.

The 1985 original and the 2002 improved version differ in two specific ways. **First, the interpolation curve.** The 1985 version used the cubic Hermite function `s(t) = 3t² − 2t³`, which has its first derivative zero at t = 0 and t = 1 but its second derivative equal to 6 and −6 at those points, respectively—a **C1-only** continuity. The 2002 version replaced this with the quintic `fade(t) = 6t⁵ − 15t⁴ + 10t³` (in Horner form: `t * t * t * (t * (t * 6 - 15) + 10)`), which has both first *and* second derivatives zero at t = 0 and t = 1, achieving **C2 continuity** (Perlin, 2002). The practical consequence is that the 1985 version produces a visible second-derivative discontinuity at every lattice boundary—"the normals in these regions change suddenly"—which creates grid-aligned seam lines in bump-mapped or displaced surfaces (Perlin, GPU Gems Chapter 5, developer.nvidia.com). The quintic interpolant eliminates this artifact entirely.

**Second, the gradient set.** The 1985 version selected from 256 pseudo-random gradient vectors distributed on a sphere. Their irregular distribution caused some gradients to cluster, producing visible "splotchy" regions where adjacent aligned gradients created unusually high or low noise values. The 2002 version replaced these with exactly **12 fixed gradient vectors**: the midpoints of the edges of a cube, specifically `(±1,±1,0)`, `(±1,0,±1)`, and `(0,±1,±1)` (Perlin, 2002). These 12 directions are evenly distributed, eliminating statistical clumping. Perlin noted that human pre-attentive vision is sensitive to statistical anomalies in orientation but insensitive to granularity of orientations (Perlin, GPU Gems Chapter 5, citing Grill-Spector et al., 1995). Because all gradient components are 0 or ±1, the dot product reduces to additions and subtractions—no multiplications needed. To avoid the expensive `% 12` operation, the 12 gradients are extended to 16 by duplicating four of them, allowing selection via `hash & 15` (a fast bitwise AND). Perlin's reference implementation encodes this in a compact `grad()` function using bit tricks: the low 4 bits of the hash select among 16 cases, with bits controlling which two coordinate axes to use and their signs (cs.nyu.edu/~perlin/noise/).

Scratchapixel raises a counterpoint: the fixed 12-direction gradients in the improved version can "create vertical and horizontal lines you don't see in the original version" because the gradients are aligned along the xy, xz, and yz planes (scratchapixel.com, "Improved Perlin Noise"). This means the 2002 improvement trades one form of artifact (splotchiness from random gradient clustering) for another (subtle axis-alignment from fixed gradient directions). In practice the 2002 artifacts are far less objectionable, but the tradeoff exists.

Beyond the interpolant and gradient changes, several other failure modes affect Perlin noise implementations. The **permutation table wraparound** causes the noise to repeat every 256 integer units in each dimension—coordinates are masked with `& 255`, so `noise(0.5)` equals `noise(256.5)`. This is rarely a problem at typical scales (the noise is sampled at fractional coordinates, so 256 units is large), but for very large worlds the periodicity becomes visible. Solutions include larger tables, alternative hash functions like SmallXXHash (Catlike Coding, catlikecoding.com), or explicit period control. **Zero at lattice points** is an inherent property of gradient noise—the dot product of any gradient with a zero-length displacement is zero. This can create a visible "grid crosshatch" in some applications; a hybrid GRADVAL approach (summing gradient and value noise) fills in the zeros (Accidental Noise Library documentation). And **using linear interpolation** instead of the cubic or quintic fade produces faceted noise with visible creases at every lattice boundary, unsuitable for any application that consumes noise derivatives.

The output of Perlin noise is a scalar float, theoretically in [−1, 1] but practically bounded by approximately ±0.707 in 2D and ±0.866 in 3D (depending on gradient alignment). The output feeds into terrain heightfield generation, texture synthesis, material blending, and as the input to fractal layering (fBm, ridged multifractal, etc.—which are outside this document's scope). Inigo Quilez provides a GLSL implementation that returns the noise value *and* its analytical gradient as a `vec3`, enabling efficient normal computation for lighting without finite-difference approximation (iquilezles.org/articles/gradientnoise/).

## 7. Simplex noise and OpenSimplex noise

Perlin noise evaluates 2^N lattice corners per point (4 in 2D, 8 in 3D, 16 in 4D), and its hypercubic grid imposes axis-aligned directional bias. In 2001, Perlin introduced **simplex noise** to address both problems by replacing the hypercubic lattice with a simplicial lattice—triangles in 2D, tetrahedra in 3D—where each simplex has only N+1 vertices. This reduces the per-point evaluation from O(2^N) to O(N²) and, because simplices are more isotropic than hypercubes, reduces directional artifacts. Stefan Gustavson's 2005 paper "Simplex noise demystified" (cgvr.cs.uni-bremen.de/teaching/cg_literatur/simplexnoise.pdf) provides the definitive tutorial-depth explanation of the algorithm.

The simplicial grid is obtained by **skewing** a regular hypercubic grid along its main diagonal. In 2D, the skew transform adds `s = (x + y) × F₂` to each coordinate, where **F₂ = (√3 − 1) / 2 ≈ 0.3660** (Gustavson, 2005). This maps equilateral triangles to right isosceles triangles that tile as squares, allowing cell identification by simple `floor()` operations. The inverse **unskew** uses **G₂ = (3 − √3) / 6 ≈ 0.2113**. In 3D, F₃ = 1/3 and G₃ = 1/6; in 4D, F₄ = (√5 − 1)/4 and G₄ = (5 − √5)/20. The general formula is `F_N = (√(N+1) − 1) / N` (Gustavson, 2005). Catlike Coding derives these constants geometrically: in 2D, setting the equilateral triangle edge length such that the skewed form tiles as unit squares yields the quadratic `6v² − 6v + 1 = 0`, whose smaller root is G₂ (catlikecoding.com/unity/tutorials/pseudorandom-noise/simplex-noise/).

After skewing the input coordinates to identify the containing cell, the algorithm determines which of the N! simplices within that cell contains the query point. In 2D, this requires a single comparison: if `x₀ > y₀` (where x₀, y₀ are the displacement from the unskewed cell origin), the point is in the lower triangle, traversing corners (0,0)→(1,0)→(1,1); otherwise it is in the upper triangle, traversing (0,0)→(0,1)→(1,1). In 3D, six pairwise comparisons of the displacement coordinates determine which of six tetrahedra the point occupies. In 4D, a lookup table maps the six pairwise comparison results to one of 24 valid simplex traversal orderings (Gustavson, 2005).

Unlike Perlin noise, simplex noise does **not interpolate** between corner contributions. Instead, it **sums** radially attenuated kernel contributions from each of the N+1 simplex corners. For each corner with gradient g and displacement vector d, the contribution is `(R² − |d|²)⁴ × dot(g, d)`, where R² is the kernel radius. The fourth-power falloff ensures the kernel reaches zero before crossing into adjacent simplices, so only the N+1 corners of the containing simplex contribute. The kernel radius R² presents a subtle implementation choice: the mathematically correct value for eliminating discontinuities is **R² = 0.5** (where the falloff exactly reaches zero at simplex boundaries), but Perlin's reference implementation and Gustavson's code use **R² = 0.6** for 3D and 4D, which extends the kernel slightly beyond simplex boundaries and produces small discontinuities. KdotJPG (Kurt Spencer, author of OpenSimplex) notes these jumps are "undetectable in most ordinary use cases" (NoisePosti.ng), but they can appear in derivative-based applications. In 2D, the final result is scaled by a constant (Gustavson uses `return 70.0 × (n₀ + n₁ + n₂)`) to normalize the output range to approximately [−1, 1].

The advantages over Perlin noise are concrete: **3 gradient evaluations in 2D** instead of 4, **4 in 3D** instead of 8; no sequential interpolation chain (all contributions are independent, enabling SIMD parallelism); a well-defined continuous analytical gradient that is simpler to compute than the "interpolated product of high-order polynomials" in classic noise (Gustavson, Shadertoy commentary); and better visual isotropy because the simplicial lattice does not have the strong axis-aligned bias of a hypercubic grid.

Simplex noise was covered by US Patent 6,867,776 (filed January 8, 2002; granted March 15, 2005; assigned to Nokia; expired January 8, 2022), which specifically covered implementations in 3D and higher using the simplicial lattice decomposition (patents.google.com/patent/US6867776B2/en). This patent drove the creation of **OpenSimplex noise** by Kurt Spencer in September 2014, released into the public domain. OpenSimplex differs from simplex noise in a fundamental structural way: where simplex noise squashes a hypercubic honeycomb *down* the main diagonal (producing the standard simplex/orthoscheme decomposition), OpenSimplex stretches a hypercubic honeycomb *out* along the main diagonal, producing a different lattice—the tetrahedral-octahedral (simplectic) honeycomb in 3D instead of the tetragonal disphenoid honeycomb (Wikipedia, "OpenSimplex noise"; Kurt Spencer, gist.github.com/KdotJPG/b1270127455a94ac5d19). OpenSimplex also uses a larger kernel radius, evaluating more contributing vertices per point (4 in 2D vs. simplex's 3; 7+ in 3D vs. simplex's 4), producing smoother output at some performance cost.

**OpenSimplex2** (github.com/KdotJPG/OpenSimplex2) is a major revision that comes in two variants. **OpenSimplex2** (formerly OpenSimplex2F / FastSimplexStyleNoise) "looks the most like Simplex" and is about as fast. Its 3D implementation constructs a rotated body-centered-cubic (BCC) grid as the offset union of two rotated cubic grids, finding the closest 2 points on each. Its 2D implementation is standard simplex noise with an improved **24-direction gradient set** (replacing the original 12) to reduce visible feature repetition in thresholded single-octave noise. **OpenSimplex2S** (formerly SuperSimplexNoise) "looks the most like 2014 OpenSimplex"—smoother, with larger vertex contribution ranges (finding the closest 4 points per sub-grid in 3D instead of 2). OpenSimplex2S is recommended for ridged noise applications because the larger kernel avoids the "dotty" appearance that OpenSimplex2 can exhibit when individual octave layers are passed through `abs()` (KdotJPG, OpenSimplex2 README).

The primary failure mode of simplex-family noise is **gradient set quality**. Older implementations used a 12-direction set (cube edge midpoints) that allowed "some squareness to sneak back in" (NoisePosti.ng). KdotJPG's February 2020 update to OpenSimplex2 replaced this with a 24-direction set in 2D to improve probability symmetry. The original 2014 OpenSimplex had its own issues: it "produced more straight parts and was not as probabilistically symmetric" and broke down entirely in 5D+, "producing diagonal bands of visibly-varying value ranges" (KdotJPG, OpenSimplex2 README). Higher-dimensional simplex noise also exhibits **dimensional cross-section artifacts**: a 2D slice of 3D simplex noise looks visually different from (and generally worse than) native 2D simplex noise, and quality degrades further with each additional dimension (Wikipedia, "Simplex noise"). **Domain rotation**—rotating the sampling coordinates so that the griddiest planes don't align with the observer's viewing plane—can mitigate remaining directional artifacts in all simplex-family noise types (NoisePosti.ng).

The output is a scalar float in approximately [−1, 1], identical in format and downstream usage to Perlin noise. The choice between Perlin, simplex, and OpenSimplex is discussed in the comparison section below.

## 8. Worley noise (Voronoi noise)

All the noise types above produce smooth, continuous scalar fields. Worley noise (also called cellular noise or Voronoi noise) produces something fundamentally different: a scalar field whose value at each point is determined by the *distance to nearby feature points*. The result is a cellular, tessellated pattern—flagstones, cracked mud, biological cells, reptile scales—that cannot be approximated by any combination of gradient noise. Steven Worley introduced this as "a cellular texture basis function" at SIGGRAPH 1996, providing a family of functions F₁, F₂, ..., Fₙ where Fₙ(x) is the distance from x to the nth-closest feature point (Worley, 1996, cedric.cnam.fr/~cubaud/PROCEDURAL/worley.pdf).

The algorithm scatters feature points throughout space and, for each query point, finds the distances to the nearest ones. The acceleration structure divides space into a grid of **unit cubes** at integer coordinates (a point at (1.2, 3.3, 2.3) lies in cube (1, 3, 2) via floor operations). Each cube's feature points are generated deterministically by hashing the cube's integer coordinates to seed a fast LCG, then drawing point counts from a Poisson distribution with mean λ ≈ 3–4, clamped to [1, 9] (Worley, 1996). In practice, most modern implementations simplify to **exactly one feature point per cell**, which greatly reduces code complexity with minimal quality loss. Each point's position is the hash output mapped to [0, 1] relative to the cell's base corner.

To evaluate F₁ (or F₁ through Fₙ simultaneously), the algorithm checks the query point's own cell and all neighboring cells. In 2D, this means a **3×3 neighborhood** (9 cells); in 3D, a **3×3×3 neighborhood** (27 cells). For each feature point encountered, compute the squared distance to the query point (avoiding the square root until final output) and maintain a sorted list of the n smallest distances. Worley's original paper includes early-out optimizations: by comparing the current nth-closest distance against the minimum possible distance to each neighboring cube's face, entire rows of cubes can be eliminated, and "typically only 1–3 cubes actually need testing" (Worley, 1996). GPU implementations usually check the full neighborhood without early termination, as branch divergence is more expensive than redundant distance computations on parallel hardware.

The F₁ through Fₙ values are always non-decreasing: 0 ≤ F₁(x) ≤ F₂(x) ≤ ... ≤ Fₙ(x). Common combinations yield distinct visual characters: **F₁** produces radial gradients around each feature point (the basic Voronoi distance field); **F₂ − F₁** equals zero at Voronoi cell boundaries (where the point is equidistant from two features) and produces vein-like or lattice-like ridges; **F₂** alone has more complex internal structure. Worley generated approximately 40 different linear combinations `C₁F₁ + C₂F₂ + C₃F₃ + C₄F₄` experimentally and found F₁ through F₄ and their combinations produce distinct textures, while F₅+ begin to look similar (Worley, 1996). The routine also outputs a **unique cell ID** (hashed cube coordinates plus feature point index), enabling constant-color-per-cell shading for stained-glass, mosaic, or territory-map effects.

The choice of **distance metric** fundamentally changes the visual character. Euclidean distance (`√(dx² + dy²)`) produces organic, rounded cells. Manhattan/L₁ distance (`|dx| + |dy|`) produces diamond-shaped cells—Worley noted these "make surfaces like random right angle channels; useful for spaceship hulls" (Worley, 1996). Chebyshev/L∞ distance (`max(|dx|, |dy|)`) produces square cells. Squared Euclidean (`dx² + dy²`) is faster to compute (no square root) and produces shapes similar to Euclidean with different value gradients.

Inigo Quilez contributes two important variants. **Smooth Voronoi** replaces the hard `min()` operation (which creates C0-discontinuous boundaries) with a smooth minimum function—either exponential or polynomial—that blends distances from all neighboring feature points weighted by proximity (iquilezles.org/articles/smoothvoronoi/). This produces smooth, organic cell boundaries instead of sharp ridges. **Voronoise** (Quilez, 2014) is a parametric generalization unifying gradient noise and Voronoi noise with two parameters: u controls jitter (0 = regular grid / noise-like, 1 = fully jittered / Voronoi-like) and v controls the metric blend (interpolation vs. minimum operation). At u=0, v=1 it produces gradient noise; at u=1, v=0 it produces Voronoi; intermediate values blend between them (iquilezles.org/articles/voronoise/). The implementation uses a 5×5 kernel (vs. the standard 3×3) to accommodate full jitter range.

For **correct Voronoi edge detection** with uniform-width borders, simple F₂−F₁ thresholding produces lines whose thickness varies with cell spacing. Quilez's two-pass approach (iquilezles.org/articles/voronoilines/) first finds the closest cell point in a 3×3 search, then performs a 5×5 search centered on that cell's coordinates to compute the exact perpendicular distance to each bisecting line between the closest point and each neighbor: `edgeDist = dot(0.5 × (a + b), normalize(b − a))`. The second pass must be centered on the closest *cell* (not the query point's cell) to avoid discontinuities at cell corners where the closest feature point changes.

The failure modes of Worley noise are diagnostic. **Missing neighbor cell checks** produces the most visible artifact: sharp seams at cell boundaries where the true closest point is in an unchecked adjacent cell. The seams form a regular grid pattern that is immediately recognizable. Stefan Gustavson's GPU optimization reduces the search to a 2×2 kernel to save texture lookups, but this creates discontinuity artifacts at tile edges (The Book of Shaders, thebookofshaders.com/12/). **Non-Euclidean distance metrics** can extend further than Euclidean in diagonal directions, potentially requiring a larger search neighborhood (4×4 or 5×5) to guarantee correctness. **Poor hash function quality** makes the underlying grid visible—feature points don't move far enough from cell centers to break regularity. And **floating-point precision loss** far from the origin causes feature points to snap to coarse positions; Quilez's cell-space trick (working only with fractional parts after extracting integer cell coordinates) eliminates this by discarding all integer coordinate information before computing distances (iquilezles.org/articles/smoothvoronoi/).

The output of Worley noise is one or more float distance values (F₁, F₂, etc.) and optionally an integer cell ID per query point. The distance values feed into texture synthesis (mapping distance to color or height), edge detection (thresholding F₂−F₁), and pattern generation. The cell ID feeds into per-cell coloring, territory assignment, or per-cell random variation (hash the cell ID to determine a cell's color, biome, or material). Unlike gradient noise, Worley noise naturally produces *boundaries* between regions, making it the primitive of choice for any pattern that requires distinct cells or territories.

## Perlin, simplex, OpenSimplex, and Worley compared

These four noise types differ in their lattice geometry, evaluation mechanism, and visual character. **Classic/improved Perlin noise** evaluates gradient dot products at the 2^N corners of a hypercubic cell and interpolates them with a smoothstep or smootherstep curve; it scales as O(2^N) and produces smooth, cloudy patterns with visible axis-aligned directional bias—features tend to align at 0° and 45° to the coordinate axes. **Simplex noise** replaces the hypercube with N+1 simplex vertices, sums radially attenuated kernels instead of interpolating, scales as O(N²), and produces more isotropic patterns with higher apparent frequency at the same input scale; it is roughly **2× faster in 3D** and dramatically faster in 4D+ than Perlin. **OpenSimplex2** uses a modified simplicial lattice (BCC grid in 3D) with improved gradient tables, achieving visual quality close to simplex with no patent encumbrance; the "F" variant matches simplex speed and character, while the "S" variant is smoother and slightly slower. **Worley noise** is categorically different—it computes distances to scattered feature points rather than blending gradients—and produces cellular, tessellated patterns (stone, scales, cracks) that no gradient noise can replicate.

The decision of when to use each is straightforward once the visual target is identified. For smooth, cloud-like fields (terrain height, fog density, color variation), **OpenSimplex2** is the default choice: it has no patent issues, negligible axis bias, and competitive performance. Use Perlin noise only when matching legacy content or when a specific implementation's exact output must be reproduced. Use Worley noise when the desired pattern has visible cells, borders, or distance-field features—organic stone, cracked earth, cellular biology, voronoi-based territory maps. Worley and gradient noise serve entirely different visual purposes and are frequently combined (e.g., Worley cell boundaries modulating Perlin terrain), not alternatives to each other.

## Random placement versus Poisson disk sampling

The quality difference between uniform random placement and Poisson disk sampling is immediately visible when placing discrete objects. A uniform random distribution of 500 tree positions across a terrain tile will produce multiple clusters of 3–5 overlapping trees and several conspicuous treeless clearings roughly 2–3× the average spacing—because the expected maximum gap in a uniform random point process grows logarithmically. The same 500 points generated via Bridson's algorithm with an appropriate minimum distance r produce a visually even forest with no overlaps and no bare patches, because the minimum distance guarantee eliminates both clumping and large gaps. The blue noise spectral character of Poisson disk samples mimics natural distributions (retinal cone cells, forest canopies viewed from above), which is why the result looks organic rather than mechanical.

The implementation cost of Poisson disk sampling is modest: Bridson's algorithm is O(N), requires ~50–100 lines of code beyond what uniform random placement needs, and adds a background grid of integers as auxiliary memory. For real-time applications where even this cost matters (e.g., spawning thousands of particles per frame), stratified jittered sampling provides a cheaper approximation, and pre-computed blue noise textures provide per-pixel randomization at zero runtime cost beyond a texture lookup. Use uniform random placement only when the application genuinely does not care about spatial distribution quality—particle effects with very short lifetimes, debug visualizations, or cases where the point count is so low that clumping is statistically unlikely.

## Source landscape

**Primary papers.** Ken Perlin's "An Image Synthesizer" (SIGGRAPH 1985) defines gradient noise and remains the conceptual foundation for all lattice-based noise; his "Improving Noise" (2002) is a short, specific fix document that introduces the quintic interpolant and 12-gradient set. Steven Worley's "A Cellular Texture Basis Function" (SIGGRAPH 1996) defines cellular noise with its Fn distance family, acceleration structure, and dozens of example combinations—no other source covers the Poisson-distributed multi-point-per-cell variant with the same rigor. Robert Bridson's "Fast Poisson Disk Sampling in Arbitrary Dimensions" (SIGGRAPH 2007) is a single-page paper that completely specifies an O(N) algorithm—unmatched in brevity-to-impact ratio. Stefan Gustavson's "Simplex Noise Demystified" (2005) is the only source that explains simplex noise's skewing transforms, simplex traversal, and kernel summation at true implementation depth with worked examples—Perlin's own descriptions are compressed for hardware audiences.

**Tutorial series.** Catlike Coding (Jasper Flick) provides Unity/C# implementations of value, Perlin, simplex, and Voronoi noise with unusually rigorous mathematical derivations of constants (the skew factor derivation via quadratic formula is unique to this series). The Book of Shaders (Patricio Gonzalez Vivo, chapters 10–13) offers interactive GLSL editors where each code change updates a live preview—invaluable for developing intuition about how parameters affect output. Red Blob Games (Amit Patel) covers jittered grid vs. Poisson disk tradeoffs with interactive sliders and frequency-domain analysis that no other tutorial provides. Inigo Quilez's articles at iquilezles.org cover smooth Voronoi, Voronoi edges, Voronoise, and analytical noise derivatives with production-quality GLSL code—the only source for the smooth-minimum Voronoi variant and the parametric noise-Voronoi unification.

**GDC talks.** Squirrel Eiserloh's "Math for Game Programmers: Noise-Based RNG" (GDC 2017, gdcvault.com/play/1024365) is the definitive argument for replacing sequential PRNGs with stateless hash functions in game engines, including the SquirrelNoise5 reference implementation—no other source covers coordinate hashing with this level of game-specific practical advice.

**Interactive tools.** Shadertoy (shadertoy.com) hosts Quilez's reference implementations of Voronoi, Voronoise, and smooth Voronoi as live-editable GLSL shaders—the most immediate way to experiment with parameter changes. The Book of Shaders' built-in editor serves the same role for value, gradient, and cellular noise. FastNoiseLite (Auburn/FastNoiseLite on GitHub) provides a single-header C/C# library covering all major noise types with a visual preview tool—uniquely valuable for rapid comparison across noise types with identical parameters.

**Repositories.** KdotJPG's OpenSimplex2 (github.com/KdotJPG/OpenSimplex2) provides Java, C#, Rust, and GLSL implementations of patent-free simplex-family noise with detailed documentation of the BCC lattice construction and gradient table improvements—the only source for the OpenSimplex2/2S algorithmic distinction. Christoph Peters' free blue noise textures (momentsingraphics.de/BlueNoise.html) are pre-computed, CC0-licensed, tileable void-and-cluster textures used in Unity's post-processing stack—the standard resource for shader-based blue noise dithering.
