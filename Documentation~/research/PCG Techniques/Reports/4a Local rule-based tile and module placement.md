# Local rule-based tile and module placement

**Every technique in this pass answers the same question: given a cell on a grid and the state of its neighbors, which tile or module goes here?** The four systems covered — autotiling, Wang tiles, adjacency-rule placement, and socket-based modular assembly — form a spectrum from fully deterministic lookup to expressive constraint-driven selection. All share the property of locality: they evaluate compatibility at or near the placement site, require no global propagation state, and never backtrack. This makes them fast, predictable, and easy to integrate into both hand-authored and procedurally generated maps. They are also the building blocks from which Wave Function Collapse and Model Synthesis are constructed, a relationship explored in Pass 4b.

---

## 1. Autotiling selects the right sprite so you don't have to

Autotiling solves a deceptively tedious problem: given a grid where each cell holds a logical terrain state (wall or floor, grass or water), automatically pick the visually correct tile so edges, corners, and transitions between terrain types look seamless. Without it, a level designer placing a 47-tile blob tileset must manually choose among dozens of border variants for every single cell — a process that is slow, error-prone, and hostile to iteration. Autotiling decouples the logical map (what terrain is where) from the visual map (which sprite to draw), meaning a procedural generator or a designer's brush can write binary values while the autotiler handles all visual selection downstream. The technique is reactive, not generative: it does not decide *what* to place, only *how it looks* given what's already there (Boris the Brave, "Beyond Basic Autotiling," 2021).

The core algorithm encodes neighbor states into a bitmask and uses that bitmask as an index into a lookup table of tile sprites. In the simplest variant — **4-bit or marching-squares autotiling** — only the four corners (or equivalently four cardinal neighbors) are checked, each assigned a power of two: `index = NW×8 + SW×4 + SE×2 + NE×1`. This produces 2⁴ = 16 possible indices and requires exactly 16 tiles (Boris the Brave, "Tileset Roundup," 2013; Envato Tuts+, "How to Use Tile Bitmasking"). The more capable **8-bit blob variant** checks all eight neighbors (four cardinal, four diagonal), but raw 8-bit encoding yields 256 combinations — far more tiles than any artist wants to draw. The critical optimization is **corner elimination**: a diagonal neighbor is visually relevant only when both adjacent cardinal neighbors are also of the same terrain type. If the cell to the north is empty, a north-edge border is drawn regardless of the northeast corner's state. The rule is applied before computing the bitmask: `if not left or not top: topLeft = False` (and symmetrically for all four corners). This dependency constraint reduces the 256 raw combinations to exactly **47 visually distinct tiles** plus one empty tile (Boris the Brave, "Classification of Tilesets," 2021). The full algorithm in steps: (1) for each cell, read the eight neighbor states as booleans; (2) apply corner elimination to force each diagonal to false if either adjacent cardinal is false; (3) compute the 8-bit bitmask from the masked values; (4) look up the result in a 256-to-47 mapping table or directly in a 47-entry table indexed by the valid bitmask values. Boris the Brave provides the canonical Python lookup table mapping 47 specific decimal values (0, 2, 8, 10, 11, 16, 18, 22 … 248, 250, 251, 254, 255) to tile indices 0–46 ("Tileset Roundup," 2013).

Several important variants trade tile count against art effort and visual quality. The **quarter-tile (sub-blob)** approach decomposes each blob tile into four quadrants, each of which has only five possible states (inner curve, outer curve, horizontal divide, vertical divide, solid center). Per-quadrant index computation checks three neighbors: `topLeft_quad = left + 2×top; if left and top and topLeft: topLeft_quad = 4` (Boris the Brave, "Quarter-Tile Autotiling," 2023). This requires only 20 sub-tiles (or 13 in the "micro-blob" reduction that reuses sub-tiles across quadrants), making it dramatically cheaper for artists while producing output visually identical to a full 47-tile blob set — this is the method RPG Maker VX uses internally. The **dual-grid** approach goes further, requiring only **5 tiles** by placing the visual tilemap at a half-cell offset from the logical tilemap and using a marching-squares lookup on the offset grid (Excalibur.js, "Dual Tilemap Autotiling Technique," 2025). The tradeoff is reduced visual variety and added conceptual complexity around the offset. At the other extreme, exploiting rotation and mirror symmetry — Boris the Brave's notation **S-V2E2-RM-Blob** — reduces the blob set to just 16 rotationally symmetric tiles, though this is only acceptable when rotated sprites look natural (Boris the Brave, "Classification of Tilesets," 2021).

The most common failure mode is **missing tile mappings**: if the tileset lacks sprites for all 47 blob configurations, some bitmask values produce blank or default tiles. Godot's terrain system addresses this with a fallback heuristic that scores candidate tiles by shared edges (2 points each) and shared corners (1 point each), selecting the highest-scoring alternative (Godot proposal #3851). A subtler bug is **incorrect bit ordering** — different engines, tutorials, and community resources assign the eight neighbor bits in different sequences, and the mapping table must match. The Excalibur.js team notes this mapping is "tedious and brittle," with one author finding seven bitmask values that mapped to the same tile during initial implementation (Excalibur.js, "Autotiling Technique," 2024). The most fundamental limitation is the **combinatorial explosion with multiple terrain types**: binary autotiling (terrain A vs. not-A) is manageable, but three terrain types on a marching-squares grid require 3⁴ = 81 tiles, and blob-pattern equivalents scale far worse. Boris the Brave calls this "the major shortcoming of standard autotiling" ("Beyond Basic Autotiling," 2021). Practical workarounds include layering multiple binary autotile passes with transparency (the most common approach), alpha-mask blending between texture layers (used by Factorio), and terrain-priority ordering that handles transitions pairwise.

Autotiling is the simplest technique in this pass and the one most directly replaced by WFC when complexity demands grow. It connects to Wang tiles in a precise way: in Boris the Brave's classification, blob autotiling is **S-V2E2-Blob** (square grid, binary values on both vertices and edges, with the blob restriction), while classic Wang tiles are **S-E2** (binary values on edges only) and marching-squares autotiling is **S-V2** (binary values on vertices only). All three are Wang-tile-family systems under this taxonomy. Autotiling also connects directly to dungeon layout (Pass 3): a dungeon generator that outputs a binary wall/floor grid can hand that grid to an autotiler for visual finishing, completely separating generation logic from tile art. Boris the Brave describes "Editable WFC" as the bridge between autotiling and WFC — a system where WFC acts as a generalized autotiler, automatically resolving multi-terrain transitions that would require hundreds of hand-specified lookup entries in a bitmask system. Townscaper is the most visible example of this approach ("Editable WFC," 2022).

## 2. Wang tiles enforce edge compatibility across a grid

Wang tiles, first defined by mathematician Hao Wang in 1961, are square tiles with colored edges arranged so that adjacent tiles share matching edge colors, with no rotation or reflection permitted. In game development, the mathematical property of interest is not the original decidability question but rather **stochastic Wang tiling**: designing tile sets that always offer at least two valid choices at each placement step, guaranteeing non-periodic output from a simple greedy algorithm (Cohen et al., "Wang Tiles for Image and Texture Generation," SIGGRAPH 2003). With *c* colors per edge, a complete tile set contains c⁴ tiles; for 2 colors this means **16 tiles**, for 3 colors, 81. Cohen et al. showed that the minimum set for guaranteed stochastic choice is 2×K² tiles, where K is the number of edge colors — just 8 tiles for 2 colors (Cohen et al., 2003). Sean Barrett argues for a different design point: the **complete stochastic set** containing every possible edge combination (16 tiles for 2 colors), which has the practical advantage that tile choices have no influence beyond immediate neighbors and trivially supports pre-placement of special content before filling (Barrett, nothings.org/gamedev/herringbone).

The placement algorithm is a scanline traversal. (1) Select any tile for the northwest corner. (2) Fill the top row left-to-right, choosing tiles whose west edge matches the east edge of the previously placed tile. (3) For each subsequent row, select the first tile so its north edge matches the south edge of the tile above, then continue rightward matching both north and west constraints simultaneously. At each step, at most **two constraints** must be satisfied (north and west neighbors); the east and south edges are free, becoming constraints for future tiles (Cohen et al., 2003). With a complete stochastic set, at least two tiles satisfy every constraint combination, making each placement an independent random choice — trivially guaranteeing non-periodic output with no possibility of getting stuck. This is the key algorithmic distinction from WFC: Wang tile placement is O(1) per cell with zero failure probability when the tileset is complete, whereas WFC trades this simplicity for global constraint propagation that can fail and require backtracking.

The most significant variant distinction is **edge-colored vs. corner-colored** Wang tiles. Classic edge-colored tiles (Boris the Brave's classification: **S-E2**) constrain edges but leave diagonal neighbors unconstrained, producing visible artifacts at the point where four tiles meet — the well-documented **corner problem** in texture synthesis (Lagae & Dutré, "An Alternative for Wang Tiles," 2006). Corner-colored tiles (**S-V2**) assign values to vertices instead, directly constraining diagonal neighbors and eliminating this artifact class. In practice, most game autotiling systems use corner-colored Wang tiles because corner values correspond naturally to terrain types painted at grid vertices. Sean Barrett's **herringbone Wang tiles** use 1:2 and 2:1 rectangles in a herringbone layout, isomorphic to hexagonal tiling, with each rectangle effectively having 6 edges — this breaks up the straight grid lines that make standard Wang tile repetition visible and enables richer connectivity control via 4 independent corner-color classes (Barrett, nothings.org/gamedev/herringbone). Barrett released an implementation as `stb_herringbone_wang_tile.h` (2014).

The primary failure modes are incomplete tilesets (placement gets stuck when no tile matches the required constraint combination), visible repetition patterns (especially with minimal tile counts — 8 tiles produce noticeable periodicity), and the inability to produce any global structure. Barrett notes this directly: stochastic Wang tiling is "extremely incoherent, since there's no planning or structure to anything," making it suitable for cave-like environments but problematic for structured content (Barrett, nothings.org). **Implementation-level documentation for Wang tiles in game contexts is notably thinner than for bitmask autotiling or WFC.** The placement algorithm is almost too simple to warrant tutorials (it is literally "scanline, pick randomly from valid tiles"), while the tile *construction* algorithm for texture synthesis (Cohen's diamond-patch image quilting) receives more academic attention than the placement algorithm itself. The best practical references remain Barrett's herringbone page, Boris the Brave's classification system, and the Tiled map editor's Wang tile support (added via GSoC in 2017). Developers should understand Wang tiles primarily as the mathematical framework underlying both autotiling and the adjacency model used by WFC — Boris the Brave explicitly equates WFC's adjacency model with Wang tiles ("Constraint-Based Tile Generators," 2021).

## 3. Adjacency-rule systems sit between lookup tables and constraint solvers

Adjacency-rule systems define which tiles or modules can legally neighbor each other, one constraint per grid edge. Boris the Brave defines the adjacency model as a system that "sets requirements about pairs of tiles that are directly adjacent to each other" ("Constraint-Based Tile Generators," 2021). The critical distinction for this pass is that adjacency rules are both a **data structure** — the constraint information consumed by solvers like WFC and Model Synthesis — and a **simple placement algorithm** in their own right, when used greedily without propagation or backtracking. In the greedy variant, the algorithm is: for each cell in some traversal order, examine already-placed neighbors, filter the candidate tile set to those compatible with all placed neighbors per the rule set, pick one (randomly or by weight), and move on. Isaac Dykeman describes this explicitly as "greedy placement": "Place a tile as long as all its edges line up with existing tiles, without regard for whether this placement will create a partial tiling that cannot be completed without removing existing tiles" (Dykeman, "Procedural Worlds from Simple Tiles," 2017).

Adjacency rules can be specified in three ways: **sample-based extraction**, where the system scans a hand-authored example map and records every observed tile adjacency pair; **label or socket matching**, where each tile edge is annotated with a typed label and two tiles are compatible only if their facing labels match; and **content-based inference**, where the system examines pixel colors at tile borders or mesh vertex positions at module faces to determine compatibility automatically ("Constraint-Based Tile Generators," 2021). Sample-based extraction is the quickest to prototype but scales poorly — Boris the Brave warns it is "often too precise — the model will learn details of the input that are incidental, and not intended by the user," especially as tile counts grow. Label-based specification (the precursor to socket systems) avoids this by letting designers annotate tiles independently, but requires more upfront design effort. Content-based inference, used by Oskar Stålberg in both Bad North and Townscaper, is the most scalable for large tilesets since new tiles are automatically compatible without manual annotation. Paul Merrell's **Model Synthesis** (2007) formalized the use of adjacency constraints for procedural generation, framing it as a constraint satisfaction problem solved via arc consistency (AC-3/AC-4). Boris the Brave characterizes Merrell's fundamental insight as: "Any constraint-based solver can be modified to be a constraint-based generator via random choice" ("Model Synthesis and Modifying in Blocks," 2021). WFC, created by Maxim Gumin in 2016, reimplements essentially the same algorithm with the addition of a min-entropy cell-selection heuristic and a focus on 2D examples that made it far more accessible (Merrell, paulmerrell.org/model-synthesis).

**An honest note on documentation**: pure greedy adjacency placement — without WFC's propagation machinery — has minimal dedicated literature. It is almost always discussed as a naive baseline before introducing WFC as the superior alternative. Dykeman's 2017 blog post is the most explicit standalone treatment; Boris the Brave's extensive writing treats adjacency rules primarily as input to WFC/Model Synthesis solvers, not as a standalone placement strategy. The greedy algorithm's pseudocode from Dykeman is: initialize the entire map to UNDECIDED; while undecided tiles remain, if any valid placement exists, select randomly from valid placements weighted by tile probabilities and place it; otherwise, pick a random undecided tile and set all its neighbors to UNDECIDED (a backjumping recovery step). This algorithm has no halt guarantee on adversarial tilesets and cannot detect unsolvable rule sets (Dykeman, 2017). The primary failure mode is **dead ends**: past placements constrain future options to zero, with no mechanism to recover gracefully. Dykeman's backjumping — clearing a local neighborhood and retrying — is a pragmatic patch, but order-dependence means different traversal orders produce wildly different failure rates. Dykeman later developed a more sophisticated approach for Generate Worlds (2019): precompute each tile type's **allowed neighborhood** (the set of tiles remaining possible at each surrounding cell after running arc consistency to fixpoint), cache this offline, then at runtime apply it as a local set intersection per placement. This gives propagation-like pruning power without runtime propagation cost, occupying a middle ground between pure greedy and full WFC (Dykeman, "The Constrained Tile Placement Algorithm behind Generate Worlds," 2019).

Adjacency-rule systems connect to the other techniques in this pass in a precise hierarchy. Autotiling is a **degenerate case** where the rule set is implicit in the bitmask design and exactly one tile is valid per neighbor configuration — no selection or randomness is involved. Wang tiles are adjacency rules where compatibility is determined by edge or corner color matching. Socket-based systems (covered next) are a compact encoding mechanism for adjacency rules, reducing O(n²) explicit pair rules to O(n × faces) socket annotations. And WFC/Model Synthesis use adjacency rules as their constraint data but add propagation (AC-3/AC-4) and heuristic cell selection, gaining the ability to enforce consistency across the entire grid at the cost of potential contradictions requiring backtracking or restarts. For dungeon layout (Pass 3), adjacency rules naturally express room-to-room connectivity constraints — a corridor module can neighbor a room module but not a boss-room module directly. For biome placement (Pass 5), adjacency rules can enforce terrain transition logic (desert cannot directly neighbor tundra) at a coarser scale than autotiling's per-tile transitions.

## 4. Socket-based modular placement encodes adjacency as typed connectors

Socket-based (or connector-based) modular placement annotates each face or edge of a tile or 3D module with typed connectors — sockets — and determines placement legality by testing whether facing sockets are compatible. Rather than maintaining an explicit N×N adjacency matrix for every pair of tiles in every direction, the designer assigns O(faces) labels per module and derives compatibility from socket matching. This compression is the technique's core value proposition: **a 400-module tileset like Bad North's would require 400 × 400 × 6 = 960,000 explicit directional rules as a raw adjacency table, but needs only 400 × 6 = 2,400 socket annotations** from which all rules are inferred (Boris the Brave, "Constraint-Based Tile Generators," 2021; Tessera paper, FDG 2021). The placement algorithm, in its simplest local form, mirrors greedy adjacency placement: for each cell, query the sockets of already-placed neighbors on their facing sides, filter candidate modules to those whose sockets are compatible with all placed neighbors, select one (randomly, by weight, or by priority), and place it. In practice, socket systems are most often paired with a WFC or Model Synthesis solver that adds propagation, but the socket-compatibility test itself is purely local.

The socket typing system is where most of the design complexity lives, and getting it right determines whether the system produces coherent output or frequent contradictions. The simplest scheme is **direct type matching**: each socket is an integer or string, and two sockets are compatible if and only if they are equal. This works for vertical sockets (top and bottom faces in 3D), where two stacked tiles view the shared face from the same orientation. For horizontal sockets (north, south, east, west), a critical subtlety arises: when two tiles sit side by side, their shared face is viewed from opposite directions, so one tile's "right" is the other's "left." This creates the **symmetric vs. asymmetric socket** distinction. A symmetric socket (conventionally suffixed `_s`) represents a face pattern unchanged by mirroring — socket `A_s` matches another `A_s`. An asymmetric socket `A` matches only its mirror `A_f`, not itself. The compatibility function becomes: for vertical faces, test `a == b`; for horizontal faces, if the socket is marked symmetric, test `a == b`; otherwise test `a == flip(b)` where `flip` toggles the `_f` suffix (Marian Kleineberg, marian42.de; Steven Landow, landow.dev; Wauthier, "Using WFC for 2D and 3D Level Generation," 2023). In 3D cube systems, vertical sockets may also carry a **rotation index** (0–3) to ensure geometric alignment between vertically stacked modules that are not rotationally invariant on their top/bottom faces (Wauthier, 2023).

Boris the Brave's **Tessera** system pushes socket expressiveness further with a **sub-face painting** approach: each face of a cube tile is subdivided into a 3×3 grid of independently paintable color cells, and two tiles can be adjacent if and only if all nine sub-face pairs match as mirror images. This nine-label-per-face system is a strict superset of Wang tiles, marching cubes, and blob patterns; it handles rotation and reflection automatically (the sub-face pattern rotates with the tile), and the small painted patterns serve as visual mnemonics that are easier to reason about than abstract socket IDs. The central sub-face can encode path-constraint information, enabling Tessera's global connectivity checks (Newgas, "Tessera: A Practical System for Extended WFC," FDG 2021). An alternative to manual socket annotation is **content-based inference**: scanning mesh vertex positions or pixel colors at tile boundaries and declaring two tiles compatible if their geometry matches. Oskar Stålberg uses this approach in both Bad North and Townscaper — when tiles are imported from the modeling tool, edges on all faces are scanned and compared automatically, meaning new tiles become compatible with existing ones without any manual socket setup ("Constraint-Based Tile Generators," 2021; Game Developer, "How Townscaper Works"). This is the most scalable approach for large tilesets but requires that tiles be authored with physically matching boundaries.

Rotation and symmetry handling is a major implementation concern, particularly in 3D. For each base module on a square grid, four rotational variants are generated by cycling sockets: north→east, east→south, south→west, west→north. For cube grids, up to **24 distinct rotations** exist (48 with reflections), and for a given pair of cube faces there are 8 possible alignment configurations (Tessera paper, FDG 2021). DeBroglie and Tessera define symmetry classes named after letters with similar geometric symmetry: **X** (fully symmetric, 1 variant needed), **T** (one reflection axis, 4 variants), **I** (2-fold rotational, 2 variants), **L** (no symmetry, 4 variants or 8 with reflections). Exploiting these symmetries is not merely convenient — it directly impacts performance. Tessera's documentation warns that "doubling the number of tiles makes Tessera approximately 4× as slow" due to the quadratic scaling of constraint propagation checks. Disabling unnecessary rotations for symmetric tiles is one of the most effective optimizations available.

Oskar Stålberg's **Townscaper** is the most technically ambitious socket-based system in production and illustrates how multiple techniques from this pass combine. The system layers three mechanisms: (1) an **irregular quadrilateral grid** generated by starting with hexagons, randomly removing edges, and subdividing to ensure all faces are quads, producing organic European-street layouts; (2) a **binary voxel model** where each cell corner is on or off, indexed via a modified marching-cubes lookup into hand-crafted architectural modules (walls, roofs, arches, stairs) — Stålberg reported building modules for all **378 possible bit-pattern cases** (@OskSta, Twitter, October 2019); and (3) **WFC as a selection layer** that chooses among multiple valid modules for a given bit pattern based on socket compatibility with neighbors, using a **priority-based deterministic heuristic** rather than random selection. This determinism is crucial for the game's design: users learn that specific block patterns reliably produce specific architectural features (lighthouses, garden courtyards, arched walkways). The system operates as **"driven WFC"** — user clicks modify a hidden solidity/color data layer, and WFC recomputes visible tile assignments from that data as constraints. Boris the Brave distinguishes this from his own "editable WFC" approach, where the tile map itself is the editable artifact: in Townscaper, "the actual data edited is never touched by a generator; instead, the game just needs to recompute the processed result" ("Editable WFC," 2022). Bad North (also Stålberg) uses a similar 3D WFC with over **400 hand-crafted tiles**, content-based socket inference at import, per-island tile subset pre-selection for visual consistency, and a custom navigability heuristic that ensures the resulting terrain is walkable at each WFC step (Tessera paper, FDG 2021; Gumin, WFC GitHub README).

Socket systems have several characteristic failure modes beyond the dead-ends common to all adjacency-based methods. **Incomplete module libraries** cause frequent contradictions when certain socket combinations at T-junctions or corners have no valid module — the WFC GitHub notes that some tilesets are "non-Wang," meaning their adjacency data cannot be reduced to simple edge coloring, and context-dependent compatibility (two Corner tiles cannot be adjacent but both connect through a Connection tile) requires explicit pair rules that sockets alone cannot express (mxgmn, WFC GitHub README). **Visual discontinuity** can occur even when socket matching succeeds: sockets track geometric or type compatibility but not necessarily material, color, or lighting continuity, producing technically valid but visually jarring module combinations. Townscaper addresses potential contradictions pragmatically: "it is actually allowed to fail silently… it just realizes that there's no way these two cells could have modules that fit together perfectly, so let's just ignore it" (Stålberg, via Game Developer). For dungeon generation (Pass 3), socket systems appear as **exit-marker systems**: each room or corridor module has positioned and oriented exit markers tagged with connection types (e.g., "corridor," "room," "boss"), and the placement algorithm aligns exit transforms rather than snapping to a grid. The Elder Scrolls II: Daggerfall (1996) used a primitive version of this, assembling dungeons from up to 32 pre-fabricated blocks with two connecting passages per cardinal direction at fixed positions (UESP Wiki, "Daggerfall: Dungeons"; Seredynski, "Bake Your Own 3D Dungeons," Game Developer). Modern implementations extend this with tag hierarchies, spatial overlap checking, and loop-closure passes to avoid the pure tree structures that naive exit-marker growth produces.

## Local rule-based systems form a coherent spectrum toward WFC

The four techniques in this pass arrange along an axis of increasing expressiveness and decreasing determinism. **Autotiling** is fully deterministic: one bitmask, one tile, zero choices. **Wang tiles** introduce controlled non-determinism through stochastic selection from a complete set, but the scanline algorithm is still trivially simple, O(1) per cell, and cannot fail. **Adjacency-rule systems** used greedily add the possibility of multiple tile types and richer constraint vocabulary (sockets, labels, sample-extracted rules), but lose the no-failure guarantee — dead ends become possible without propagation. **Socket-based modular placement** adds the richest typing systems (symmetric/asymmetric sockets, sub-face painting, content-based inference, exit markers) and naturally scales to 3D, but the combinatorial complexity of large module libraries makes a pure greedy approach increasingly fragile. This is exactly the point where WFC and Model Synthesis enter: they take the same adjacency rules and socket data as input but add arc-consistency propagation and heuristic cell selection, converting a fragile greedy process into a robust constraint solver. Pass 4b covers that transition — the key framing is that WFC does not replace these local techniques but rather *consumes* them, using autotiling-style bitmask logic, Wang-tile-style edge matching, or socket-defined adjacency rules as its constraint model while adding the propagation machinery that makes global consistency achievable.

## Source landscape

**Tutorials and explainers.** Boris the Brave's "Autotile Compendium" (boristhebrave.com, 2021) and the linked "Classification of Tilesets" article are the single best resource for understanding how autotiling, Wang tiles, and blob patterns relate to each other through a unified naming system — no other source provides this taxonomy. The Excalibur.js "Autotiling Technique" blog post (2024) gives the clearest step-by-step walkthrough of implementing 8-bit blob autotiling from scratch in JavaScript, including the exact neighbor-offset array and lookup table. Envato Tuts+ "How to Use Tile Bitmasking to Auto-Tile Your Level Layouts" covers the 4-bit variant with worked binary-to-decimal examples suitable for developers encountering bitmasks for the first time. Isaac Dykeman's "Procedural Worlds from Simple Tiles" (ijdykeman.github.io, 2017) is the most explicit treatment of greedy adjacency placement as a standalone algorithm, including the backjumping recovery strategy — uniquely valuable because most adjacency-rule content assumes WFC as the solver. Boris the Brave's "Constraint-Based Tile Generators" (2021) provides the definitive taxonomy separating model, model inference, solver, and heuristic as independent composable components — essential reading for anyone designing a tile generation pipeline. Sean Barrett's herringbone Wang tile page (nothings.org/gamedev/herringbone) is the best practical resource on Wang tile variants for game-level generation, including the "complete stochastic set" concept and connectivity analysis that appear nowhere else in the literature.

**Interactive tools and editors.** Boris the Brave's Tileset Creator (boristhebrave.com/permanent/23/03/tileset-creator/) is an interactive browser tool for exploring classified tileset types — the only tool that lets developers directly experiment with the S-V2, S-E2, and S-V2E2-Blob classification categories. The Tiled map editor (mapeditor.org) provides native Wang tile support for both edge-colored and corner-colored variants, added via a 2017 GSoC project, and is the standard open-source tool for 2D tile-based level design.

**Libraries and repositories.** DeBroglie (boristhebrave.github.io/DeBroglie) is an open-source C# library implementing WFC with AC-4, backtracking, adjacent and overlapping models, path constraints, and multi-grid support — the most feature-complete open-source WFC implementation available. Maxim Gumin's original WFC repository (github.com/mxgmn/WaveFunctionCollapse) remains the canonical reference implementation with extensive documentation of tile symmetry classes and links to notable uses. Sean Barrett's `stb_herringbone_wang_tile.h` (2014) is a single-header C library for herringbone Wang tile generation — the only ready-to-use library specifically for this variant. The `ghx_proc_gen` Rust library (github.com/Henauxg/ghx_proc_gen) provides a clean socket-based API with explicit connection definitions between socket types, useful as a reference for socket system architecture.

**Academic papers.** Cohen et al., "Wang Tiles for Image and Texture Generation" (SIGGRAPH 2003) is the seminal paper bridging Wang tiles and texture synthesis, containing the scanline stochastic tiling algorithm and the diamond-patch tile construction method. Paul Merrell's "Example-Based Model Synthesis" (i3D 2007) and subsequent PhD dissertation (UNC Chapel Hill, 2009) established the adjacency-constraint-as-CSP formulation and the "modifying in blocks" contradiction-handling strategy — essential precursor context for WFC. Lagae and Dutré, "An Alternative for Wang Tiles: Colored Edges versus Colored Corners" (2006) provides the definitive analysis of the corner problem in edge-colored Wang tiles and demonstrates corner-colored tiles as the fix. Adam Newgas, "Tessera: A Practical System for Extended WFC" (FDG 2021) documents the sub-face painting system, AC-4 implementation, path constraints, and multi-grid support — the most comprehensive peer-reviewed treatment of a production WFC system.

**Game developer talks and interviews.** The Game Developer article "How Townscaper Works" (based on AI and Games / Tommy Thompson's interview with Stålberg) is the most detailed public breakdown of Townscaper's three-layer architecture (irregular grid, marching cubes, WFC). Stålberg's EPC 2018 talk "Wave Function Collapse in Bad North" covers the navigability heuristic, content-based socket inference, and multi-cell tile design — the best primary source on applying WFC to a shipped 3D game. Boris the Brave's "Editable WFC" blog post (2022) formalizes the dirty-cell heuristic and similarity-based tile selection that make WFC usable as an interactive autotiler, directly connecting the simplest technique in this pass to the most complex.
